Quick Recap

Before We Begin — Do You Remember?

Let's test your understanding of payment fundamentals from Class 12. These concepts are the foundation for everything we'll build today.

Q1: A customer pays ₹500 for an order. What amount does your backend send to Razorpay when creating an order?

Reveal Answer

50000 (fifty thousand paise). Payment gateways expect amounts in the smallest currency unit — paise for INR, cents for USD. ₹500 × 100 = 50000 paise.

But why? Try this in your Python terminal:

>>> 0.1 + 0.2
0.30000000000000004    # NOT 0.3!

>>> 0.1 + 0.2 == 0.3
False                  # Python says 0.3 != 0.3

# Now imagine this in payments:
>>> price = 10.10          # ₹10.10
>>> qty = 3
>>> price * qty
30.299999999999997     # Should be ₹30.30!

# Customer is charged ₹30.299999... or ₹30.30?
# Over millions of transactions, these tiny errors add up
# to REAL money discrepancies.

# With integers (paise), this NEVER happens:
>>> price_paise = 1010     # 1010 paise = ₹10.10
>>> price_paise * 3
3030                   # Exactly 3030 paise = ₹30.30. Always.

This is IEEE 754 floating-point — computers store decimals as binary fractions, and some values (like 0.1) can't be represented exactly. Integers don't have this problem, which is why every PG uses the smallest currency unit.

Official docs confirming this:
Razorpay: "amount — The amount to be paid by the customer in paise" — razorpay.com/docs/api/orders/create
Stripe: "amount — in the smallest currency unit (e.g., 100 cents to charge $1.00)" — docs.stripe.com/api/payment_intents/create

Great explainer video: Why 0.1 + 0.2 != 0.3 (YouTube) →

Q2: What's the difference between a Callback and a Webhook? Which one should you trust to update your database?

Reveal Answer

Callback = Browser redirect after payment. The PG redirects the user's browser to a URL you specify (your "callback URL"). This happens in the user's browser.

How do you set up a callback?

  • In API request: Pass callback_url when creating a payment link or order — "callback_url": "https://yoursite.com/payment-result/"
  • In Razorpay Dashboard: Settings → Checkout → set a default redirect URL
  • In frontend JS: The handler function in Razorpay Checkout options runs after payment

The API approach (callback_url) is better — it's per-payment, configurable, and doesn't depend on frontend code.

Webhook = Server-to-server POST from the PG to your backend. Not through the user's browser at all. Cryptographically signed so you can verify it's really from Razorpay.

Important: Callback does NOT mean payment was successful.

The callback just means the user came back from the PG's page. They could have:

  • Paid successfully → callback arrives with razorpay_payment_id
  • Payment failed → callback still arrives (with error info)
  • Closed the browser mid-payment → callback never arrives at all
  • Manually typed your callback URL → fake callback, no real payment

This is why you never trust the callback to update your database. Use it only to show the user a "thank you" or "payment failed" page. The webhook is the only reliable signal.

TL;DR: Callback = for UX (redirect the user somewhere nice). Webhook = for data (actually update your DB). Trust the webhook, not the callback.

Q3: Why do we use UUIDs instead of auto-incrementing IDs (1, 2, 3...) for payment records?

Reveal Answer

Sequential IDs are guessable. If your payment ID is pay_1042, an attacker knows pay_1041 and pay_1043 exist and can try to access them. UUIDs like a3f7b2c1-... are random and unguessable — this prevents enumeration attacks.

Q4: In a payment flow, the frontend sends the order amount to the backend. Can a malicious user change ₹5000 to ₹1 in the request and pay less?

Reveal Answer

Not if you design it correctly. The frontend should only send the order_id. The backend should calculate the amount itself from the order items in the database, then pass that to Razorpay. Never trust the frontend for amounts.

Additionally, the Razorpay order is created with a fixed amount. Even if someone tampers with the checkout, Razorpay won't accept a payment that doesn't match the order amount.

Q5: Why do PGs verify webhook signatures using hmac.compare_digest() instead of a simple == comparison?

Reveal Answer

A regular == comparison short-circuits — it returns False as soon as it finds the first mismatched character. An attacker can measure how long the comparison takes and figure out the secret one character at a time. This is called a timing attack. hmac.compare_digest() always takes the same amount of time regardless of where the mismatch is.

Q6: Your Razorpay API keys are in settings.py. A teammate accidentally pushes the code to a public GitHub repo. What's the risk, and how should you have stored the keys?

Reveal Answer

Risk: Anyone can find the keys via GitHub search and make API calls on your behalf — create fake refunds, access customer data, or drain your account.

Solution: Environment variables. Store secrets outside code:

  • Local dev: Use a .env file (added to .gitignore) or export RAZORPAY_KEY_ID='...' in your ~/.zshrc / ~/.bashrc
  • Production: Set via your hosting platform (AWS EB config, Heroku config vars, Docker env)
  • In Django: Read with os.environ.get('RAZORPAY_KEY_ID')

~/.bashrc and ~/.zshrc are shell config files that load every time you open a terminal. Any export KEY=VALUE in them becomes available to all programs in that session.

Great. Now the real question...

We learned that webhooks are the source of truth for payment updates. But webhooks are just HTTP requests — and HTTP requests can fail. What happens when a webhook never reaches your server? That's exactly what reconciliation solves.

The Problem

Why Do We Need Reconciliation?

In Class 12, we learned webhooks are the best way to get payment updates. But what happens when webhooks fail?

The Uncomfortable Truth About Webhooks

Webhooks are not guaranteed to reach your server. Even with retries, things can go wrong:

  • Your server was down during all retry attempts
  • Network issues between PG and your server
  • Your webhook endpoint returned 200 OK but crashed before updating DB
  • PG's webhook system itself had issues (rare, but happens)

What is Reconciliation?

Reconciliation = Comparing your database with the Payment Gateway's records to find and fix mismatches.

Your Database

Order IDAmountStatus
ORD-001₹500Paid
ORD-002₹1200Pending
ORD-003₹800Pending

Payment Gateway

Payment IDAmountStatus
pay_001₹500Captured
pay_002₹1200Captured
pay_003₹800Failed

Mismatches Found!

  • ORD-002: You think it's "Pending", but PG says "Captured" — Customer paid but didn't get the product!
  • ORD-003: You think it's "Pending", but PG says "Failed" — Customer didn't pay, order should be cancelled

Reconciliation Fixes This

A reconciliation job runs periodically (e.g., every hour), fetches payment statuses from PG, and updates your database to match reality.

If reconciliation can fix all mismatches, why don't we just skip webhooks entirely and only rely on reconciliation?

Reveal Answer

Because reconciliation is polling-based — it runs at fixed intervals (e.g., every hour). That means a customer could pay and wait up to an hour before their order is confirmed. Webhooks give you real-time updates (within seconds). So webhooks are the primary mechanism, and reconciliation is the safety net.

Real Scenarios

What Can Go Wrong?

Understanding failure modes helps you build robust systems. Here are real scenarios that happen in production.

1. Webhook Never Arrived

Your server was deploying new code (downtime) exactly when Razorpay sent the webhook. All retry attempts failed. Customer paid ₹5000 but order shows "pending".

🛡️ How to Prevent

Reconciliation cron job — Run a periodic job (e.g., every hour) that fetches payment statuses from Razorpay's API and compares with your DB. This catches any webhook that never arrived. It's your ultimate safety net.

2. Webhook Arrived, DB Update Failed

Your webhook handler returned 200 OK (so PG won't retry), but the database was temporarily unavailable. Payment captured, but your DB still says "pending".

🛡️ How to Prevent

Only return 200 after a successful DB update. If your DB write fails, return 500 so the PG knows to retry. Structure your handler as: verify → update DB → then return 200. Never return 200 optimistically.

3. Partial Failure

Order has 3 items. Payment succeeded, but inventory update for 1 item failed. Customer charged, 2 items shipped, 1 item forgotten.

🛡️ How to Prevent

Database transactions — Wrap all related updates in transaction.atomic(). Either all changes succeed (order status + all inventory items), or none do. The DB rolls back everything on any failure. No more "2 of 3 items shipped".

4. Duplicate Webhook Processing

Your server was slow, Razorpay retried, you processed the same webhook twice. Customer charged once, but your system shows 2 orders.

🛡️ How to Prevent

Idempotency keys — Store each webhook's event_id in a WebhookEvent table. Before processing, check: "Have I seen this event_id before?" If yes, return 200 and skip. This makes your handler safe to call multiple times with the same data.

5. Refund Not Recorded

Customer support issued refund directly from Razorpay dashboard. Webhook missed. Your DB shows "paid", but money was returned.

🛡️ How to Prevent

Reconciliation + listen for refund webhooks. Subscribe to refund.created events in your webhook handler. And your reconciliation cron should also check for refund status, not just payment status. The cron is your catch-all for anything the webhook missed — including manual actions from the dashboard.

Automation

Scheduled Jobs (Crons)

Reconciliation needs to run automatically at regular intervals. But the concept of scheduling jobs goes far beyond just payments — it's one of the most fundamental patterns in backend engineering.

What is a Cron Job?

A cron job is a task that runs automatically at scheduled times — without anyone pressing a button or hitting an API. The name comes from Unix's cron daemon (from the Greek word chronos = time), a background service that has been scheduling tasks on servers since the 1970s.

Think of it like setting an alarm on your phone. You configure when it should go off, and the system handles the rest. The difference? Instead of waking you up, it runs a piece of code.

How a Cron Job Works
⏰ Scheduler "Run at 2:00 AM daily"
➔

➔
💻 Your Script Runs Does the work
➔
💤 Sleeps Until next schedule

1 Django Management Commands + System Cron

Django has a built-in concept called management commands — these are scripts you can run from the terminal with python manage.py <command_name>. You've already used some: runserver, migrate, createsuperuser. They're all management commands!

The idea is simple: write your job as a management command, then use Linux's crontab to run it on a schedule.

The Generic Pattern (works for any Django project)

yourapp/management/commands/your_task.py
▼ 
# File location matters! Must follow this structure:
# yourapp/
#   management/
#     __init__.py
#     commands/
#       __init__.py
#       your_task.py    <-- Your command goes here

from django.core.management.base import BaseCommand

class Command(BaseCommand):
    help = 'Description of what this command does'

    # Optional: Accept command line arguments
    def add_arguments(self, parser):
        parser.add_argument('--dry-run', action='store_true',
                            help='Show what would happen without making changes')

    def handle(self, *args, **options):
        # This is where your job logic goes
        # You have full access to Django models, settings, etc.

        if options['dry_run']:
            self.stdout.write('DRY RUN MODE - no changes will be made')

        # Do your work here...
        self.stdout.write(self.style.SUCCESS('Task completed successfully!'))

# Usage:
# python manage.py your_task
# python manage.py your_task --dry-run

Our Restaurant Project: Reconciliation Command

orders/management/commands/reconcile_payments.py
▼ 
from django.core.management.base import BaseCommand
from django.conf import settings
from orders.models import Order
import razorpay
from datetime import datetime, timedelta

client = razorpay.Client(auth=(settings.RAZORPAY_KEY_ID, settings.RAZORPAY_KEY_SECRET))

class Command(BaseCommand):
    help = 'Reconcile pending orders with Razorpay'

    def handle(self, *args, **options):
        # Find orders that have been "pending" for more than 1 hour
        one_hour_ago = datetime.now() - timedelta(hours=1)
        pending_orders = Order.objects.filter(
            status='pending',
            created_at__lt=one_hour_ago,
            razorpay_order_id__isnull=False
        )

        self.stdout.write(f'Found {pending_orders.count()} pending orders to check')

        for order in pending_orders:
            try:
                # Fetch payments for this Razorpay order
                payments = client.order.payments(order.razorpay_order_id)

                for payment in payments['items']:
                    if payment['status'] == 'captured':
                        order.status = 'paid'
                        order.razorpay_payment_id = payment['id']
                        order.save()
                        self.stdout.write(self.style.SUCCESS(
                            f'Order {order.id}: Updated to PAID'
                        ))
                        break
                    elif payment['status'] == 'failed':
                        order.status = 'failed'
                        order.save()
                        self.stdout.write(self.style.WARNING(
                            f'Order {order.id}: Updated to FAILED'
                        ))
                        break
                    else:
                        self.stdout.write(
                            f'Order {order.id}: Payment status {payment["status"]}, skipping'
                        )
            except Exception as e:
                self.stdout.write(self.style.ERROR(
                    f'Order {order.id}: Razorpay error - {e}'
                ))

What does this cron expression do?  30 2 * * 0

Reveal Answer

Every Sunday at 2:30 AM.
Minute=30, Hour=2, Day=*(any), Month=*(any), Weekday=0(Sunday).
This is a common schedule for weekly database backups — runs at a quiet time when traffic is low.

⚠️ Key Things to Keep in Mind with System Cron

  • Always use absolute paths — cron doesn't know your working directory or virtualenv
  • Redirect output to a log file — >> /var/log/file.log 2>&1 captures both stdout and stderr
  • Environment variables aren't loaded — cron runs with a minimal shell, so your .bashrc / .env won't be sourced
  • Test manually first — always run the command by hand before scheduling it
  • Monitor failures — cron silently swallows errors unless you set up logging

System crontab works, but what if your cron job crashes halfway? What if you need to retry failed jobs? What if you're running 20 different scheduled tasks and need to monitor them all?

This is where Celery comes in — it's like crontab on steroids.

2 Celery Beat (Production-Grade)

Celery is a task queue — it lets you run code in the background, retry on failure, and monitor everything. Celery Beat is the scheduler that tells Celery when to run tasks. Together, they replace crontab with something far more powerful.

Why is Celery Better for Production?

System Cron (Simple but Limited)
📅 Crontab OS-level scheduler
➔
💻 Script Runs In main process
➔
❌ If it fails? Gone. No retry.
Celery + Beat (Production-Ready)
📅 Celery Beat Smart scheduler
➔
📨 Redis/RabbitMQ Message queue
➔
⚙️ Celery Workers Multiple workers
➔
🔄 Auto-Retry On failure

System Cron Limitations

  • Job fails? Silent failure. No automatic retry.
  • Running on 3 servers? Cron runs on all 3 (duplicate work!)
  • Want to see which jobs ran? Parse log files manually
  • Job takes 2 hours but scheduled every hour? Overlap! Two copies running
  • Need to run 50 tasks concurrently? Blocks the server

Celery Solves All of These

  • Job fails? Automatic retry with exponential backoff
  • 3 servers? Tasks go to a shared queue — only one worker picks it up
  • Built-in monitoring dashboard (Flower)
  • Task already running? Skip or queue the next one
  • 50 tasks? Distributed across multiple workers

from celery import shared_task
from django.conf import settings
from orders.models import Order
import razorpay

client = razorpay.Client(auth=(settings.RAZORPAY_KEY_ID, settings.RAZORPAY_KEY_SECRET))

@shared_task
def reconcile_payments():
    """Reconcile pending orders with Razorpay - runs every hour"""
    pending_orders = Order.objects.filter(
        status='pending',
        razorpay_order_id__isnull=False
    )

    updated_count = 0
    for order in pending_orders:
        payments = client.order.payments(order.razorpay_order_id)
        for payment in payments['items']:
            if payment['status'] == 'captured':
                order.status = 'paid'
                order.razorpay_payment_id = payment['id']
                order.save()
                updated_count += 1
                break

    return f'Reconciled {updated_count} orders'

3 Cloud Schedulers

If you're running serverless (AWS Lambda, Google Cloud Functions) or don't want to manage Redis/Celery infrastructure, cloud schedulers trigger an HTTP endpoint on a schedule. You write the logic in a regular Django view, and the cloud handles timing.

When to Use Which?

Approach Best For Retry on Failure? Monitoring? Scales?
Crontab + Management Command Side projects, simple tasks, learning No Manual (logs) No
Celery Beat Production apps, multiple tasks, teams Yes (built-in) Yes (Flower UI) Yes (add workers)
Cloud Scheduler Serverless, microservices, pay-per-use Yes (configurable) Yes (cloud dashboard) Yes (auto)

TL;DR Recommendation

Learning / small project? Start with management commands + crontab. It works, it's simple, and you'll understand what's happening.
Going to production? Use Celery Beat. The extra setup pays for itself immediately with retry, monitoring, and no-duplicate guarantees.
Already on AWS/GCP? Cloud schedulers save you from managing Celery infrastructure.

from django.db import models

class Order(models.Model):
    STATUS_CHOICES = [
        ('pending', 'Pending'),
        ('paid', 'Paid'),
        ('failed', 'Failed'),
        ('refunded', 'Refunded'),
    ]

    user = models.ForeignKey('auth.User', on_delete=models.CASCADE)
    amount = models.DecimalField(max_digits=10, decimal_places=2)
    status = models.CharField(max_length=20, choices=STATUS_CHOICES, default='pending')

    # Razorpay-specific fields
    razorpay_order_id = models.CharField(max_length=255, blank=True, null=True)
    razorpay_payment_id = models.CharField(max_length=255, blank=True, null=True)

    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    def __str__(self):
        return f"Order {self.id} - ₹{self.amount} - {self.status}"


class WebhookEvent(models.Model):
    """Track processed webhooks to prevent duplicate processing"""
    razorpay_event_id = models.CharField(max_length=255, unique=True)
    event_type = models.CharField(max_length=100)
    processed_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        indexes = [
            models.Index(fields=['razorpay_event_id']),
        ]

Why do we need a separate WebhookEvent model? Can't we just check if the Order is already "paid"?

Reveal Answer

Checking order status alone isn't enough. What if you receive a refund.created webhook for an order that's already "paid"? You'd process it. But if Razorpay retries the same refund webhook, you'd process it again. The WebhookEvent model tracks each unique event by ID, so you can safely say "I've already handled this exact event" regardless of order status.

5 Webhook Handler (The Important Part!)

orders/views.py - Webhook Handler
▼ 
import json
import razorpay
from django.conf import settings
from django.http import HttpResponse
from django.views.decorators.csrf import csrf_exempt
from django.db import transaction
from .models import Order, WebhookEvent

client = razorpay.Client(auth=(settings.RAZORPAY_KEY_ID, settings.RAZORPAY_KEY_SECRET))

@csrf_exempt
def razorpay_webhook(request):
    payload = request.body
    signature = request.META.get('HTTP_X_RAZORPAY_SIGNATURE', '')

    # 1. Verify webhook signature (CRITICAL for security!)
    try:
        client.utility.verify_webhook_signature(
            payload.decode('utf-8'), signature, settings.RAZORPAY_WEBHOOK_SECRET
        )
    except razorpay.errors.SignatureVerificationError:
        return HttpResponse(status=400)  # Invalid signature

    event = json.loads(payload)

    # 2. Check for duplicate (idempotency)
    event_id = event.get('event_id', '')
    if WebhookEvent.objects.filter(razorpay_event_id=event_id).exists():
        return HttpResponse(status=200)  # Already processed

    # 3. Process the event
    try:
        with transaction.atomic():
            if event['event'] == 'payment.captured':
                handle_payment_success(event['payload']['payment']['entity'])
            elif event['event'] == 'payment.failed':
                handle_payment_failure(event['payload']['payment']['entity'])
            elif event['event'] == 'refund.created':
                handle_refund(event['payload']['refund']['entity'])

            # 4. Record that we processed this event
            WebhookEvent.objects.create(
                razorpay_event_id=event_id,
                event_type=event['event']
            )

        return HttpResponse(status=200)

    except Exception as e:
        # Return 500 so Razorpay retries
        return HttpResponse(status=500)


def handle_payment_success(payment):
    """Handle successful payment"""
    order_id = payment.get('notes', {}).get('order_id')
    if order_id:
        Order.objects.filter(id=order_id).update(
            status='paid',
            razorpay_payment_id=payment['id']
        )
        # TODO: Send confirmation email, update inventory, etc.


def handle_payment_failure(payment):
    """Handle failed payment"""
    order_id = payment.get('notes', {}).get('order_id')
    if order_id:
        Order.objects.filter(id=order_id).update(status='failed')


def handle_refund(refund):
    """Handle refund"""
    payment_id = refund.get('payment_id')
    Order.objects.filter(
        razorpay_payment_id=payment_id
    ).update(status='refunded')

How Reconciliation Works at Scale (Real Production)

Level Frequency What It Does
Webhook-level Every 1 hour Catches missed webhooks — compares "pending" payments in your DB against Razorpay API. This is what we built today.
Settlement-level Daily Razorpay settles money to your bank on a T+2 basis (2 business days after capture). Daily reconciliation matches Razorpay's settlement reports against your DB.
Financial reconciliation Monthly Full audit — match bank statements with PG settlement reports. Catches fees, chargebacks, and refund discrepancies.

Razorpay's tools: The Dashboard has a built-in Reconciliation section under Reports → Settlements. Large merchants use the Razorpay Settlements API to pull transaction data in bulk for automated reconciliation. Some merchants negotiate T+1 or instant settlement for faster cash flow.

What Merchants Actually Do with Razorpay

In the real world, Razorpay merchants reconcile using a combination of approaches:

  • Daily CSV download: Most small-medium merchants download the daily transaction report from Razorpay Dashboard → Reports → Transactions and match it against their order database (often in a spreadsheet or accounting tool).
  • Settlement report matching: When Razorpay settles money to your bank (T+2), they provide a settlement report listing which payments were included. Merchants match this against their bank statement to verify the correct amount arrived. Razorpay groups multiple payments into one bank transfer, so you need the report to know which payments are included.
  • Automated via API: Larger merchants (e.g., Swiggy, Zomato) build automated reconciliation pipelines: a cron job calls the /settlements/recon/combined API after each settlement, compares with the DB, and flags discrepancies automatically. Alerts go to a Slack channel or PagerDuty.
  • Third-party tools: Some merchants use dedicated reconciliation services (like Cashfree Reconciliation, Zoho Books integration) that connect to Razorpay's API and auto-reconcile.

The key insight: No matter how automated your webhooks are, you always need a separate reconciliation process because money moving through banks is a different system from payment status updates. A payment can be "captured" in Razorpay but the actual money settles to your bank days later — and sometimes the amounts don't match due to fees, chargebacks, or currency conversion.

🧪 Quick Quiz

▼

Q1: Why do we store razorpay_event_id in the database?

Show Answer

To prevent duplicate processing. If Razorpay retries a webhook and we've already processed it, we skip it. This is called idempotency.

Q2: Why do we return 500 (not 200) when the webhook handler crashes?

Show Answer

So Razorpay knows the webhook wasn't processed successfully and will retry. If we returned 200, Razorpay would think it succeeded and won't retry.

Q3: Why do we need both webhooks AND reconciliation?

Show Answer

Webhooks are real-time but not 100% reliable. Reconciliation is the backup that catches any missed webhooks. Belt and suspenders approach!

Q4: Your cron job runs every hour. A customer pays at 2:01 PM and the webhook fails. When is the earliest their order could be updated?

Show Answer

At 3:00 PM — when the next cron run happens. But our reconciliation also checks for orders older than 1 hour, so the order (created at 2:01 PM) won't be picked up until 3:01 PM at the earliest. In practice, the 4:00 PM cron run would catch it. This is why webhooks are the primary mechanism!

Q5: You're building a SaaS product that serves customers in the US, Europe, and India. Which payment gateway(s) would you choose?

Show Answer

Stripe for US/Europe (best global coverage and developer experience), and Razorpay for India (UPI support is a must for Indian customers). Many companies use multiple PGs for different regions. The integration pattern is the same — just different SDKs!
