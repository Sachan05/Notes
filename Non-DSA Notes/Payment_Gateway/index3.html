Class 14 • Building Production-Ready APIs

Your API returns 10,000 items. The app takes 8 seconds to load. Users leave.

Core Concept

Pagination

Return data in chunks, not all at once.

A The Problem

❌ Without Pagination

GET /menu/items/

Response size: 5.2 MB
Response time: 4.3 seconds
Server memory: 180 MB
SQL: SELECT * FROM items
(loads ALL 50,000 rows)

✅ With Pagination

GET /menu/items/?page=1

Response size: 4.8 KB
Response time: 45 ms
Server memory: 2 MB
SQL: SELECT * FROM items LIMIT 20
(loads only 20 rows)

B Classic Implementation — Offset / Page Number

The most common approach. Client sends a page number, server calculates the offset.

GET /menu/items/?page=1
?page=1 → LIMIT 20 OFFSET 0  |  ?page=2 → LIMIT 20 OFFSET 20  |  ?page=3 → LIMIT 20 OFFSET 40
SQL Under the Hood
▼
-- Page 1
SELECT * FROM menu_items LIMIT 20 OFFSET 0;

-- Page 2
SELECT * FROM menu_items LIMIT 20 OFFSET 20;

-- Page 5000 (PROBLEM: DB must scan & skip 100,000 rows!)
SELECT * FROM menu_items LIMIT 20 OFFSET 100000;
settings.py — Global Pagination
▼ 
# This paginates EVERY list view automatically
REST_FRAMEWORK = {
    'DEFAULT_PAGINATION_CLASS': 'rest_framework.pagination.PageNumberPagination',
    'PAGE_SIZE': 20,
}
Custom PageNumberPagination Class
▼ 
from rest_framework.pagination import PageNumberPagination

class MenuPagination(PageNumberPagination):
    page_size = 10                    # Default items per page
    page_size_query_param = 'size'    # Client can override: ?size=50
    max_page_size = 100               # Cap — never more than 100

class MenuItemListView(generics.ListAPIView):
    queryset = MenuItem.objects.select_related('category').all()
    serializer_class = MenuItemSerializer
    pagination_class = MenuPagination

✅ Pros

  • Simple to implement
  • User can jump to any page
  • Easy to show page numbers

❌ Cons

  • Slow on large offsets (page 5000)
  • Inconsistent — inserts shift results

C Why OFFSET Degrades — Prove It Live

What Happens Inside the Database
Page 1 Skip 0 rows ~2ms
➔
Page 100 Skip 2,000 rows ~50ms
➔
Page 5000 Skip 100,000 rows ~3-5 seconds

The DB literally reads and throws away all the skipped rows. More skipping = more wasted work.

The Core Problem

With 1M+ rows, page 5000 can take 3-5 seconds while page 1 takes 2ms. The DB literally reads and throws away 100,000 rows just to find the 20 you want.

D Advanced — Cursor-Based Pagination

Instead of page numbers, use the last item's ID as a "cursor". This is how Instagram, Twitter, and Stripe work.

GET /menu/items/?cursor=eyJpZCI6MjB9
"Give me items AFTER item #20" → No skipping, just a WHERE clause

OFFSET 100000

SELECT * FROM items
LIMIT 20 OFFSET 100000;

-- Scans 100,000 rows
-- Throws them all away
-- Returns 20

WHERE id > 100000

SELECT * FROM items
WHERE id > 100000
ORDER BY id LIMIT 20;

-- Index lookup: O(1)
-- Jumps straight to id 100001
-- Returns 20
DRF CursorPagination
▼ 
from rest_framework.pagination import CursorPagination

class FeedCursorPagination(CursorPagination):
    page_size = 20
    ordering = '-created_at'      # Must be an orderable field
    cursor_query_param = 'cursor'  # ?cursor=cD0yMDI0LTA...

class FeedView(generics.ListAPIView):
    queryset = Post.objects.all()
    serializer_class = PostSerializer
    pagination_class = FeedCursorPagination

# Response includes next/previous cursor URLs
# Client just follows the "next" link — no page numbers needed

✅ Pros

  • Fast at any depth — no scanning
  • Consistent results — no duplicates

❌ Cons

  • Can't jump to page 50
  • Must scroll sequentially

Best for: Infinite scroll feeds (Instagram, Twitter), mobile apps

You're building a social media feed (like Instagram). Should you use page numbers or cursor pagination? Why?

Reveal Answer

Cursor. New posts are being added constantly. With page numbers, scrolling to page 2 might show duplicates from page 1 (because new posts pushed everything down). Cursor says "show me everything AFTER the last post I saw" — no duplicates, consistent experience.

Deep Dive

Searching — From LIKE to Semantic

Five levels of search, from basic SQL to AI-powered semantic matching.

MCQ: You have a name column with a B-tree index. Which query can use the index?

A) WHERE name LIKE '%chicken%'
B) WHERE name LIKE 'chicken%'
C) WHERE name LIKE '%chicken'
D) Both B and C

Reveal Answer

B) LIKE 'chicken%' — Only a prefix search (no leading wildcard) can use a B-tree index. The index is sorted alphabetically, so the DB can binary-search to "chicken" and scan forward. A leading % means "could start with anything" — the index is useless, full table scan.

MCQ: You add db_index=True to the name field. Your search query is WHERE LOWER(name) LIKE 'chicken%'. Will the index be used?

A) Yes — it's a prefix search, indexes always help
B) No — LOWER() wraps the column, so the index is bypassed
C) Only on PostgreSQL, not on MySQL
D) Only if the column is VARCHAR, not TEXT

Reveal Answer

B) No. Wrapping a column in a function like LOWER(name) means the DB can't use the regular B-tree index on name — it would need to compute LOWER() for every row first. This is why Django's icontains (case-insensitive) is slower than contains (case-sensitive). Fix: create a functional index on LOWER(name) in PostgreSQL, or use a case-insensitive collation.

A Classic Search — LIKE '%query%'

How it works in Django ORM: filter(name__icontains='chicken')
SQL generated: WHERE LOWER(name) LIKE '%chicken%'

DRF SearchFilter — The Easy Way
▼ 
from rest_framework import filters

class MenuItemListView(generics.ListAPIView):
    queryset = MenuItem.objects.select_related('category').all()
    serializer_class = MenuItemSerializer
    pagination_class = MenuPagination
    filter_backends = [filters.SearchFilter]
    search_fields = ['name', 'description', 'category__name']

# Now clients can search:
# GET /menu/items/?search=chicken
# GET /menu/items/?search=butter chicken  (matches "butter" OR "chicken")

Search Field Prefixes

Prefix SQL Lookup URL Example What It Matches
(none) icontains
LIKE '%x%'		 /menu/items/?search=chicken "chicken" matches "Butter Chicken", "Chicken Biryani", "Grilled Chicken Salad"
^ istartswith
LIKE 'x%'		 /menu/items/?search=butter "butter" matches "Butter Chicken", "Butter Naan" — NOT "Paneer Butter Masala"

The URL is always the same (?search=...) — the prefix in search_fields changes how DRF interprets it on the backend.

Mixing prefixes — real-world example
▼ 
class MenuItemListView(generics.ListAPIView):
    queryset = MenuItem.objects.select_related('category').all()
    serializer_class = MenuItemSerializer
    filter_backends = [filters.SearchFilter]

    search_fields = [
        '^name',             # startswith — fast, uses index
                              # "butter" → finds "Butter Chicken", "Butter Naan"
                              # but NOT "Paneer Butter Masala"

        'description',       # icontains (default) — slower, full scan
                              # "spicy" → finds anything with "spicy" anywhere

        '=category__name',  # exact match on category
                              # "Desserts" → only Desserts category, not "Dessert Pizza"
    ]

# When to use what:
#
# ^  (startswith)  → Use on name/title fields. FASTEST — can use DB index.
#                    Best for: autocomplete-style "type and match" UX.
#                    Example: Amazon product name search.
#
# (none) icontains → Use on description/body fields where match can be anywhere.
#                    SLOWEST — full table scan, but most flexible.
#                    Example: searching blog post content.
#
# =  (exact)       → Use on category/status/type fields with known values.
#                    FASTEST — simple equality check, always uses index.
#                    Example: filtering by exact category name.
#
# @  (full-text)   → Use on large text fields in PostgreSQL.
#                    FAST — uses GIN/GiST index, understands word stems.
#                    Example: searching articles, product descriptions at scale.
#
# Performance ranking:  =  >  ^  >  @  >  (none/icontains)
#                      fastest              slowest

This works great for < 10K items.

But Wait — Should Search Be GET or POST?

Our DRF search uses GET /menu/items/?search=chicken. But Elasticsearch uses POST /_search with a JSON body. Why?

The Intuition

Simple search fits in a URL: ?search=chicken&price_max=300 — easy.
Complex search doesn't: Imagine a query with 5 boolean conditions, 3 nested filters, geo-distance, aggregations, and custom scoring. That JSON body could be 500+ characters — browser URLs cap at ~2000 chars and get unreadable fast. So Elasticsearch, GraphQL, and Algolia use POST — not because it's "better", but because the query body simply doesn't fit in a URL.

🗃️ Regular Database Index

D The Limitation — Lexical Search Only

Everything above — LIKE, full-text search, Elasticsearch — is lexical search. It matches exact words and tokens.

The Problems with Lexical Search

  • "laptop" won't match "notebook computer" even though they mean the same thing
  • "best chicken" won't rank "Butter Chicken (most popular)" higher than "Chicken Salad"
  • No understanding of meaning, synonyms, or intent

E The Prashant-Croissant Problem — Phonetic Search

🔎 🥐

"Prashant" ↔ "Croissant"

You type "prashant" into Google. It suggests "croissant".
You type "chiken tikka" on Swiggy. It shows "Chicken Tikka".

How? Phonetic matching + fuzzy search.

Our LIKE '%query%' returns ZERO results for "chiken" (misspelled).

Watch: The Prashant-Croissant Controversy →

Phonetic search converts words to phonetic codes and matches words that SOUND similar, regardless of spelling.

Algorithm How It Works Example
Soundex Maps to a 4-character code based on consonant sounds Prashant → P623, Croissant → C625
Metaphone Better English phonetics, handles silent letters Smith → SM0, Schmidt → XMT
Double Metaphone Handles multiple pronunciations per word Returns primary + alternate code
Caverphone Optimized for matching names Used in NZ electoral rolls
Beider-Morse Multi-language phonetic matching Handles names across languages

Used in: spell checkers, "Did you mean...?" suggestions, name matching in databases, Elasticsearch phonetic plugin.

F Semantic Search — The Future

Beyond matching words — understanding MEANING. "laptop" matches "notebook computer" because they mean the same thing.

How Semantic Search Works
🔍 Query "spicy chicken"
➔
🧠 Embedding Model ML transforms text to vector
➔
🔢 Vector [0.8, 0.2, 0.1, ...]
↓
🔢 Compare Vectors Cosine similarity
➔

➔

Where Semantic Search is Used

  • ChatGPT retrieval — finds relevant documents to answer your question
  • Google Search (BERT) — understands search intent, not just keywords
  • Amazon product search — "something to keep me warm" matches "woolen sweater"

Tools: OpenAI Embeddings, Sentence-BERT, Pinecone, Weaviate, pgvector

G Search Quiz

Q1: A user searches for "chiken tikka" (misspelled). Which search approach finds the right result?

Reveal Answer

Phonetic search (Soundex/Metaphone) or fuzzy search (Levenshtein distance). Lexical search (LIKE or Elasticsearch exact match) would return nothing because "chiken" doesn't match "chicken".

Q2: A user searches for "something to keep me warm in winter". Your store has "woolen sweaters". Which approach matches this?

Reveal Answer

Semantic search. The query has no words in common with "woolen sweaters", but the MEANING is related. Only embedding-based search catches this.

Q3: A user searches for "paneer" in a restaurant menu with 500 items. Which approach is sufficient?

Reveal Answer

Simple LIKE / icontains. 500 items is tiny — no need for Elasticsearch or embeddings. DRF's SearchFilter is more than enough.

Sort

Sorting

Let users control the order of results — cheapest first, newest first, A-Z.

GET /menu/items/?ordering=-price
ordering=price → cheapest first  |  ordering=-price → most expensive first  |  - means descending
views.py — Add Sorting with OrderingFilter
▼ 
from rest_framework import filters

class MenuItemListView(generics.ListAPIView):
    queryset = MenuItem.objects.select_related('category').all()
    serializer_class = MenuItemSerializer
    pagination_class = MenuPagination
    filter_backends = [filters.SearchFilter, filters.OrderingFilter]
    search_fields = ['name', 'description', 'category__name']
    ordering_fields = ['name', 'price', 'created_at']  # Whitelist!
    ordering = ['-created_at']   # Default sort
GET /menu/items/ → Default: newest first
GET /menu/items/?ordering=price → Cheapest first
GET /menu/items/?ordering=-price → Most expensive first
GET /menu/items/?ordering=name → A-Z
GET /menu/items/?ordering=category__name,price → By category, then price

⚠️ Always Whitelist ordering_fields

Without ordering_fields, DRF allows sorting by any field — including sensitive ones. A malicious user could probe for field names. Always explicitly list which fields are sortable.

Your menu has 1 million items. GET /menu/items/?ordering=price takes 3 seconds. What's wrong and how do you fix it?

Reveal Answer

The price column has no database index. Without an index, the DB must sort ALL 1 million rows before returning the first 20.

Fix: Add a database index:

price = models.DecimalField(..., db_index=True)

With the index, the DB reads the already-sorted index and returns 20 items in 2ms instead of 3s.

Search + Sort + Pagination — one powerful query.

GET /menu/food/? search=paneer ordering=price page=1 size=5
Search for "paneer" → sort by price (cheapest) → first page → 5 per page

Performance Tips

🗃️
Database Indexes
Add db_index=True on columns you sort or filter by. Without indexes, sorting 1M rows is a full table scan.
🔗
select_related
Pagination doesn't fix N+1. Always use select_related('category') for ForeignKey joins in paginated views.
🔢
COUNT(*) is Expensive
Offset pagination needs total count. On huge tables, consider cursor pagination or caching the count.
🔒
max_page_size
Always cap page size. Without max_page_size, a client can request ?size=1000000 and crash your server.

Key Takeaways

  • Always paginate list endpoints — never return unbounded results
  • Offset pagination for admin/dashboards, cursor for feeds/mobile
  • SearchFilter + OrderingFilter — both built into DRF, 3 lines each
  • Whitelist fields — always specify search_fields and ordering_fields
  • Add indexes on columns you search, sort, or filter by
  • select_related — pagination doesn't fix N+1
  • Cap page size with max_page_size
