### 1.Scalability :

Scalability means the ability of a system to handle increased load or traffic without performance degradation — and ideally, with minimal increase in cost and complexity. There are two main types:

**1. Vertical Scaling :** you boost your server's capacity with more CPU, memory, or storage. It's good for smaller apps, but it has limits because you can't infinitely upgrade hardware.
    
**2. Horizontal Scaling :** You add more servers/nodes to distribute load. The system handles requests across multiple machines.

* **main strategies used to Achieve Scalability :**

    - **1. Load Balancing :**
      - What: Distributes incoming traffic evenly across multiple servers.
      - Why: Prevents any single server from becoming a bottleneck.
      - Tools: Nginx, HAProxy, AWS Elastic Load Balancer, Cloudflare.
         
    - **2. Caching :**
      - What: Store frequently accessed data in fast memory (RAM) for quick retrieval.
      - Why: Reduces database and computation load.
      - Types:
        - Client-side caching (browser cache)
        - CDN caching (static assets)
        - Application caching (in-memory results)
        - Database query caching
        - Tools: Redis, Memcached, Varnish, Cloudflare CDN, Akamai.
          
    - **3. Database Replication :**
      - Create multiple copies of the same database. App sends write queries to master-db and read queries to replica-db.
      - Master node handles writes.
      - Slave (read replica) nodes handle reads
      - Result: Scales read-heavy workloads.
      - Tools: MySQL Replication, PostgreSQL Streaming Replication, MongoDB Replica Sets.

    - **4. Database Sharding (Partitioning) :**
      - What: Split large databases into smaller, independent chunks (shards).
      - Why: Each shard handles only part of the data → less load per node.
      - Types:
        - Horizontal sharding: Split by user ID, region, etc.
        - Vertical sharding: Split tables by function (e.g., users in one DB, orders in another).

    - **5. Horizontal Scaling (Scale Out) :**
      - What: Add more servers to handle load instead of upgrading existing ones.
      - Why: Cost-effective, fault-tolerant, cloud-friendly.
      - How:
        - Stateless app servers
        - Shared data stores (cache/DB)
        - Load balancing in front
       
    - **6. Stateless Architecture :**
      - What: Keep no user/session data on the server itself.
      - Why: Allows easy scaling — any server can handle any request.
      - How:
        - Use cookies or JWTs for session tokens.
        - Store session data in Redis or database.
        - Example: Web server doesn’t remember users — authentication happens via tokens.
       
    - **7. Asynchronous Processing / Message Queues :**
      - What: Offload heavy or time-consuming tasks to background workers.
      - Why: Makes frontend requests fast and scalable.
      - Tools: Kafka, RabbitMQ, Celery, SQS.
      - Use cases:
        - Sending emails
        - Processing images/videos
        - Updating analytics
        - Example: When a user uploads a file → immediately return response → process file in background.

    - **8. Microservices Architecture :**
      - What: Break large monolithic app into smaller, independent services.
      - Why: Each service can scale independently and be deployed separately.
      - Tools: Docker, Kubernetes, Istio.
      - Example: Separate auth-service, payment-service, notification-service.
     
    - **9. Distributed Storage Systems :**
      - What: Store data across multiple nodes for scalability and fault tolerance.
      - Why: Handle large data volumes efficiently.
      - Tools: Amazon S3, HDFS, Ceph, Cassandra, DynamoDB.
      - Example: Millions of images/videos stored in S3 with replication.


    - **10. Auto Scaling and Monitoring :**
      - What: Automatically adjust resources (servers/containers) based on load.
      - Why: Cost optimization + performance stability.
      - Tools: AWS Auto Scaling Groups, Kubernetes HPA, Prometheus + Grafana for metrics.
      - Example: Scale from 5 to 50 instances during traffic spikes, then back down.
     
    - **11. Content Delivery Networks (CDNs) :**
      - What: Cache and deliver static content (images, videos, JS, CSS) from geographically distributed edge servers.
      - Why: Reduces latency and backend load.
      - Tools: Cloudflare, Akamai, AWS CloudFront.
      - Example: Users in India fetch images from Cloudflare Mumbai node instead of your U.S. server.
     
    - **12. API Gateway and Rate Limiting :**
      - What: Manage incoming API requests, throttle excessive usage, and secure endpoints.
      - Why: Prevents overload and abuse.
      - Tools: Kong, NGINX, AWS API Gateway.


| Layer                     | Strategy                               | Description                                                                                         |
| ------------------------- | -------------------------------------- | --------------------------------------------------------------------------------------------------- |
| **Frontend**              | **Caching (CDN, Browser Cache)**       | Serve static content (images, JS, CSS) faster and reduce backend load.                              |
| **Application Layer**     | **Load Balancing**                     | Distribute traffic evenly across multiple app servers (e.g., Nginx, HAProxy, AWS ELB).              |
| **Database Layer**        | **Database Replication**               | Create read replicas to scale reads; master node handles writes.                                    |
| **Database Layer**        | **Sharding (Partitioning)**            | Split data horizontally (by user ID, region, etc.) across multiple DBs to handle large data volume. |
| **Storage Layer**         | **Distributed File Systems**           | Use S3, HDFS, or similar to scale storage independently.                                            |
| **Background Processing** | **Message Queues (Kafka, RabbitMQ)**   | Decouple heavy tasks from user-facing requests.                                                     |
| **Caching Layer**         | **In-memory Cache (Redis, Memcached)** | Reduce DB load by caching frequent queries or results.                                              |
| **Microservices**         | **Service-level scaling**              | Each service can scale independently based on load.                                                 |
| **Auto-scaling**          | **Cloud scaling policies**             | Automatically add/remove instances based on CPU/memory/load.                                        |

### 2.Availability :  
Availability in system design refers to how consistently a system is operational and accessible when needed. In simple terms — a highly available system is one that stays “up” and serving users most of the time, even in the face of failures. 

**Key Strategies to Achieve High Availability :**

- **1. Redundancy :**
  - Have multiple instances of critical components. If one fails, another takes over.
  - Example: Two web servers behind a load balancer; if one crashes, traffic is redirected to the other.

- **2. Failover Mechanisms :**
  - Automatic switching to a backup system/component when a failure is detected.
  - Example: Primary database fails → system switches to a standby replica.
    
- **3. Load Balancing :**
  - Distributes incoming traffic across multiple servers.
  - Prevents overloading one server and ensures continuous service.
  - Tools: NGINX, HAProxy, AWS ELB, Google Cloud Load Balancer

- **4. Replication :**
  - Keep copies of data across multiple nodes or regions.
  - Ensures data availability even if one node fails.
  - Example: Database replication (master-slave or multi-master setups).

- **5. Health Checks & Monitoring :**
  - Regularly test system components (via heartbeats, health probes).
  - Use monitoring tools to detect issues early.
  - Tools: Prometheus, Grafana, Datadog, CloudWatch

- **6. Eliminate Single Points of Failure (SPOF) :**
  - Any component that can bring down the whole system if it fails is a SPOF.
  - Use redundancy or partitioning to remove them.
  - Example: Use multiple availability zones or backup load balancers.

- **7. Geographic Distribution :**
  - Deploy systems across multiple data centers or regions.
  - Protects against natural disasters, regional outages.
  - Example: Multi-region architecture on AWS (us-east-1 + ap-south-1).
 
- **8. Graceful Degradation :**
  - System continues to work in a limited mode instead of total failure.
  - Example: When recommendation service is down, show static products instead of an error.

- **9. Automated Recovery :**
  - Use orchestration tools like Kubernetes or Auto Scaling Groups to automatically restart or replace failed instances.

- **10. Backups & Disaster Recovery :**
  - Regular data backups and clear disaster recovery plans.
  - Helps recover from large-scale failures.
