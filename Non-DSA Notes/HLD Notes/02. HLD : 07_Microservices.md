## * Monolithic : 
Single codebase that consists of all the services/modules

* Pros :
  - Single Deployment
  - Different services can communicate with each other via a simple function call. (No Network calls/overhead)
  - End to end testing becomes easy.
  - Easy to maintain and monitor
  - Easy to Debug
 
* Cons :
  - Deployment will take huge time.
  - A small issue in one of the service/module can bring entire application down - (Cascading Failure)
  - No Tech stack flexibility
  - Developer Onboarding becomes diffficult
  - Tight coupling between services.
  - No slective scaling of individual services/modules.
    - for ex: amount of traffic on search service >> amount of traffic on order service.
   
## * Microservices :
Different codebases for different services. 

* Pros :
  - No cascading failure
  - Selective scaling is possible
  - Developer onboarding is possible
  - Tech stack flexibility
  - Individual deployment of services
  - Bug fixing becomes easy
 
* Cons :
  - Lots of Deployments
  - Monitoring becomes difficult
  - Different services will talk to each other via network call, so that will add extra latency
  - Testing becomes difficult

## * Distributed Transactions :
In the context of microservices, maintaining consistency across different services is crucial but also challenging. This session delves into distributed transactions, particularly the **Two-Phase Commit and SAGA patterns**, which help ensure consistency across multiple services in a distributed system.

* Consistency in Microservices :
  - Monolithic Systems: In a monolith with a single database, transactions and consistency are maintained using ACID properties. For instance, transferring money between two accounts involves a single transaction that guarantees atomicity, consistency, isolation, and durability.
  - Distributed Systems Challenges: When dealing with transactions across services, such as transferring money between accounts in different banks or maintaining stock levels across inventory and sales systems, achieving consistency becomes complex. This introduces the need for distributed transaction management .


* Two-Phase Commit (2PC) : Two-Phase Commit is a protocol that ensures all parts of a distributed transaction either commit or roll back operations to maintain consistency. The process involves:
  - Voting Phase (Are You Ready?): All participating services check if they’re ready to commit the transaction, essentially taking a lock to prevent other operations. If any service cannot commit, the transaction is rolled back.
  - Commit Phase: Upon receiving affirmative votes from all services, the transaction is committed, and the changes are persisted across services. If any service fails during the commitment, all services must roll back their changes .
  - Downsides :
    - Long-Lasting Locks: Holding locks for an extended period can degrade performance, leading to reduced system throughput and slower user response times.
    - Anti-Pattern in Modern Microservices: Due to the downsides, such as reduced concurrency and increased wait times, 2PC is often considered an anti-pattern in microservice architectures .

* SAGA Pattern : The SAGA pattern breaks down a distributed transaction into a sequence of smaller transactions, each performed by a single service. It involves compensatory transactions that undo changes if a subsequent transaction fails, thus ensuring consistency without long-held locks.
  - Choreography: Each service performs its transaction and publishes an event for the next transaction, avoiding a single point of failure .
  - Orchestration: A centralized coordinator manages the sequence of operations, ensuring that compensations are performed if a step fails .
  - Benefits :
    - Improved Resilience and Scalability: Services can operate more independently and handle failures locally, allowing for better fault tolerance and system scalability .
    - Avoidance of Long-Held Locks: As each transaction is isolated and local to a service, there are no prolonged locks that affect system performance.


* CQRS : command query request segregation
  - cqrs pattern is an idea of seggragating read and write requests/operations.
  - to follow cqrs pattern, we add different machines for read and write operations (slave/cache/CDN)
  - it is used for eventual consistency

* Circuit Breaker Pattern :
  - If notification service is trying to come up and all other services attacks/bombards notification service with all pending requests then the chances of it going down will be very high (Thundering Herd effect) and new requests will be waiting till the older requests are getting processed. it has 2 categories of errors :
    - Regular error : usernotfound/auth/..
    - Non Transient errrors : Internal server error, timeout error
   
* if the rate of non transient error goes very high, it means one service is not healthy. if service is not healthy -> break the circuit.
  - closed state : service is healthy + requests can flow
  - open state : service is unhealthy + break the circuit
  - half open state : testing/monitoring phase - serivce is coming up, send the requests in controlled manner.
