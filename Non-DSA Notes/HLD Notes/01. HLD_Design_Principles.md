### 1.Scalability :

Scalability means the ability of a system to handle increased load or traffic without performance degradation — and ideally, with minimal increase in cost and complexity. There are two main types:

**1. Vertical Scaling :** you boost your server's capacity with more CPU, memory, or storage. It's good for smaller apps, but it has limits because you can't infinitely upgrade hardware.
    
**2. Horizontal Scaling :** You add more servers/nodes to distribute load. The system handles requests across multiple machines.

* **main strategies used to Achieve Scalability :**

    - **1. Load Balancing :**
      - What: Distributes incoming traffic evenly across multiple servers.
      - Why: Prevents any single server from becoming a bottleneck.
      - Tools: Nginx, HAProxy, AWS Elastic Load Balancer, Cloudflare.
         
    - **2. Caching :**
      - What: Store frequently accessed data in fast memory (RAM) for quick retrieval.
      - Why: Reduces database and computation load.
      - Types:
        - Client-side caching (browser cache)
        - CDN caching (static assets)
        - Application caching (in-memory results)
        - Database query caching
        - Tools: Redis, Memcached, Varnish, Cloudflare CDN, Akamai.
          
    - **3. Database Replication :**
      - Create multiple copies of the same database. App sends write queries to master-db and read queries to replica-db.
      - Master node handles writes.
      - Slave (read replica) nodes handle reads
      - Result: Scales read-heavy workloads.
      - Tools: MySQL Replication, PostgreSQL Streaming Replication, MongoDB Replica Sets.

    - **4. Database Sharding (Partitioning) :**
      - What: Split large databases into smaller, independent chunks (shards).
      - Why: Each shard handles only part of the data → less load per node.
      - Types:
        - Horizontal sharding: Split by user ID, region, etc.
        - Vertical sharding: Split tables by function (e.g., users in one DB, orders in another).

    - **3. Database Replication :**
      - Create multiple copies of the same database. App sends write queries to master-db and read queries to replica-db.
      - Master node handles writes.
      - Slave (read replica) nodes handle reads
      - Result: Scales read-heavy workloads.
      - Tools: MySQL Replication, PostgreSQL Streaming Replication, MongoDB Replica Sets.

    - **5. Horizontal Scaling (Scale Out) :**
      - What: Add more servers to handle load instead of upgrading existing ones.
      - Why: Cost-effective, fault-tolerant, cloud-friendly.
      - How:
        - Stateless app servers
        - Shared data stores (cache/DB)
        - Load balancing in front
       
    - **6. Stateless Architecture :**
      - What: Keep no user/session data on the server itself.
      - Why: Allows easy scaling — any server can handle any request.
      - How:
        - Use cookies or JWTs for session tokens.
        - Store session data in Redis or database.
        - Example: Web server doesn’t remember users — authentication happens via tokens.
       
    - **7. Asynchronous Processing / Message Queues :**
      - What: Offload heavy or time-consuming tasks to background workers.
      - Why: Makes frontend requests fast and scalable.
      - Tools: Kafka, RabbitMQ, Celery, SQS.
      - Use cases:
        - Sending emails
        - Processing images/videos
        - Updating analytics
        - Example: When a user uploads a file → immediately return response → process file in background.

    - **8. Microservices Architecture :**
      - What: Break large monolithic app into smaller, independent services.
      - Why: Each service can scale independently and be deployed separately.
      - Tools: Docker, Kubernetes, Istio.
      - Example: Separate auth-service, payment-service, notification-service.
     
    - **9. Distributed Storage Systems :**
      - What: Store data across multiple nodes for scalability and fault tolerance.
      - Why: Handle large data volumes efficiently.
      - Tools: Amazon S3, HDFS, Ceph, Cassandra, DynamoDB.
      - Example: Millions of images/videos stored in S3 with replication.


    - **10. Auto Scaling and Monitoring :**
      - What: Automatically adjust resources (servers/containers) based on load.
      - Why: Cost optimization + performance stability.
      - Tools: AWS Auto Scaling Groups, Kubernetes HPA, Prometheus + Grafana for metrics.
      - Example: Scale from 5 to 50 instances during traffic spikes, then back down.
     
    - **11. Content Delivery Networks (CDNs) :**
      - What: Cache and deliver static content (images, videos, JS, CSS) from geographically distributed edge servers.
      - Why: Reduces latency and backend load.
      - Tools: Cloudflare, Akamai, AWS CloudFront.
      - Example: Users in India fetch images from Cloudflare Mumbai node instead of your U.S. server.
     
    - **12. API Gateway and Rate Limiting :**
      - What: Manage incoming API requests, throttle excessive usage, and secure endpoints.
      - Why: Prevents overload and abuse.
      - Tools: Kong, NGINX, AWS API Gateway.



| Layer                     | Strategy                               | Description                                                                                         |
| ------------------------- | -------------------------------------- | --------------------------------------------------------------------------------------------------- |
| **Frontend**              | **Caching (CDN, Browser Cache)**       | Serve static content (images, JS, CSS) faster and reduce backend load.                              |
| **Application Layer**     | **Load Balancing**                     | Distribute traffic evenly across multiple app servers (e.g., Nginx, HAProxy, AWS ELB).              |
| **Database Layer**        | **Database Replication**               | Create read replicas to scale reads; master node handles writes.                                    |
| **Database Layer**        | **Sharding (Partitioning)**            | Split data horizontally (by user ID, region, etc.) across multiple DBs to handle large data volume. |
| **Storage Layer**         | **Distributed File Systems**           | Use S3, HDFS, or similar to scale storage independently.                                            |
| **Background Processing** | **Message Queues (Kafka, RabbitMQ)**   | Decouple heavy tasks from user-facing requests.                                                     |
| **Caching Layer**         | **In-memory Cache (Redis, Memcached)** | Reduce DB load by caching frequent queries or results.                                              |
| **Microservices**         | **Service-level scaling**              | Each service can scale independently based on load.                                                 |
| **Auto-scaling**          | **Cloud scaling policies**             | Automatically add/remove instances based on CPU/memory/load.                                        |

### 2.Availability :  
Availability in system design refers to how consistently a system is operational and accessible when needed. In simple terms — a highly available system is one that stays “up” and serving users most of the time, even in the face of failures. 

**Key Strategies to Achieve High Availability :**

- **1. Redundancy :**
  - Have multiple instances of critical components. If one fails, another takes over.
  - Example: Two web servers behind a load balancer; if one crashes, traffic is redirected to the other.

- **2. Failover Mechanisms :**
  - Automatic switching to a backup system/component when a failure is detected.
  - Example: Primary database fails → system switches to a standby replica.
    
- **3. Load Balancing :**
  - Distributes incoming traffic across multiple servers.
  - Prevents overloading one server and ensures continuous service.
  - Tools: NGINX, HAProxy, AWS ELB, Google Cloud Load Balancer

- **4. Replication :**
  - Keep copies of data across multiple nodes or regions.
  - Ensures data availability even if one node fails.
  - Example: Database replication (master-slave or multi-master setups).

- **5. Health Checks & Monitoring :**
  - Regularly test system components (via heartbeats, health probes).
  - Use monitoring tools to detect issues early.
  - Tools: Prometheus, Grafana, Datadog, CloudWatch

- **6. Eliminate Single Points of Failure (SPOF) :**
  - Any component that can bring down the whole system if it fails is a SPOF.
  - Use redundancy or partitioning to remove them.
  - Example: Use multiple availability zones or backup load balancers.

- **7. Geographic Distribution :**
  - Deploy systems across multiple data centers or regions.
  - Protects against natural disasters, regional outages.
  - Example: Multi-region architecture on AWS (us-east-1 + ap-south-1).
 
- **8. Graceful Degradation :**
  - System continues to work in a limited mode instead of total failure.
  - Example: When recommendation service is down, show static products instead of an error.

- **9. Automated Recovery :**
  - Use orchestration tools like Kubernetes or Auto Scaling Groups to automatically restart or replace failed instances.

- **10. Backups & Disaster Recovery :**
  - Regular data backups and clear disaster recovery plans.
  - Helps recover from large-scale failures.

## 3. Consistency : 

Consistency refers to the uniformity and accuracy of data across a distributed system — ensuring that all nodes (servers or replicas) see the same data at the same time (or eventually).

**Types of Consistency :**
There are two broad categories — **Strong** and **Eventual**, with several variants in between.

- **1. Strong Consistency :**
  - After an update completes, all nodes see the same updated value immediately.
  - Like a single system — reads always return the latest write.
  - Use Case: Banking transactions, inventory management, or systems needing strict accuracy. **Ex : traditional RDBMS (e.g., PostgreSQL, MySQL in single-node mode).**
    > **How to achieve :**
  - Use a single leader (master) for all writes.
  - Apply synchronous replication — write only succeeds when all replicas confirm.
  - Use distributed consensus algorithms like Paxos or Raft.


- **2. Eventual Consistency :**
  - Updates propagate asynchronously, and replicas converge to the same state over time.
  - A read may return stale data temporarily.
  - Use Case: Social media likes, product views, caching systems, DNS. Example: Amazon DynamoDB, Cassandra, Couchbase.
    **How to achieve :**
  - Use asynchronous replication.
  - Implement read repair, hinted handoff, or anti-entropy processes to sync nodes.
  - Combine with versioning (vector clocks, timestamps) to resolve conflicts.

- **3. Causal Consistency :**
  - If one operation causes another, all nodes must observe them in that order.
  - Preserves cause-and-effect relationships but allows concurrent updates.
  - Use Case: Chat apps (message order), collaborative editing. Example: COPS system, Redis CRDTs.
   **How to achieve :**
  - Use causal metadata (timestamps or vector clocks) to track dependencies.
  - Delay reads until all causal dependencies are visible.


- **4. Read-Your-Writes Consistency :**
  - Definition: Once you write something, you’ll always read your own latest data (even if others see stale data).
  - Use Case: User profiles, dashboards showing your own actions. Example: Facebook timeline after posting.
    **How to achieve :**
  - Route a user’s reads/writes to the same replica or session.
  - Use session stickiness or session tokens.

- **5. Session Consistency :**
  - Definition: Ensures consistency within a single session (like user login session).
  - Use Case: Shopping carts, personal settings.
  - How to achieve: Keep all operations of one session routed to the same node.


* **How to achieve Consistency :**

| Strategy                             | Description                                              | Used In                   |
| ------------------------------------ | -------------------------------------------------------- | ------------------------- |
| **Replication Protocols**            | Control how writes are propagated (synchronous vs async) | Databases, message queues |
| **Leader-Follower Model**            | One node handles writes; followers replicate             | PostgreSQL, MongoDB       |
| **Consensus Algorithms**             | Ensure all nodes agree on data order/state               | Raft, Paxos, ZAB          |
| **Quorum-Based Writes/Reads**        | Require a majority of nodes to agree (e.g., R + W > N)   | Cassandra, DynamoDB       |
| **Versioning & Conflict Resolution** | Track versions using vector clocks/timestamps            | Dynamo, Riak              |
| **Read Repair / Anti-Entropy**       | Background processes fix stale replicas                  | Cassandra                 |
| **Session Stickiness**               | Direct same user requests to same replica                | Web apps, CDNs            |

### 4. Reliability : 


Reliability in system design refers to the ability of a system to consistently perform its intended functions correctly and without failure over time. A reliable system continues to work as expected, even when parts of it fail.

* **How to Achieve Reliability :**

 - **1. Redundancy :**
     - Duplicate critical components (servers, databases, network paths, etc.) so that if one fails, another takes over.
     - Example:
      - Use replica databases for failover.
      - Load balancers distribute requests among multiple servers.


 - **2. Failover and Replication :**
      - Automatically switch to a standby or backup system when a primary system fails.
      - Example:
        - Primary DB → Secondary DB (Hot Standby)
        - Active-Passive or Active-Active setups.

- **3. Monitoring and Alerting :**
     - Continuously track system health and performance.
     - Use tools like Prometheus, Grafana, Datadog, or AWS CloudWatch.
     - Detect issues early before they cause downtime.
    
- **4. Graceful Degradation :**
     - The system should partially function when some components fail, instead of crashing completely.
     - Example: If recommendation service is down, show cached results instead of an error.

- **5. Retry and Backoff Strategies :**
     - Implement automatic retries with exponential backoff for transient failures (like temporary network errors).
     - Prevents overloading the system with repeated requests.

- **6. Data Replication and Backups :**
     - Regularly back up data to different locations or data centers.
     - Use distributed storage systems (e.g., S3, Cassandra, HDFS) to prevent single-point data loss.

- **7. Fault Isolation :**
     - Design using microservices or modular architecture so that a fault in one service doesn’t take down the entire system.


- **8. Testing and Chaos Engineering :**
     - Test for reliability proactively:
       - Load Testing – check system behavior under heavy load.
       - Failure Injection / Chaos Testing – simulate real failures (e.g., using Netflix’s Chaos Monkey).
         
- **9. Circuit Breaker Pattern :**
     - Prevent cascading failures by temporarily stopping requests to a failing service.
     - Example: Use libraries like Hystrix or Resilience4j in microservice systems.
      
- **10. Idempotent Operations :**
     - Ensure operations can be safely retried without unintended effects.
     - Example: Re-sending a payment request shouldn’t charge twice.

### 5. Fault Tolerance :

Fault tolerance is the ability of a system to continue operating properly even if one or more of its components fail. The goal is to prevent total system failure and ensure high availability, reliability, and resilience. A fault-tolerant system is designed so that failures are expected, detected, and managed gracefully — without disrupting the overall service.
Example: If one server in a cluster fails, another automatically takes over with minimal or no downtime.

* **Techniques to Achieve Fault Tolerance :**
  
* **1. Redundancy :**
     - Hardware Redundancy: Duplicate power supplies, disks (RAID), or servers.
     - Software Redundancy: Multiple instances of a service running in parallel.
     - Network Redundancy: Multiple routes and connections.
     - Example: In AWS, deploying across multiple Availability Zones (AZs) ensures redundancy.

* **2. Replication :**
     - Data Replication: Keeping identical data on multiple nodes or datacenters.
     - State Replication: Synchronizing system state across nodes (e.g., leader-follower models).
     - Example: In distributed databases like MongoDB or Cassandra, replicas ensure continued data access during node failures.
    
* **3. Failover & Failback :**
     - Failover: Automatically routing traffic to a backup system.
     - Failback: Switching back to the primary once it’s healthy again.
     - Example: A load balancer detects that a backend server is down and redirects requests to another.
    
* **4. Graceful Degradation :**
     - Design so that non-critical services can fail without impacting the core functionality.
     - Example: If a recommendation engine fails on an e-commerce site, users can still browse and purchase items.
    
* **5. Checkpointing and Rollback :**
     - Save system state periodically.
     - On failure, restart from the last checkpoint instead of from scratch.
     - Example: In distributed computing frameworks like Hadoop, tasks are retried from last checkpoint.
    
* **6. Isolation :**
     - Ensure failures don’t cascade:
     - Use microservices to isolate faults in specific services.
     - Use circuit breakers (e.g., in Netflix Hystrix) to prevent failing dependencies from overloading the system.
    
* **7. Monitoring & Alerts :**
     - Continuous health monitoring allows fast fault detection and automated remediation.
       - Health checks
       - Heartbeat signals
       - Watchdog timers


* **Fault Tolerance vs. High Availability Load Balancing :**
  - **Fault Tolerance :**
    - Primary Goal: Maintain system functionality despite failures.
    - Key Techniques: Redundancy, replication, failover mechanisms, error detection, and correction.\
    - Redundancy: High level of redundancy (multiple components performing the same task).
    - Examples: RAID (Redundant Array of Independent Disks), Distributed databases with replication.
    - Impact on Performance: May slightly impact performance due to redundancy checks and error handling.

   - **High Availability Load Balancing :**
     - Primary Goal: Maximize uptime and resource utilization by balancing load.
     - Key Techniques: Load distribution algorithms (round-robin, least connections, etc.), health checks, failover.
     - Redundancy: Moderate redundancy (enough to balance the load and ensure availability).
     - Examples: DNS load balancing, Application load balancers (like NGINX, HAProxy).
     - Impact on Performance: Generally improves performance by distributing workload evenly.

### 6. CAP Theorem :

The CAP Theorem (also known as Brewer’s Theorem) is a fundamental concept in distributed system design that explains the trade-offs between **Consistency, Availability, and Partition Tolerance**. It helps architects decide what guarantees a distributed system can or cannot provide when failures occur.
The CAP theorem states that in any distributed data system, it is impossible to simultaneously guarantee all three of the following properties:  


| Property                    | Meaning                                                                                                     |
| --------------------------- | ----------------------------------------------------------------------------------------------------------- |
| **Consistency (C)**         | Every read receives the most recent write or an error — i.e., all nodes see the same data at the same time. |
| **Availability (A)**        | Every request (read/write) receives a valid response, even if some nodes are down.                          |
| **Partition Tolerance (P)** | The system continues to function despite network failures or message loss between nodes.                    |

> **When a network partition (communication failure) occurs between nodes:**  
> **You must choose between Consistency and Availability — you cannot have both.**  
> **That is, a distributed system can only guarantee two out of the three properties at a time:**

  

 Type                                        | Guarantees                                                                                                                       | Example Systems                                                             |
| ------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| **CP (Consistency + Partition Tolerance)**  | Prioritizes data accuracy over availability. If nodes can’t communicate, the system may reject requests to maintain consistency. | HBase, MongoDB (in certain configs), Zookeeper                              |
| **AP (Availability + Partition Tolerance)** | Prioritizes uptime over strict consistency. Nodes may return stale data but always respond.                                      | Cassandra, DynamoDB, Couchbase                                              |
| **CA (Consistency + Availability)**         | Works only when there are **no network partitions**, which is unrealistic in distributed systems.                                | Traditional single-node RDBMS like PostgreSQL, MySQL (non-distributed mode) |



> In real-world distributed systems, **network partitions are inevitable, so Partition Tolerance (P) is a must.**    
> Therefore, the real choice is between:  
    > **Consistency (C)** → Strong data accuracy but lower availability  
    > **Availability (A)** → Higher uptime but eventual consistency  


* Example Scenarios
  - **Banking System (CP System) :**
    - Consistency is critical — you can’t show incorrect balances.
    - If a partition happens, some transactions may be delayed.
    
  - **E-commerce Product Catalog (AP System) :**
    - Availability is more important — users should still browse products even if some updates lag behind.
    - Eventual consistency is acceptable.



* **CP Systems (Consistency + Partition Tolerance) :**
  - Prioritize data correctness over availability.
  - During partition, may deny requests to maintain data consistency.
  - Examples :
    - **Google Spanner**  - Ensures global strong consistency via synchronized clocks (TrueTime).
    - **HBase** - Based on Bigtable; ensures consistent reads/writes.
    - **MongoDB (in “majority” write concern mode)** - Rejects writes if replicas not available.
    - **Zookeeper / etcd** - Ensures consistency in leader election and configuration state.
    - **Redis Sentinel (in quorum mode)** - Prioritizes consistency of leader state. 

  
* **AP Systems (Availability + Partition Tolerance) :**
  - Always respond to requests, even if data might be stale.
  - Often use eventual consistency.
  - Examples :
    - **Amazon DynamoDB**  - Prioritizes uptime and availability, with eventual consistency.
    - **Cassandra** - Uses tunable consistency; often configured as AP.
    - **Couchbase** - Prioritizes availability, eventually syncing data across nodes.
    - **Riak** - Modeled after Amazon Dynamo, AP by design.
    - **Eureka (Netflix)** - Service registry prioritizing availability. 

 
* **CA Systems (Consistency + Availability) :**
  - Only work in absence of partitions.
  - Not truly distributed or rely on a single node or synchronous cluster.
  - Examples :
    - **PostgreSQL / MySQL (single instance)** - Strong consistency, full availability — but not partition tolerant.
    -  **Oracle RDBMS (single-node)** - Traditional relational DB.
    -  **Redis (standalone)**  - Consistent and available until the single node fails. 

---------------

The CAP Theorem says:
In the presence of a Partition (P), a distributed system can choose either Availability (A) or Consistency (C) — but not both
However, CAP doesn’t describe what happens when there is no partition (which is most of the time).
That’s where PACELC comes in.

* **PCAELC Theorem :**
  
**If there is a partition (P) — the system must trade off between Availability (A) and Consistency (C).
Else (E), when the system is running normally — it must trade off between Latency (L) and Consistency (C).**

> **PA/ELC → “If Partition (P) happens → choose A or C; Else (E) → choose Latency or Consistency.”**

* Why PACELC matters :
  - CAP only talks about failure conditions (network partitions).
  - PACELC adds the normal operation trade-offs that affect user experience — like choosing between fast responses (low latency) or strong consistency.

| System Type                            | During Partition (P)                                             | Else (E)                           | Example Systems                  |
| -------------------------------------- | ---------------------------------------------------------------- | ---------------------------------- | -------------------------------- |
| **PA/EL** (Availability + low Latency) | Favors **Availability** during partitions                        | Favors **low Latency** normally    | Amazon DynamoDB, Cassandra, Riak |
| **PC/EC** (Consistency + Consistency)  | Favors **Consistency** during partitions                         | Favors **Consistency** normally    | Google Spanner, Zookeeper, HBase |
| **PA/EC**                              | Availability during partition, but Consistency during normal ops | Cosmos DB (with specific settings) |                                  |
