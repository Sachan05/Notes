* **CACHING :** the first 3 case studies in L3 explains Caching can occur at: Browser, CDN, Application, Database layers.

**1) Case Study - Scaler DSA Problem Submissions :**
   - **Problem :**
     - Each submission needs large input/output files.
     - Fetching them from remote file storage takes ~2 seconds → slow submissions.
     
   - **Explored Approaches :**
     - **a) TTL-based Cache**
       - Low TTL: frequent cache expiry → many misses → still slow
       - High TTL: stale data issue when files are updated
       - Not reliable for frequently changing files
       
     - **b) Global Cache**
       - Storing files in a single central cache causes:
         - Huge network overhead
         - Limited storage (if in-memory)
         - High eviction rate
         - Not suitable for large files
         
     - **c) Local Cache**
       - Fastest access (from local disk)
       - But updating a file across all servers is difficult
       - Invalidation issue


   - **Best Solution: Use File Metadata (Database-driven invalidation)**
     - Store file metadata (path, updated_at, etc.) in a MySQL table.
     - When a submission comes:
       - Check DB metadata (problem_id + updated_at).
       - Look for file with name pattern: problemId_updatedAt_input.txt
       - If exists → use local cached copy.
       - If not → fetch from file storage and save locally.

      - Why this works
        - Always serves the latest file
        - No manual invalidation required
        - Cache invalidation becomes automatic through metadata
        - Efficient local caching without stale data

---------------

**2) Case Study - Contest Rank List :**
   - **Problem :** Rank list updates frequently during a contest. Computing and fetching from DB repeatedly overloads the database.
   
   - **Solution: Global Cache (Redis)**
     - Periodically compute rank list (e.g., every 1 minute)
     - Store this static snapshot in Redis (shared global cache)
     - All app servers read from Redis → only 1 cache miss per minute

   - **Why global cache works well here**
     - Data is the same for everyone
     - Updates are periodic
     - Avoids re-computing or re-fetching on every request

**3) Case Study - Facebook Newsfeed :**
   * **Challenges :**
     - Posts and users are distributed across many machines (sharded)
     - Newsfeed requires posts from all friends → inefficient to fetch from many shards
     - Caching per-user newsfeed is too costly and hard to maintain:
       - Huge storage
       - Thousands of updates per post (fan-out)
       - Hard to change algorithm

  * **Optimized Approach :**
    - Estimate that recent posts (last 30 days) is around 180GB.
    - Store all recent posts in a separate replicated database.
    - Fetch newsfeed efficiently:
      - Get friend_ids
      - Query recent posts from those friends from the “recent posts DB”
        SELECT * FROM all_posts WHERE user_id IN friend_ids LIMIT x OFFSET y
    

  * **Why this works :**
    - Only recent posts matter
    - Disk-based caching is still much faster than remote storage
    - Easy replication to handle high read load
    - No need to pre-compute per-user feeds
    - Old posts can be deleted to save space
    - **This approach uses much lesser storage and approach than the previous system. Here the cache is stored in a hard disk, not in RAM, but still, this is much faster than getting data from an actual storage system.** 




* **Metadata-based invalidation is best for frequently changing large files**
* **Global cache wins when all users need the same data (e.g., rank list)**
* **For heavy, dynamic workloads like newsfeed, store recent data in fast-access storage and query efficiently**

## 4] TYPEAHEAD :


  * **MVP :**
    - Get suggestions for the given prefix
    - Minimum 3 characters would be required to get suggestions
    - Top 5-10 suggestions
    - Suggestions should be based on **popularity (frequency no./ based on no of writes)**

  * **Scale Estimation :**
    - we need to determine  1) No of read/write queries, 2) Amount of Data
    - 
    - No of users = 3 Billion
    - DAU (daily active users) : usually 5-10%. lets take 33% for popular systems like google/FB - 1B
    - Avg. no of searches/user/day - 10
    - No of searches per day - 10 Billion (write queries)
    - As we type each character, typeahead system sends a **Read request** for **get_suggestions**. when we actually press enter and search, that becomes a write request for typeahead/search system. so google search function will act as write query for typeahead system.
    - Avg. no of typeahead suggestions request for every search query - 5
    - No of searches per day = writes per day = 10 B
    - No fo Reads per day = get_suggestions = 50 B
    - **It is a both Read + Write Heavy system** : when the read is abnormally higher like 100/500 times greater than write system, then we say its a read or write heavy system.
    - **QPS : Queries per second**
      - **write QPS :**
        - writes/day = 10 B/(24*60*60) = 10*10^9/86400(10^4)  ~ 10^5
      - **Read QPS :** 5 * write qps ~ 5*10^5
    - **This is the avg traffic, but we need to design for Peak traffic - we roughly take 2-3 times of avg for Peak traffic**
    - 
   
    - **Amount of Data :**
      - write queries = 10 B/day
      - it will consist of exisitng words(we only to update their frequency) + new terms(store them too)
      - New search queries : lets say 10% = 1 B
      - space requeired for storing new word :
        - query(30 bytes) + frequency(8 bytes) + metadata(20 bytes) ~ 100 bytes per write query
        - total storage per day : 1 B * 100 bytes =  10^9 * 100 bytes ~ 100 GB
        - estimating for 10 years : 100GB * 365 * 10 ~ 400 TB
        - **Hence, sharding qould be required with time**
        - on Avg : 8 TB/machine => No of machines = 50
        - Cosnidering 70% capacity utilization, roughly 70 machines would be required
        - Considering 2 replicas for each machines => Total machines would be 210 



  * **Design Tradeoffs :**
    - Highly Consistent vs Highly Available
      - we require **highly available + eventaully consistent** system for typeahead
    - Latency : require **super low latency** (multiple levels of caching required)

  * **Deep Dive :**
    - getSuggestion(prefix_term, limit = 5)
    - updateFrequency(search_term)
      - Asynchronous Job performed via an internal call
      - The Google service which provides search results to a user’s query makes an internal call to Google’s Typeahead service to update the frequency of the search term.



  * - No of writes - 100K (write request - search functionality)
    - No of reads - 500K  (read request - get_suggestions functionality)
    - Both read and write heavy system
    - High Availability + ultra low latency
    - AP system + eventaully consistent
    - APIs :
      - get_suggestions(Prefix) : (**Prefix** based string searching)
     

  * ##**Solutions :**##
  
  * **1) Trie :**
    A Trie is a prefix tree where each node represents a character. It supports fast prefix lookup in O(length of prefix). It is one of the most common structures for building autocomplete, dictionary, and prefix-based search systems. It's called a prefix tree because all words with the same prefix share the same path.

   * **Why Tries Are Good for Autocomplete : For prefix "mo":**
     - Traverse only 2 characters
     - Instantly reach a node from where all "mo*" results exist
     - Collect top-k suggestions
     - This avoids scanning all words, making it blazing fast.
     
   * **Disadvantages of Tries :**
     - High memory usage : (Every character = a node, often with pointers)
     - Not good for billions of words
     - Hard to store compressed data
     - That’s why big companies use a compressed version (FST - Finite State Transducer).
     
   * **But using only Tries, It is very unoptimized :**
     - To get the suggestions for the given prefix, we'll have to recursively iterate the complete subtree and find the top K suggestions.
     - Trie can't fit in a single machine, hence would require sharding, and sharding a trie is not supported by default by an DB, so it needs to be done manually. 


  * **2) Trie + Hashmap :**
       - At each node, maintain a map with top K suggestions with the prefix
       - Sharding is still an issue in this approach
       - Also, it will waste huge space as we are storing a map at each node.


  * **3) Hashmap Base Solution :**
       - HM #1 : frequency map -> query:frequency
       - HM #2 : suggestions map  -> store top K suggestions for a given prefix
       - we can use any Key:value pair DB
         - NoSQL DBs : Reddis, DynamoDB
         - Sharding is supported by default by these DBs based on key of the key:value pair
         -
       - **Write Query :** In this case, when a write request comes(search functionality), we'll have **to update HM #1**, which will be present in a single machine only, so that's easy, it will be only a **single shard/Intra-shard query**. then it will need go **to update HM #2** to update the frequency for all prefixes in HM #2 in suggestions map. for ex : we need to change the frequency of microsoft for all the prefixes of microsoft in HM #2, which can be present in multiple machines with prefix as 'mic' and this will be **inter-shard query, hence a fan-out request.**
       - We can optimise this process by using **batch processing** as we dont need to update the frequency one by one every time a search happens, also we need eventual consistency anyway so it doesnt matter if few updatations happen at real time or not. How we can do this is, when the search query comes, it will update the frequency, let's say in the cache somewhere, maybe global cache. then we can decide particular threshold, lets say after count increases by 1K searches or maybe after 1hr or so, then we make a DB call to update the frequency by 1K in HM #2. so batch processing reduces the write latency by good margin(by a factor of 1k here). if a new term comes that's not already present in cache, just initialise it with 0, then again after threshold reaches, update it in background or sync up with DB.These updates can happen asynchronously to the shards storing the prefixes of the search term.
       
       - **Read Query :** so based on a prefix, it'll go to that particular shard to look for top k suggestions. since its already sharded based on a prefix as key, it will only be intra-shard query. so ready queries are very much optimised with low latency. To further optimise read queries, we can store the most widely used prefixes in the cache to avoid DB call. we can use global cache, and also local cache like client side cache to store most frequenctly used prefixes for searching. If we use OS with Higher RAM, it will allocate more resources, can store more cache for a particular application. hence, vey few calls will go outside the system.

   * Sharding : When scaling a Trie for autocomplete, we shard it based on prefixes.
If we consider the first three characters of a search term, we get around 50,000 possible prefix combinations (26 letters + numbers → 36³ ≈ 50k), meaning up to 50k potential shards.

However, this creates uneven traffic:
popular prefixes like "mic" or "the" get heavy load, while rare prefixes like "aaa" stay idle.

To fix this, we group multiple prefixes together and assign them to shards using consistent hashing.
The sharding key is the first three characters of the search term, and hashing maps that prefix to a balanced shard.

**For the key–value store (Hashmap DB), sharding is simpler because it already supports sharding via consistent hashing using the key itself.**

## 5] Messenger :


* **Design a messaging app like FB Messenger/Whatsapp :**
  - MVP
  - Scale Estimation
  - Design Tradeoff
  - Architectural Design (API, Design Flow)

 
* **MVP :**
  - Send/Recieve messages(Text, Image, Audio, Video)
  - Message History
  - List of conversations
  - Group conversations
  - Edit/Delete a message
  - Message Status (sent, delivered, seen)

 
* **Scale Estimation :**
  - No of FB user : 3B
  - Daily active users (30%) : 1B
  - Avg no of messages sent by each active user per day : 10 to 20
  - Total no of meesages per day : 20B
  - Write Queries : 20B
  - Read Queries(3x) :  60B
  - Write QPS : (20*10^9)/(24*60*60) ~ (20*10^9)/10^5) ~ 200K
  - Read QPS(3x) : 600K
  - Peak write QPS(2/3x) : 400K
  - **Hence, its both a Read + Write Heavy System**
 
  - **Storage :**
  - Message
    - id : 8B
    - timestamp : 8B
    - content :
      - text : 100B
      - image/video (actual will be stored in S3) : only S3 url will be stored here ~ 50B
    - sender_id : 8B
    - reciever_id : 8B
    - metadata : 50B
    -  ~ 300 B
   
    -  storage per day : 20B * 300 Bytes
    -  20 * 10^9 * 300 ~  6*10^12 = 6 TB
    -  for 10 years : 10*365* 6TB = 24000 TB = 24PB
   
    -  **We definitely need Sharding!!**


* **Design tradeoff : (PACELC)**
  - We need highly consistent system (consisteny here means that if we are sending 5 messages, all 5 texts are sent and recieved and also in same order otherwise inconsistency can lead to miscommunications as chat context would be missing. Availaibility here would mean all texts are being sent instantly evn without order). sometimes, there is a clock symbol infront of texts which means system is unavailable for that moment.(frontend keeps re-trying sending the request/text till the system is availble. sometimes when the client has exhausted teh limit for re-try, it shows ! sign as message could not be delivered. then you have to do it manually)
  - Low Latency (Real time chat)
 


* **APIs :** In a case of an app like messenger where consistency is super important, one thing to consider should be that your write APIs are idempotent. You need to consider this because your primary caller is a mobile app which could have an intermittent network. As such, if the API is called multiple times, due to application level retries, or if data reaches multiple times due to network level retries, you should not create duplicate messages for it.
  
* **How to make the system Idempotent:**
  - We can use a messageId - something that is different across different messages, but same for the same message retried. Imagine everytime we send a message “hello”, the moment “hello” is generated, a new messageId is generated. Now, when we send this message to the backend, instead of saying user A is sending user B a message “Hello”, we say user A is sending userB a message “Hello” with messageId as xyz. Then even if the system gets the same message again then it can identify that it already has a message with messageID xyz and hence, this new incoming message can be ignored. This however, won’t work if messageID is not unique across 2 different messages (If I type “Hello” twice and send twice manually, they should be considered 2 different messages and should not be deduplicated).
  - How to generate Unique messageId: We can possibly use the combination of:
    - Timestamp(date and time), (maybe timestamp of last types character)
    - senderID
    - deviceID
    - recipientID (To be able to differentiate if I broadcast a message).



* **APIs :** Before thinking of the APIs, think of the use cases we would need to support. What kind of views do we have?
  - First view is the view when I open the app. Which is a list of conversations (not messages) with people I recently interacted with (has name of friend/group, along with a snippet of messages. Let’s call that getConversations.
  - If I click into a conversation, then I get the list of most recent messages. Let’s call that getMessages.
  - And finally, in that conversation, I can send a message.
  - So, corresponding APIs:
    - SendMessage(sender, recipient, text, messageId)
    - getMessages(userId, conversationId, offset, limit)
    - getConversations(userId, offset, limit)
    - CreateUser(---).
    - Offset: Where to start
    - Limit: How many messages after that is limit. Offset and limit are usually used to paginate (if page sizes can be different across different clients).




* **Sharding :**
  - **userId:** All conversations and messages should be on the same machine. Essentially, every user has their own mailbox.
  - **ConversationId:** Now all messages of a conversation go on the same machine.
  
 
* **1) sharding by user_id :** So every user will be assigned to one of the machines.
     - **1:1 conversation :**
       - send_msg -> 2 machines (Imagine if Sachin sends a message “hello” to Anshuman, Now this means we have 2 different writes, in Sachin’s machine and in Anshuman’s machine as well and they both have to succeed at the same time. there should be 2 writes that need to happen and somehow they still need to be consistent which is a difficult task.)
       - get_msg -> 1 machine (We can go to a machine corresponding to Sachin and for that we can get the messages for a particular conversation.)
       - get_conversation -> 1 machine (It’s pretty easy. Imagine if Sachin says, get the most recent conversation. We go to a machine corresponding to Sachin and get the most recent conversations and return that.)
         
     - **Group conversation :**
       - send_msg -> fan_out
       - get_msg -> 1 machine
       - get_conversation -> 1 machine
       
* **2) sharding by Conversation_id :** Here for every conversation we will have a separate machine.
     One solution might be to have a Secondary database: In this database we can have user to list of conversations (sorted by recency of the last message sent - along with metadata of conversations - snippet, last Message Timestamp, etc.). * we will maintain 2 separate DBs, a) Message_DB, for send_msg n get_msg APIs, where sharding key will be conversation_id/group_id, another Db will be b) conversations_DB, for get_conversations API where sharding key is user_id. here we'll store the list of convos user had along with the most recent messeage with it.

     - **1:1 conversation:**
       - send_msg -> 3 machines (If let’s say Sachin sends the message in the conversation b/w Sachin and Anshuman, In this case, we have to add the message to SachinAnshuman Database and then in the secondary database we have to change the ordering of conversations in both Sachin and Anshuman’s list of conversations. Therefore a single send message has 3 writes.)
       - get_msg -> 1 machine (it will work fine. If we say get the last 10 messages with the conversation of Anshuman and Sachin. Since they are sharded by conversationId, it will have one machine which has all the messages.)
       - get_conversation -> 1 machine (Now again we can’t go to any one of the databases, we have to go to the secondary database and have to read from here.)
      
     - **Group conversation :**
       - send_msg -> fan out (Hence, we use CACHE)
       - get_msg -> 1 machine
       - get_conversation -> 1 machine


* For systems like, Slack, MS Team, Telegram that can have large groups, userID based sharding will be ridiculously expensive as every single sendMessage will leads to 1000 writes in a 1000 member group. Hence, they relax on the ordering of threads in getConversations (best effort) and instead use conversationId based sharding.
  - For 1:1 messages-> UserId seems to be a better choice (2 writes vs 3). That being said, you can’t go terribly wrong with conversationID either. For the purpose of this doc, we will use userID.
  - In sharding based on userId, 2 operations are working perfectly fine: getMessage, gerConversation.
  - 

* But the problem is with sendMessage, when we send a message hello, it was written to 2 different machines and if one of those writes fails then probably both the machines become inconsistent.
  
* **sendMessage consistency :**
  - Consistency means: If user1 sends a message “Hi”, and does not get an error, then it should imply that the message has been delivered to user2. User2 should get the message.
  - If user1 sends a message to user2, how should we sequence the write between user1 DB and user2 DB to ensure the above consistency
  -
  - Case1: write to sender/user1 first.
    - If it fails then we return an error.
    - If it succeeds, then: We write to recipient / user2 shard: again it can have 2 possibilities:
    - Success: Then the system is in consistent state and they return success.
    - Failure: Rollback and return error.

  - Case2: Write to recipient/user2 shard:
    - Failure: Simply return an error. Succeed: It has reached to recipient so system is in consistent state. Then we write to sender shard:
    - Success: System is in consistent state, return success. Failure: Add it to the queue and keep retrying so that eventually it gets added to the sender’s shard.

  - Out of these 2 cases, case2 is much better because the sender sends the message and the recipient gets the message. The only problem is when the sender refreshes he cannot see the message. Not the good behavior but better of the two behaviors.
  - Because case1 is very dangerous, Sender sends the message and when he refreshes the message is still there but the recipient never got it.



* **Choosing the right DB / cache:**
  Choosing the right DB here is very tricky as this system is both read heavy and write heavy. As we have discussed in the past, both compete with each other, and it’s best to reduce this system to either read heavy or write heavy on the storage side to be able to choose one DB. Also, this requires a massive amount of sharding. So, we are probably looking for a NoSQL storage system that can support high consistency.
  - Reduction to read vs write heavy :
    If we were building a loosely consistent system where we cared about trends, we could have looked to sample writes / batch writes. But here, we need immediate consistency. So, absorbing writes isn’t feasible. You’d need all writes to be immediately persisted to remain highly consistent.
    That means, the only read option is to somehow absorb the number of reads through heavy caching. But remember that you’d need to cache a lot for it to absorb almost all of reads (so much that it won’t fit on a single machine) and this cache has to be very consistent with the DB. Not just that, you’d need to somehow handle concurrent writes for the same user to not create any race condition.
    - Consistency of cache: We can use write-through cache.
    - Lots of data to be cached: We would need to shard cache too.
    - Handle write concurrency in cache: How about we use appservers / business logic servers as cache. We can take a write lock on user then.

   A simple way to do this might be to use appservers as cache, and have them be tied to a set of users (consistent hashing of users -> appservers). This would also let you take a write lock per userID when writes happen, so that writes happen sequentially and do not create race condition in the cache. Since you cannot cache all the users, you can look at doing some form of LRU for users.
  - Pros:
    - Can scale horizontally. Throw more appservers and you can cache more information.
    - Race conditions and consistency requirements handled gracefully.
  - Cons:
    - If the server goes down, things are unavailable till reassignment of the user happens to another app server. Since, this might take a few seconds, this causes unavailability of a few seconds every time the appserver is down.
    - When the app server restarts, there is a cold cache start problem. The initial few requests are slower as the information for the user needs to be read from the DB.
      
* Right DB for Write heavy, consistent system
If we successfully absorb most of the reads, so that they rarely go to the DB, then we are looking for a DB that can support write-heavy applications. HBase is good with that. It allows for column family storage structure which is suited to messages/mailbox, and is optimized for high volumes of writes.
