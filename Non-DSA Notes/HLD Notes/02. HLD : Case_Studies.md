* **CACHING :** the first 3 case studies in L3 explains Caching can occur at: Browser, CDN, Application, Database layers.

**1) Case Study - Scaler DSA Problem Submissions :**
   - **Problem :**
     - Each submission needs large input/output files.
     - Fetching them from remote file storage takes ~2 seconds → slow submissions.
     
   - **Explored Approaches :**
     - **a) TTL-based Cache**
       - Low TTL: frequent cache expiry → many misses → still slow
       - High TTL: stale data issue when files are updated
       - Not reliable for frequently changing files
       
     - **b) Global Cache**
       - Storing files in a single central cache causes:
         - Huge network overhead
         - Limited storage (if in-memory)
         - High eviction rate
         - Not suitable for large files
         
     - **c) Local Cache**
       - Fastest access (from local disk)
       - But updating a file across all servers is difficult
       - Invalidation issue


   - **Best Solution: Use File Metadata (Database-driven invalidation)**
     - Store file metadata (path, updated_at, etc.) in a MySQL table.
     - When a submission comes:
       - Check DB metadata (problem_id + updated_at).
       - Look for file with name pattern: problemId_updatedAt_input.txt
       - If exists → use local cached copy.
       - If not → fetch from file storage and save locally.

      - Why this works
        - Always serves the latest file
        - No manual invalidation required
        - Cache invalidation becomes automatic through metadata
        - Efficient local caching without stale data

**2) Case Study - Contest Rank List :**
   - **Problem :** Rank list updates frequently during a contest. Computing and fetching from DB repeatedly overloads the database.
   
   - **Solution: Global Cache (Redis)**
     - Periodically compute rank list (e.g., every 1 minute)
     - Store this static snapshot in Redis (shared global cache)
     - All app servers read from Redis → only 1 cache miss per minute

   - **Why global cache works well here**
     - Data is the same for everyone
     - Updates are periodic
     - Avoids re-computing or re-fetching on every request

**3) Case Study - Facebook Newsfeed :**
   * **Challenges :**
     - Posts and users are distributed across many machines (sharded)
     - Newsfeed requires posts from all friends → inefficient to fetch from many shards
     - Caching per-user newsfeed is too costly and hard to maintain:
       - Huge storage
       - Thousands of updates per post (fan-out)
       - Hard to change algorithm

  * **Optimized Approach :**
    - Estimate that recent posts (last 30 days) is around 180GB.
    - Store all recent posts in a separate replicated database.
    - Fetch newsfeed efficiently:
      - Get friend_ids
      - Query recent posts from those friends from the “recent posts DB”
        SELECT * FROM all_posts WHERE user_id IN friend_ids LIMIT x OFFSET y
    

  * **Why this works :**
    - Only recent posts matter
    - Disk-based caching is still much faster than remote storage
    - Easy replication to handle high read load
    - No need to pre-compute per-user feeds
    - Old posts can be deleted to save space
    - **This approach uses much lesser storage and approach than the previous system. Here the cache is stored in a hard disk, not in RAM, but still, this is much faster than getting data from an actual storage system.** 




* **Metadata-based invalidation is best for frequently changing large files**
* **Global cache wins when all users need the same data (e.g., rank list)**
* **For heavy, dynamic workloads like newsfeed, store recent data in fast-access storage and query efficiently**

**4] TYPEAHEAD :**


  * **MVP :**
    - Get suggestions for the given prefix
    - Minimum 3 characters would be required to get suggestions
    - Top 5-10 suggestions
    - Suggestions should be based on **popularity (frequency no./ based on no of writes)**

  * **Scale Estimation :**
    - we need to determine  1) No of read/write queries, 2) Amount of Data
    - 
    - No of users = 3 Billion
    - DAU (daily active users) : usually 5-10%. lets take 33% for popular systems like google/FB - 1B
    - Avg. no of searches/user/day - 10
    - No of searches per day - 10 Billion (write queries)
    - As we type each character, typeahead system sends a **Read request** for **get_suggestions**. when we actually press enter and search, that becomes a write request for typeahead/search system. so google search function will act as write query for typeahead system.
    - Avg. no of typeahead suggestions request for every search query - 5
    - No of searches per day = writes per day = 10 B
    - No fo Reads per day = get_suggestions = 50 B
    - **It is a both Read + Write Heavy system** : when the read is abnormally higher like 100/500 times greater than write system, then we say its a read or write heavy system.
    - **QPS : Queries per second**
      - **write QPS :**
        - writes/day = 10 B/(24*60*60) = 10*10^9/86400(10^4)  ~ 10^5
      - **Read QPS :** 5 * write qps ~ 5*10^5
    - **This is the avg traffic, but we need to design for Peak traffic - we roughly take 2-3 times of avg for Peak traffic**
    - 
   
    - **Amount of Data :**
      - write queries = 10 B/day
      - it will consist of exisitng words(we only to update their frequency) + new terms(store them too)
      - New search queries : lets say 10% = 1 B
      - space requeired for storing new word :
        - query(30 bytes) + frequency(8 bytes) + metadata(20 bytes) ~ 100 bytes per write query
        - total storage per day : 1 B * 100 bytes =  10^9 * 100 bytes ~ 100 GB
        - estimating for 10 years : 100GB * 365 * 10 ~ 400 TB
        - Hence, sharding qould be required with time
        - on Avg : 8 TB/machine => No of machines = 50
        - Cosnidering 70% capacity utilization, roughly 70 machines would be required
        - Considering 2 replicas for each machines => Total machines would be 210 



  * **Design Tradeoffs :**
    - Highly Consistent vs Highly Available
      - we require **highly available + eventaully consistent** system for typeahead
    - Latency : require **super low latency** (multiple levels of caching required)

  * **Deep Dive :**
    - APIs
      - getsuggestions(prefix, limit=5)
      - updatefrequency(search_query) -> upsert 



  * - No of writes - 100K (write request - search functionality)
    - No of reads - 500K  (read request - get_suggestions functionality)
    - Both read and write heavy system
    - High Availability + ultra low latency
    - AP system + eventaully consistent
    - APIs :
      - get_suggestions(Prefix) : (**Prefix** based string searching)
     

  * Solutions :
     
  1) Trie :
    A Trie is a prefix tree where each node represents a character. It supports fast prefix lookup in O(length of prefix). It is one of the most common structures for building autocomplete, dictionary, and prefix-based search systems. It's called a prefix tree because all words with the same prefix share the same path.

    * Why Tries Are Good for Autocomplete : For prefix "mo":
      - Traverse only 2 characters
      - Instantly reach a node from where all "mo*" results exist
      - Collect top-k suggestions
      - This avoids scanning all words, making it blazing fast.
     
    * Disadvantages of Tries :
      - High memory usage : (Every character = a node, often with pointers)
      - Not good for billions of words
      - Hard to store compressed data
      - That’s why big companies use a compressed version (FST - Finite State Transducer).
     
    * But using only Tries, It is very unoptimized
      - To get the suggestions for the given prefix, we'll have to recursively iterate the complete subtree and find the top K suggestions.
      - Trie can't fit in a single machine, hence would require sharding, and sharding a trie is not supported by default by an DB, so it needs to be done manually. 


  2) Trie + Hashmap :
     - At each node, maintain a map with top K suggestions with the prefix
     - Sharding is still an issue in this approach
     - Also, it will waste huge space as we are storing a map at each node.


  3) Hashmap Base Solution :
     - HM #1 : frequency map -> query:frequency
     - HM #2 : suggestions map  -> store top K suggestions for a given prefix
     - we can use any Key:value pair DB
       - NoSQL DBs : Reddis, DynamoDB
       - Sharding is supported by default by these DBs based on key of the key:value pair
      
     - Write Query : In this case, when a write request comes(search functionality), we'll have to update HM #1, which will be present in a single machine only, so that's easy, it will be only a single shard/Intra-shard query. then it will need go to update HM #2 to update the frequency for all prefixes in HM #2 in suggestions map. for ex : we need to change the frequency of microsoft for all the prefixes of microsoft in HM #2, which can be present in multiple machines with prefix as 'mic' and this will be inter-shard query, hence a fan-out request.
     - We can optimise this process by using batch processing as we dont need to update the frequency one by one every time a search happens, also we need eventual consistency anyway so it doesnt matter if few updatations happen at real time or not. How we can do this is, when the search query comes, it will update the frequency, let's say in the cache somewhere, maybe global cache. then we can decide particular threshold, lets say after count increases by 1K searches or maybe after 1hr or so, then we make a DB call to update the frequency by 1K in HM #2. so batch processing reduces the write latency by good margin(by a factor of 1k here). if a new term comes that's not already present in cache, just initialise it with 0, then again after threshold reaches, update it in background or sync up with DB. 
     
     - Read Query : so based on a prefix, it'll go to that particular shard to look for top k suggestions. since its already sharded based on a prefix as key, it will only be intra-shard query. so ready queries are very much optimised with low latency. To further optimise read queries, we can store the most widely used prefixes in the cache to avoid DB call. we can use global cache, and also local cache like client side cache to store most frequenctly used prefixes for searching. If we use OS with Higher RAM, it will allocate more resources, can store more cache for a particular application. hence, vey few calls will go outside the system.
