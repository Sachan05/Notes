### UML :
Unified Modeling Language (UML) diagrams are crucial tools in software engineering used to visualize, specify, construct, and document the artifacts of a software system. They serve as a standard way to visualize system architecture and are primarily used for low-level design. UML diagrams help in translating complex code and algorithms into easier-to-understand visual representations, thereby reducing ambiguity and enhancing clarity.

**Types of UML Diagrams :**
- **Structural UML Diagrams :** These diagrams represent the static aspects of the system:
  - **Class Diagram :** Illustrates the classes in a system and their relationships. It's essential for understanding the overall structure of a system and is frequently used in industry and interviews.
  - **Package Diagram :** Groups related classes and illustrates the dependencies between them
  - **Object Diagram :** Depicts instances of classes at a particular moment.
  - **Component Diagram :** Describes the organization and dependencies among a set of components.
 
- **Behavioral UML Diagrams :** These diagrams represent the dynamic aspect of systems:
  - **Activity Diagram :** Describes the flow of control and data within system processes.
  - **Usecase Diagram :** Focuses on the functionalities provided by the system from an external point of view.
Essential components include system boundary, actors, use cases, includes, and extends.
  - **Sequence Diagram :** Highlights object interactions arranged in a time sequence.


* **UseCase Diagrams :** Represents features,  functional requirements of a system.
  - **system boundary** : scope of the system
  - **UseCase** : Verb/Feature (Oval shape)
  - **Actor** : User of the system/Noun/stick diagram. Individuals or entities interacting with the system
  - **Includes** : A --includes--> B, if B is a pre-requisite for A.
  - **Extends** : If one feature has multiple variants.
    - Ex : Login <---extends------ : by by google, by email, by phone.
   
* **Class Diagrams :**  Encompasses entities and their relations.
  - **Entities :** (classes, interfaces, abstract classes, and enums). Each entity is depicted in a rectangle divided into name, attributes, and operations (methods)
  - **Relations :**
    - **Inheritance or IS-A Relation:** Demonstrates that one class inherits from another using a solid line with a closed arrowhead. Inheritance
    - **Association or HAS-A Relation:** Describes relationships where one class uses or knows about another. one class having an attribute of another classs. Association types include:
      - **Aggregation:** A weak association where both entities can exist independently, depicted with an open diamond.
        - Ex : A aggregates B, if object of B can exist even if someone deletes the object of A. ticket -> show, parking spot -> car.
      - **Composition:** A strong association where one entity is part of another, represented by a filled diamond.
        - Ex : if both objects cant exist independently. Lecture -> Notes, Lecture -> Assignments.
     
  - **class :** Divide into two sections. first half includes all variables/paramters. second half all methods/functions.
    - **access modifiers**: (+) public, (-) private, (#) protected.   **static variable** - underline (_)
    - Ex :
      - variables : + name: string, - age: int, - batch : string
      - methods : + method_name : return_type
    - **Interface** : << Interface_name >>
    - **Abstract class** : italic
    - **Enums**: simple names
