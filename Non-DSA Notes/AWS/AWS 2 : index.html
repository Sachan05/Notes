💡 Recap 3
You're setting up RDS Security Group. Your Django app runs on EC2 (sg-app-12345). A developer wants to connect from their laptop using pgAdmin. What's the CORRECT inbound rule?
A PostgreSQL 5432 from 0.0.0.0/0 (so anyone can connect)
B PostgreSQL 5432 from sg-app-12345 only (developer can't connect)
C PostgreSQL 5432 from sg-app-12345 AND developer's specific IP/32
D PostgreSQL 5432 from the VPC CIDR (10.0.0.0/16)

Principle: Least Privilege Access

Only allow the minimum necessary access. Each source should be as specific as possible.

Option Who Can Connect Verdict
A: 0.0.0.0/0 Anyone on the internet ❌ Never expose DB to internet!
B: sg-app only Only EC2 app servers 😶 Good for prod, but dev can't debug
C: sg-app + Dev IP/32 App servers + specific developer ✅ Perfect! Specific and secure
D: VPC CIDR (10.0.0.0/16) All 65,536 IPs in VPC* 😶 Too broad - any compromised instance can access
* How do we get 65,536 IPs from /16?

🔢 CIDR Math Explained

An IP address has 32 bits. The /16 means:

  • First 16 bits = Network part (fixed)
  • Last 16 bits = Host part (available for devices)

Math: 216 = 2 × 2 × 2... (16 times) = 65,536 IPs

CIDR Host Bits Available IPs
/32 0 bits (2⁰) 1 (exact IP)
/24 8 bits (2⁸) 256
/16 16 bits (2¹⁶) 65,536
/0 32 bits (2³²) 4.3 billion (all IPs!)
Correct RDS Security Group Rules
# Rule 1: Allow Django app servers
PostgreSQL  5432  from sg-app-12345

# Rule 2: Allow specific developer (their office IP)
PostgreSQL  5432  from 203.45.67.89/32  # John's IP

# When John leaves or IP changes, remove/update the rule!

💡 Recap 4
When you SSH into an EC2 instance using ssh -i my-key.pem ubuntu@ip, which key is stored on the EC2 instance?
A The private key (.pem file)
B The public key
C Both keys
D A password hash

🔑 Think of it Like a Secret Handshake

🏠 Real World Analogy: The VIP Club

Imagine a VIP club where instead of passwords (which can be stolen), they use a special system:

🔒
Public Key = Your Unique Lock

The club has a lock that ONLY your key can open. You gave them this lock when you registered.

Where: Stored on EC2 in ~/.ssh/authorized_keys

🔑
Private Key = Your Secret Key

Only YOU have the key that opens your lock. You downloaded this as my-key.pem from AWS.

Where: On YOUR laptop only. NEVER share!

🎯 How the Authentication Actually Works

Step 1: You knock on the door
ssh -i my-key.pem ubuntu@ec2-ip
"Hey EC2, I'm the owner of key-pair 'my-key', let me in!"
Step 2: EC2 sends a secret test
EC2: "Prove it! Here's a random message. Only the real key can decrypt it."
(EC2 encrypts a random number using YOUR public key)
Step 3: Your laptop uses the private key
Your laptop decrypts the message (only possible with the matching private key)
Sends the decrypted answer back to EC2
Step 4: ✅ Access granted!
EC2 verifies the answer matches. You're in!

🚫 Why Passwords Are Less Secure

  • Passwords travel over network (can be intercepted)
  • Passwords can be guessed (brute force attacks)
  • Private key NEVER leaves your laptop - only the "proof" does

🔒 The Beauty of This System

Even if a hacker intercepts the entire conversation, they CANNOT get in next time because:

  • The private key was never transmitted
  • Each challenge is random (can't replay old answers)
  • Without the private key, they can't solve new challenges
💡 Recap 5
You run chmod 644 myfile.txt. What permissions does this set?
A Everyone can read, write, and execute
B Owner can read/write, everyone else can only read
C Only owner can read, no one else can access
D File is completely locked

Understanding chmod Numbers

Each digit represents permissions for: Owner | Group | Others

The Magic Numbers

Number Permission Symbol
4 Read r
2 Write w
1 Execute x

Add them up: 6 = 4+2 = read+write, 7 = 4+2+1 = read+write+execute

Breaking Down chmod 644
chmod 644 myfile.txt
      |||
      ||└── Others: 4 (read only)
      |└─── Group:  4 (read only)
      └──── Owner:  6 (read + write)

Result: rw-r--r--
        ^^^^^^^^^
        Owner Group Others

💡 Wait - Where Do I Set the Owner/Group?

chmod sets permissions (read/write/execute). chown sets ownership (who owns the file). They're separate commands!

# chmod = Change Mode (permissions)
chmod 644 myfile.txt       # WHO can do WHAT

# chown = Change Owner (ownership)
chown ubuntu:ubuntu myfile.txt   # WHO is the owner
      │      │
      │      └── Group
      └──────── Owner (user)

# Common examples:
chown www-data:www-data /var/www/html/   # Nginx user owns web files
chown ubuntu:ubuntu ~/myproject/         # Your user owns project
chown -R ubuntu:ubuntu /home/ubuntu/app/ # -R = recursive (all files inside)

Common chmod Values

chmod Symbolic Use Case
400 r-------- 🔑 SSH private keys (REQUIRED!)
644 rw-r--r-- 📄 Regular files (HTML, config)
755 rwxr-xr-x ⚙️ Scripts and executables
700 rwx------ 🔒 Private scripts/directories

⚠️ Why SSH Requires chmod 400

SSH refuses to use private keys if others can read them. Error you'll see:

Permissions 0644 for 'my-key.pem' are too open. It is required that your private key files are NOT accessible by others.

🔧 How to Actually Run Django with Gunicorn

Installing Gunicorn doesn't automatically use it! Here's the difference:

❌ WRONG (Development Only)
# You installed gunicorn but you're NOT using it!
pip install django gunicorn

# This runs Django's built-in dev server, NOT Gunicorn
python manage.py runserver 0.0.0.0:80

# Even with gunicorn installed, runserver is still single-threaded!
✅ CORRECT (Production)
# Install gunicorn
pip install gunicorn

# Run Django using Gunicorn (not runserver!)
gunicorn myproject.wsgi:application --bind 0.0.0.0:8000 --workers 3

# Breaking it down:
# myproject.wsgi:application - path to your WSGI app (in myproject/wsgi.py)
# --bind 0.0.0.0:8000 - listen on all interfaces, port 8000
# --workers 3 - run 3 worker processes (for concurrency)

# Workers rule of thumb: (2 x CPU cores) + 1
# 2 core machine = 5 workers

💡 Remember

  • runserver = Django's dev server (single-threaded, auto-reload)
  • gunicorn = Production WSGI server (multi-worker, no auto-reload)
  • Installing gunicorn does NOT automatically use it - you must run the gunicorn command!

How Load Balancing Works

  • Round Robin: Request 1 → Server 1, Request 2 → Server 2, etc.
  • Least Connections: Send to the server with fewest active connections
  • Health Checks: Don't send traffic to unhealthy servers

AWS ALB (Application Load Balancer) does all this for you.

📈 Solution 2: Auto-Scaling

Auto-Scaling = Automatically add/remove servers based on demand.

Servers scale up/down automatically. You only pay for what you use!

Auto-Scaling Rules (Example)

  • If CPU > 70% for 5 minutes → Add 2 more EC2 instances
  • If CPU < 30% for 10 minutes → Remove 1 EC2 instance
  • Minimum: 2 instances (for redundancy)
  • Maximum: 20 instances (to control costs)
🛠️ Can You Customize Auto-Scaling Rules?

Yes! You have full control over when and how scaling happens:

⚠️ Don't Confuse: EBS vs Elastic Beanstalk

EBS Elastic Beanstalk (EB)
Full Name Elastic Block Store Elastic Beanstalk
What it is Hard drive (storage) PaaS (deployment platform)
Auto-scales? ❌ No (just storage) ✅ Yes! (adds/removes EC2s)

When people say "EB auto-scaling", they mean Elastic Beanstalk, not EBS!

Where to customize in Elastic Beanstalk:

EB Console → Your Environment → Configuration → Capacity → Auto-scaling triggers

😱 The Problem: Setting This Up Manually is HARD!

To get load balancing + auto-scaling working, you'd need to:

  1. Launch multiple EC2 instances with identical configuration
  2. Create an Application Load Balancer (ALB)
  3. Configure Target Groups to register EC2 instances
  4. Create an Auto Scaling Group with launch templates
  5. Configure scaling policies and CloudWatch alarms
  6. Set up health checks
  7. Handle deployment across all instances

This could take days to set up correctly! And any misconfiguration could cause outages.

📁 Step 1: Prepare Your Project Files

📄 What is requirements.txt?

A text file listing all Python packages your app needs. Beanstalk reads this to install dependencies on EC2.

requirements.txt
# Generate this automatically from your venv:
# pip freeze > requirements.txt

Django==4.2.7
gunicorn==21.2.0
psycopg2-binary==2.9.9
whitenoise==6.6.0
boto3==1.33.0

Why it matters: Without this file, Beanstalk won't know what packages to install!

📁 What is .ebextensions/ folder?

A special folder containing configuration files that tell Beanstalk how to set up your environment. Files must end in .config and use YAML format.

⚙️ option_settings
Configure Beanstalk options (WSGI path, env variables)
💻 container_commands
Run commands after deployment (migrate, collectstatic)
📦 packages
Install system packages (like PostgreSQL client)
.ebextensions/django.config 
# This tells Beanstalk how to run Django

option_settings:
  # Where is your WSGI application?
  aws:elasticbeanstalk:container:python:
    WSGIPath: myproject.wsgi:application   # Change 'myproject' to your project name!

  # Serve static files from /static URL
  aws:elasticbeanstalk:environment:proxy:staticfiles:
    /static: static

  # Environment variables (better to use EB console for secrets!)
  aws:elasticbeanstalk:application:environment:
    DJANGO_SETTINGS_MODULE: myproject.settings

# Commands to run AFTER code is deployed
container_commands:
  01_migrate:
    command: "python manage.py migrate --noinput"
    leader_only: true   # Only run on ONE instance (not all!)
  02_collectstatic:
    command: "python manage.py collectstatic --noinput"

🔥 Common Errors and Fixes

"NotAuthorizedError - User is not authorized"

Cause: Your IAM user doesn't have Elastic Beanstalk permissions.

Fix: Add AWSElasticBeanstalkFullAccess policy to your IAM user.

"You have not yet set up your credentials"

Cause: AWS CLI not configured.

Fix: Run aws configure and enter your Access Key ID and Secret.

"ModuleNotFoundError" after deployment

Cause: Missing package in requirements.txt.

Fix: Run pip freeze > requirements.txt and redeploy.

"502 Bad Gateway" after deployment

Cause: Usually wrong WSGI path in .ebextensions/django.config.

Fix: Check that WSGIPath: myproject.wsgi:application matches your actual project folder name.

🎯 Traffic Flow: User Visits Your Django App

  1. User types myapp.com → DNS returns your IP
  2. Request hits Internet Gateway (enters your VPC)
  3. Route Table sends it to public subnet
  4. Load Balancer receives request, passes to app server
  5. Security Group checks: "Is port 80/443 allowed?" ✅
  6. EC2 (Django) in private subnet processes request
  7. Django queries RDS (also in private subnet)
  8. Response travels back the same path!

Part 3

Security Groups - Your AWS Firewall

Think of Security Groups as bouncers for your AWS resources. They decide who gets in and who doesn't!

💪 Real Story: The $50K Mistake

A developer opened SSH (port 22) to 0.0.0.0/0 (the whole internet) for "quick testing". Within hours, a bot found the server, brute-forced the password, and installed crypto mining software. The company got a $50,000 AWS bill before they noticed.

Lesson: Security Groups are not optional. They're your first line of defense!

🔒 What is a Security Group?

A Security Group is a virtual firewall that controls traffic to and from your AWS resources (EC2, RDS, etc.).

🚪

Inbound Rules

Who can COME IN to your resource? Example: "Allow HTTP traffic from the internet"

🚨

Outbound Rules

Where can your resource GO OUT to? By default, all outbound is allowed.

💡 The Basic Rule Format

Every rule has 3 parts:

Part Meaning Example
Protocol/Port What type of traffic? TCP 80 (HTTP), TCP 22 (SSH), TCP 5432 (PostgreSQL)
Source/Destination From/to where? 0.0.0.0/0 (anywhere), 1.2.3.4/32 (specific IP), sg-xxx (another SG)
Description Why this rule exists "Allow web traffic", "SSH from office"

✨ The Magic Word: "Stateful"

Security Groups are stateful. This is the #1 thing interviewers ask about!

💡 Question 5
Your Security Group has ONE inbound rule: "Allow TCP 80 from 0.0.0.0/0". A user visits your website. Can your server send the response back?
A Yes, the response is automatically allowed
B No, you need an outbound rule for port 80
C No, you need to allow the user's random port
D It depends on the Network ACL settings

✨ Stateful = Remembers Conversations

When you allow inbound traffic, the Security Group "remembers" the connection. The response is automatically allowed out - you don't need a separate outbound rule!

🌐 User
→ Allowed (port 80)
💻 EC2
← Auto-allowed (response)
🌐 User

🏠 Think of it Like a Phone Call

If someone calls you and you answer, you can talk back to them. You don't need separate permission to reply - it's part of the same conversation!

That's what "stateful" means. The Security Group tracks the conversation (state) and allows related traffic automatically.

⚖️ Security Groups vs Network ACLs

You might hear about Network ACLs (NACLs). Here's the simple difference:

Feature 🔒 Security Groups 🌐 Network ACLs
Level Instance (EC2, RDS) Subnet (all resources in subnet)
Stateful? ✅ Yes (responses automatic) ❌ No (must allow both ways)
Rules Allow only Allow AND Deny
When to use 99% of the time Extra security layer (rare)

💡 Bottom Line

Focus on Security Groups. They're easier to use (stateful), and you'll use them 99% of the time. NACLs are for advanced use cases where you need to explicitly DENY traffic.

✅ Golden Rules for Security Groups

🚫

NEVER Open SSH to 0.0.0.0/0

Bots scan for open SSH ports 24/7. Use your specific IP: 1.2.3.4/32

🔒

Least Privilege

Only open ports you actually need. Close everything else.

🔗

Reference SGs, Not IPs

For internal traffic, reference Security Group IDs instead of IP addresses.

📄

Name Everything

Use clear names: "django-web-sg", "rds-postgres-sg". Future you will thank you!

🔗 The Security Group Reference Pattern

This is a super important pattern for production Django apps!

🔥 The Problem

You have Django (EC2) and PostgreSQL (RDS). You want Django to connect to the database, but block everyone else. What do you put in the RDS security group?

  • Bad approach: Allow the EC2's IP address. But EC2 IPs can change! Auto-scaling adds new IPs!
  • Good approach: Allow traffic from the EC2's Security Group ID!

🔗 SG Reference Pattern in Action

django-sg

ID: sg-abc123

💻 EC2 (Django)
Inbound: TCP 80, 443 from 0.0.0.0/0
→
rds-sg

ID: sg-xyz789

🗃 RDS (PostgreSQL)
Inbound: TCP 5432 from sg-abc123
3-Tier Django Architecture 
# 1. Web Server Security Group (nginx-sg)
INBOUND:
  - TCP 80   from 0.0.0.0/0    # HTTP from internet
  - TCP 443  from 0.0.0.0/0    # HTTPS from internet
  - TCP 22   from YOUR_IP/32   # SSH from YOUR IP only!

# 2. App Server Security Group (django-sg)
INBOUND:
  - TCP 8000 from nginx-sg    # Only from Nginx!
  - TCP 22   from YOUR_IP/32   # SSH from YOUR IP only!

# 3. Database Security Group (rds-sg)
INBOUND:
  - TCP 5432 from django-sg   # Only from Django!

# ✨ Magic: Internet can't reach Django directly, can't reach DB at all!

Why This Pattern is Powerful

  • ⚡ Auto-scaling works: New EC2 instances automatically have access (same SG)
  • 💻 No IP management: IPs can change, SG references stay valid
  • 🔒 Defense in depth: Even if web server is hacked, attacker can't reach DB directly

💻 AWS Console: Creating a Security Group

⚙️ Step-by-Step for Django App

  1. Go to EC2 → Security Groups → Create security group
  2. Fill in basics:
    • Name: django-web-sg
    • Description: "Security group for Django web servers"
    • VPC: Select your VPC
  3. Add Inbound Rules:
    • HTTP (80) from 0.0.0.0/0 - "Web traffic"
    • HTTPS (443) from 0.0.0.0/0 - "Secure web traffic"
    • SSH (22) from [My IP] - AWS auto-fills your IP!
  4. Leave Outbound Rules as default
    (All traffic allowed out - that's usually fine)
  5. Click "Create security group"

Anyone could claim to own any domain! Google needs proof that you control it.

💡 The Logic Behind Verification

Think of it like this:

Google says: "Add this secret code to your DNS records:"

google-site-verification=abc123xyz789

You: Log into Route 53 → Add TXT record with that code

Google checks: "Let me query the DNS for myapp.com TXT records..."

Google finds your code: "Only the domain owner could add DNS records. Verified! ✅"

🎯 Real-World TXT Record Uses

Service Why They Need Verification Example TXT Record
Google Search Console Show you search rankings for YOUR site google-site-verification=abc123...
SSL Certificates (Let's Encrypt) Prove you own domain before issuing HTTPS cert _acme-challenge.myapp.com
Email (SPF) Tell mail servers "only these IPs can send email as @myapp.com" v=spf1 include:_spf.google.com ~all
Email (DKIM) Cryptographic signature to prevent email spoofing v=DKIM1; k=rsa; p=MIIBIjAN...
Microsoft 365 Prove ownership before setting up work email MS=ms12345678

⚠️ Why SPF/DKIM Matter for Your Startup

Without SPF records, emails from @yourstartup.com may:

  • Land in spam folders
  • Be rejected entirely
  • Be easily spoofed by scammers pretending to be you

Set up SPF when you configure business email!

📝 How to Add a TXT Record in Route 53

  1. Go to Route 53 → Hosted Zones → Your Domain
  2. Click "Create Record"
  3. Record type: TXT
  4. Value: Paste the verification code (keep the quotes!)
  5. TTL: 300 (5 minutes)
  6. Click "Create" and wait 5-10 minutes for propagation

🔍 A Record vs CNAME vs Alias - What's the Difference?

🏠 The "Naked Domain" Problem

Let's say you want myapp.com (without www) to work. This is called a "naked domain" or "zone apex".

Problem: CNAME records don't work for naked domains! DNS rules say zone apex can only have A records.

But what if your load balancer's IP changes? AWS invented Alias records to solve this!

Feature A Record CNAME Alias (AWS)
Points to IP address Another domain AWS resource
Works for myapp.com? ✅ Yes ❌ No! ✅ Yes
Cost $0.40/million queries $0.40/million queries 🆕 FREE for AWS
Best for Direct EC2 IP www subdomain ALB, CloudFront, S3

💡 Rule of Thumb

  • Single EC2? → A record to Elastic IP
  • Load Balancer? → Alias to ALB
  • www subdomain? → CNAME to naked domain

💻 AWS Console: Setting Up Route 53 for Django

⚙️ Step-by-Step Guide

  1. Option A: Buy domain from Route 53
    Route 53 → Registered domains → Register domain
    ~$12/year for .com. AWS handles everything!
  2. Option B: Use existing domain (GoDaddy, Namecheap, etc.)
    Route 53 → Hosted zones → Create hosted zone
    Then copy NS records to your registrar
  3. Create an A record for your domain
    • Record name: leave blank (for naked domain)
    • Record type: A
    • Value: Your EC2's Elastic IP
    • TTL: 300 (5 minutes - good for testing)
  4. Create a CNAME for www
    • Record name: www
    • Record type: CNAME
    • Value: myapp.com (your naked domain)
  5. Wait for propagation (usually 5-15 minutes)
    Can take up to 48 hours in rare cases

⚠️ Common Issue: "Invalid Host Header"

If you see this error after adding a domain, you forgot to add it to ALLOWED_HOSTS in Django settings!

Part 5

CloudWatch - Monitoring & Metrics

CloudWatch helps you monitor your AWS resources. See CPU usage, network traffic, disk I/O, and more.

💡 Question 7
Your Django app suddenly becomes slow. What's the FIRST thing you should check?
A Git log for recent changes
B CloudWatch metrics (CPU, Memory, Network)
C Restart the server
D Check the Django admin

Metrics First, Then Debug

Before making changes, understand what's happening. Metrics tell you:

  • High CPU: Code issue, need bigger instance, or add more instances
  • High Memory: Memory leak, need more RAM
  • High Network: Traffic spike, DDoS attack
  • High Disk I/O: Database bottleneck, slow queries

Real Scenario

A developer noticed their app was slow and restarted EC2. Still slow. Checked CloudWatch: CPU at 15%, Memory at 30%. The issue was RDS - CPU at 100%! A slow query was the culprit.

📈 Key EC2 Metrics

Metric What It Measures Warning Sign
CPUUtilization % of CPU being used >80% sustained = need more compute
NetworkIn/Out Bytes transferred Sudden spike = traffic surge or attack
DiskReadOps Number of read operations High = I/O bottleneck
DiskWriteOps Number of write operations High = I/O bottleneck
StatusCheckFailed Instance health Any failure = problem!

⚠️ Memory Not Included!

EC2 doesn't send memory metrics by default. You need to install the CloudWatch Agent to get memory and disk usage metrics.

🔧 Installing CloudWatch Agent

Install on Ubuntu EC2 
# Download CloudWatch Agent
wget https://s3.amazonaws.com/amazoncloudwatch-agent/ubuntu/amd64/latest/amazon-cloudwatch-agent.deb

# Install
sudo dpkg -i amazon-cloudwatch-agent.deb

# Create config (interactive wizard)
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-config-wizard

# Start agent
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \
    -a fetch-config \
    -m ec2 \
    -c file:/opt/aws/amazon-cloudwatch-agent/bin/config.json \
    -s

What CloudWatch Agent Provides
