What is Cloud Computing?

Before we dive into AWS, let's understand why companies moved from physical servers to the cloud.

💡 Question 1
Your startup just got funding. You need servers for your Django app. What do you do?
A Buy physical servers, set up a data center
B Rent servers from AWS/Azure/GCP
C Run everything on your laptop

Why Rent Instead of Buy?

Factor Buy Servers (On-Premise) Rent (Cloud)
Upfront Cost $50,000+ for servers $0 - pay as you go
Time to Start Weeks/months Minutes
Scaling Buy more hardware Click a button
Maintenance You fix everything AWS handles hardware
If startup fails Stuck with servers Stop paying, walk away

🦈 The Shark Tank Effect

Imagine your startup appears on Shark Tank India. The episode airs and suddenly 10 lakh users hit your website in 30 minutes!

  • On-Premise: Your 2 servers crash. Users see errors. You lose sales. You can't buy new servers at 10 PM.
  • Cloud (AWS): Auto-scaling spins up 20 more servers automatically. Traffic handled. Money made. Scales down after the spike.
💡 Question 2
What percentage of companies still run their servers on-premise (own data centers) today?
A Less than 10% - almost everyone is on cloud
B About 25%
C About 40-50% still on-premise or hybrid
D Over 70% - cloud is still new

Surprising Reality: On-Premise is Still Huge!

Despite the cloud hype, ~40-50% of enterprise workloads still run on-premise. Many large enterprises use hybrid setups.

⚠️ Cloud Has Downsides Too!

  • Cost at scale: At massive scale, cloud can be 2-3x more expensive than owning hardware
  • Vendor lock-in: Moving away from AWS later is painful
  • Data sovereignty: Your data is on someone else's servers
  • Outages: When AWS goes down, so does half the internet

When On-Premise Makes Sense

  • Banks/Government: Strict data regulations, security requirements
  • Predictable high load: If you always need 100 servers, owning is cheaper
  • Dukaan's approach: Running on bare metal for cost efficiency

🔐 What is SSH?

SSH (Secure Shell) is a protocol to securely connect to remote computers over an insecure network (internet).

Why SSH Exists

Before SSH, people used Telnet which sent passwords in plain text. Anyone sniffing the network could see your password! SSH encrypts everything.

How Public/Private Keys Work

🔑 Private Key
(my-key.pem - stays on YOUR laptop)
+
🔓 Public Key
(stored on EC2 instance)
  • Private key: Only you have it. Never share. Like your house key.
  • Public key: AWS puts this on the server. Like the lock on your door.
  • When you connect, they verify mathematically without sending the private key!

🛡️ Security Groups - Your Firewall

Security groups control what traffic can come IN and go OUT of your instance.

Example Security Group Rules 
# INBOUND RULES (traffic coming IN)
┌──────────┬──────────┬─────────────────┬──────────────────────┐
│ Type     │ Port     │ Source          │ Description          │
├──────────┼──────────┼─────────────────┼──────────────────────┤
│ SSH      │ 22       │ My IP           │ Your SSH access      │
│ HTTP     │ 80       │ 0.0.0.0/0       │ Web traffic (anyone) │
│ HTTPS    │ 443      │ 0.0.0.0/0       │ Secure web traffic   │
└──────────┴──────────┴─────────────────┴──────────────────────┘

Why These Specific Ports?

Port Protocol Why This Number? Can You Change It?
22 SSH Standard since 1995, everyone knows it Yes, but then specify: ssh -p 2222 ...
80 HTTP Browsers default to port 80 for http:// Yes, but users must type :8080 in URL
443 HTTPS Browsers default to 443 for https:// Yes, but users must type :8443 in URL

0.0.0.0/0 = "anywhere in the world" (all IPs). But what does this notation mean?

Understanding IP Addresses & CIDR Notation

IPv4 Address

An IP address like 192.168.1.5 is just a 32-bit number written in 4 groups of 8 bits each:

192      .  168      .  1        .  5
11000000    10101000    00000001    00000101
(8 bits)    (8 bits)    (8 bits)    (8 bits) = 32 bits total

CIDR Notation (the /number)

The /24 in 192.168.1.0/24 means "first 24 bits are the network, remaining 8 bits are for hosts".

CIDR Meaning Available IPs Example
/32 All 32 bits fixed 1 (exact IP) 192.168.1.5/32 = only 192.168.1.5
/24 First 24 bits fixed 256 (last octet varies) 192.168.1.0/24 = 192.168.1.0 to .255
/16 First 16 bits fixed 65,536 192.168.0.0/16 = 192.168.x.x
/0 No bits fixed All ~4.3 billion IPs 0.0.0.0/0 = anywhere!

Why IPv6 exists

IPv4's 32 bits = ~4.3 billion addresses. That's not enough for 8 billion people with multiple devices!

IPv6 uses 128 bits = 340 undecillion addresses (340 followed by 36 zeros).

IPv6 looks like: 2001:0db8:85a3:0000:0000:8a2e:0370:7334

In AWS Security Groups

You'll often see both:

  • 0.0.0.0/0 - Allow from any IPv4 address
  • ::/0 - Allow from any IPv6 address

⚠️ Security Best Practices

  • Never open SSH (port 22) to 0.0.0.0/0 (everyone)
  • Restrict SSH to "My IP" or your office IP range
  • Only open ports you actually need

🚀 Steps to Launch an EC2 Instance

  1. Choose AMI
    Select Ubuntu 22.04 LTS (Free tier eligible)
  2. Choose Instance Type
    Select t2.micro (Free tier: 750 hours/month)
  3. Configure Security Group
    Allow SSH (port 22) from your IP, HTTP (80)
  4. Create/Select Key Pair
    Download the .pem file and keep it safe!
  5. Launch!
    Wait ~30 seconds for the instance to start

Understanding chmod - File Permissions

chmod = "change mode" - sets who can read/write/execute a file.

Three types of permissions:

  • r (read) = 4
  • w (write) = 2
  • x (execute) = 1

Three categories of users: Owner | Group | Others

Command Meaning Owner Group Others
chmod 400 r-- --- --- Read only Nothing Nothing
chmod 755 rwx r-x r-x All Read+Execute Read+Execute
chmod 644 rw- r-- r-- Read+Write Read only Read only

Why 400 for SSH keys? SSH refuses to use private keys if anyone else can read them - it's a security requirement!

🐍 Setting Up Django on EC2

What is Gunicorn?

Gunicorn (Green Unicorn) is a production-grade WSGI server for Python apps.

  • Django's runserver: Single-threaded, for development only
  • Gunicorn: Multiple worker processes, handles concurrent requests

We didn't use Gunicorn locally because runserver is fine for development. In production, you NEED Gunicorn (or uWSGI).

Run on EC2 (after SSH) 
# Update system packages
sudo apt update && sudo apt upgrade -y

# Install Python and pip
sudo apt install python3 python3-pip python3-venv -y

# Create project directory
mkdir ~/myproject && cd ~/myproject

# Create virtual environment
python3 -m venv venv
source venv/bin/activate

# Install Django and Gunicorn
pip install django gunicorn

# Create Django project
django-admin startproject mysite .

# Add EC2 IP to ALLOWED_HOSTS (replace YOUR_EC2_IP)
sed -i "s/ALLOWED_HOSTS = \[\]/ALLOWED_HOSTS = ['YOUR_EC2_IP', 'localhost']/" mysite/settings.py

# Run on port 80 (requires sudo for ports below 1024)
sudo /home/ubuntu/myproject/venv/bin/python manage.py runserver 0.0.0.0:80

✅ Test It!

Visit http://your-ec2-ip in your browser (no port needed - 80 is default). You should see the Django welcome page!

RDS - Relational Database Service

A managed database service. AWS handles backups, updates, and scaling - you just use the database.

💡 Question 5
Why use RDS instead of installing PostgreSQL on your EC2 instance?
A RDS is faster than PostgreSQL on EC2
B RDS is free, EC2 database costs money
C AWS handles backups, updates, and failover automatically
D EC2 doesn't support databases

RDS vs Self-Managed Database

Task Self-Managed (on EC2) RDS (Managed)
Install database You do it AWS does it
Security patches You do it AWS does it
Backups You set up cron jobs Automatic daily backups
Failover You build it Multi-AZ option
Scaling Migration headache Few clicks
3 AM crash You wake up AWS handles it

When to Self-Manage?

  • You need a database not supported by RDS
  • You need fine-grained control over configuration
  • Cost optimization at massive scale

🛡️ RDS Security Group

Your RDS database should ONLY accept connections from your EC2 instance, not the public internet.

Internet
✗
RDS

✓

RDS Security Group - Inbound Rule 
┌──────────────┬──────────┬─────────────────────────┬─────────────────┐
│ Type         │ Port     │ Source                  │ Description     │
├──────────────┼──────────┼─────────────────────────┼─────────────────┤
│ PostgreSQL   │ 5432     │ sg-xxxxx (EC2 SG)       │ Allow from EC2  │
└──────────────┴──────────┴─────────────────────────┴─────────────────┘

# Note: Source is the Security Group ID of your EC2 instance
# NOT 0.0.0.0/0 (that would expose your DB to the internet!)

⚠️ Critical Security Rule

Never make your RDS publicly accessible or allow connections from 0.0.0.0/0. Your database should only be reachable from your application servers.

🔗 Connect Django to RDS

settings.py 
# Install psycopg2 for PostgreSQL support
# pip install psycopg2-binary

DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.postgresql',
        'NAME': 'mydb',           # Your DB name
        'USER': 'admin',          # Master username
        'PASSWORD': 'your-password',  # Master password
        'HOST': 'mydb.xxxxx.ap-south-1.rds.amazonaws.com',  # RDS endpoint
        'PORT': '5432',
    }
}

Finding Your RDS Endpoint

AWS Console → RDS → Databases → Select your DB → "Endpoint & port"

💡 Question 6
Your Django app on EC2 can't connect to RDS. What's the most likely cause?
A RDS doesn't support Django
B Security Group not allowing EC2 to connect to RDS
C Wrong Python version
D RDS is in a different AWS account

Common Connection Issues

  1. Security Group: RDS SG must allow inbound on port 5432 from EC2's Security Group
  2. Wrong endpoint: Copy the exact endpoint from RDS console
  3. Wrong credentials: Username/password must match what you set during RDS creation
  4. Database name: Make sure the database exists (create it manually or via Django)
  5. psycopg2 not installed: Run pip install psycopg2-binary
Debug Connection
# Test connectivity from EC2
telnet your-rds-endpoint 5432

# If connected, you'll see:
# Trying 10.0.1.123...
# Connected to your-rds-endpoint

# If not connected, check Security Group!
