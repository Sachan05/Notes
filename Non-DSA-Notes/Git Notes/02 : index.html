Q1 Merge Commit

When you run git merge feature from the main branch, Git creates a "merge commit". What makes this commit special?

A) It has no parent commits
B) It has exactly ONE parent (the feature branch)
C) It has TWO parents (main and feature branch tips)
D) It deletes the feature branch automatically

Answer: C) It has TWO parents

A merge commit is special because it points to both the tip of main AND the tip of feature. This is how Git preserves the complete history of parallel development. You can always trace back through either parent to see what happened on each branch.

M.parent[0] = last commit on main
M.parent[1] = last commit on feature

Q2 Rebase Direction (Visualization)

You have this branch structure. You want to update your feature branch with the latest changes from main.

Starting Point:

          C1 ← C2 ← C3  (feature)
         /
A ← B ← D ← E  (main)

Which commands should you run?

A) git checkout main then git rebase feature
B) git checkout feature then git rebase main
C) git checkout main then git merge feature
D) git checkout feature then git merge main

Answer: B) git checkout feature then git rebase main

Think of it as: "I (feature) want to sit on top of main"

✅ Option B (Correct)

A ← B ← D ← E  (main)
              \
               C1' ← C2' ← C3'  (feature)

Feature's commits are replayed on top of main. Main is untouched!

❌ Option A (Dangerous!)

A ← B ← C1 ← C2 ← C3  (feature)
                    \
                     D' ← E'  (main) 😱

Main gets rewritten! Everyone else's main is now broken.

Memory Trick: "I sit on the throne"
git checkout <branch-to-move> → git rebase <branch-to-sit-on>

Q3 Rebase: True or False

Which statements about git rebase are TRUE? (Multiple correct answers)

A) Rebase creates new commits with different SHA hashes
B) Rebase preserves the original commits alongside the new ones
C) Rebase results in a linear history (no merge commits)
D) Rebase is safe to use on commits already pushed to a shared branch

Answer: A and C are TRUE

✅ A) TRUE — Rebase "replays" commits on a new base, creating entirely new commits. Even if the code is identical, the SHA hash changes because the parent commit is different.
❌ B) FALSE — The original commits become "orphaned" and are eventually garbage collected (~30 days). They're not preserved in the branch history.
✅ C) TRUE — Unlike merge (which creates a merge commit with two parents), rebase creates a straight line of commits. This is why some teams prefer it for cleaner history.
❌ D) FALSE — This is the "Golden Rule"! Since rebase rewrites history (new hashes), anyone who already pulled the old commits will have conflicts.

Q4 When to Rebase

In which scenario is git rebase the appropriate choice?

A) Integrating a completed feature branch into main for deployment
B) Updating your local feature branch with latest changes from main before creating a PR
C) Combining work from two developers who pushed to the same shared branch
D) Undoing a commit that's already been pushed to production

Answer: B) Updating your local feature branch with latest changes from main

This is THE classic use case for rebase:

# You're on your feature branch, main has new commits
$ git checkout feature/my-work
$ git fetch origin
$ git rebase origin/main
# Now your commits sit on top of latest main
# Ready for a clean PR!

Why not the others?

  • A) Use merge to integrate into main — it preserves history and is safe
  • C) Shared branches should never be rebased — use merge or pull
  • D) Use git revert to undo pushed commits safely
Rule of Thumb: Rebase YOUR private branch onto the shared branch. Never the other way around.

Collaboration Basics

Git Remotes

A remote is just a copy of your repository hosted somewhere else (like GitHub). It's how you share code with others.

🌐 What is a Remote?

☁️ Remote: "origin"

github.com/you/project

The "source of truth" for your team

git push

git pull

Essential Remote Commands

Working with Remotes
# See your remotes
$ git remote -v
origin https://github.com/you/project.git (fetch)
origin https://github.com/you/project.git (push)

# Add a remote
$ git remote add origin https://github.com/you/project.git

# Push your commits to remote
$ git push origin main

# Pull latest changes from remote
$ git pull origin main

# Fetch changes without merging (safe peek)
$ git fetch origin

⬆️ git push

Upload your local commits to the remote. "Hey GitHub, here's my new work!"

⬇️ git pull

Download AND merge remote changes. pull = fetch + merge

👀 git fetch

Download changes but don't merge yet. Safe way to see what's new before integrating.

🔄 Typical Workflow

1. git pull origin main — Get latest changes

2. Make your changes, commit locally

3. git push origin main — Share your changes

🤔 Why is it called "origin"? Can I name it something else?

"origin" is just a convention — it's the default name Git uses when you clone a repository. It means "where this repo originally came from."

You can absolutely use different names! Common ones:

  • origin — Your fork or main remote
  • upstream — The original repo you forked from
  • backup — A secondary backup location
# Add a remote with custom name
$ git remote add upstream https://github.com/original/repo.git

🔍 Deep Dive: What Happens When You Clone?

Understanding Remote Tracking Branches

When you clone a repo, Git creates TWO types of branches locally:

✏️ Local Branches (Read + Write)

main
feature/login
  • You can checkout these
  • You can commit to these
  • You can modify these
  • These are YOUR working copies

👀 Remote Tracking Branches (Read Only)

origin/main
origin/feature/login
  • You cannot commit to these
  • These are bookmarks showing where remote was
  • Updated ONLY by fetch or pull
  • Like a "last known position" of remote
See all branches
$ git branch -a
* main # local branch (you're here)
feature/login # local branch
remotes/origin/main # remote tracking (read-only)
remotes/origin/feature/login # remote tracking (read-only)

Fetch vs Pull: The Real Difference

git fetch

What it does:

  1. Downloads new commits from remote
  2. Updates origin/main (tracking branch)
  3. Does NOT touch your local main

Before fetch:

main: A-B-C
origin/main: A-B-C

After fetch (remote had D-E):

main: A-B-C ← unchanged!
origin/main: A-B-C-D-E ← updated

Safe! You can review changes before merging.

git pull

What it does:

  1. Runs git fetch first
  2. Then runs git merge origin/main
  3. Modifies your local main

Before pull:

main: A-B-C
origin/main: A-B-C

After pull (remote had D-E):

main: A-B-C-D-E ← merged!
origin/main: A-B-C-D-E ← updated

Convenient! But may cause merge conflicts.

💡 When to Use Which?

Use git fetch when:

  • You want to see what changed before merging
  • You're not sure if merging will cause conflicts
  • You want to compare: git diff main origin/main

Use git pull when:

  • You trust the remote changes
  • You want to quickly sync up
  • You're ready to handle any merge conflicts

🚀 Understanding git push -u origin branch-name

What Happens When You Push a New Branch?

When you create a branch locally, the remote has no idea it exists. You need to push it.

git push origin feature

Pushes your branch, but doesn't set up tracking.

# Next time you want to push:
$ git push
fatal: The current branch feature has no upstream branch.
# You have to specify remote and branch every time!
$ git push origin feature

git push -u origin feature

Pushes AND sets up upstream tracking.

# First push with -u:
$ git push -u origin feature
Branch 'feature' set up to track 'origin/feature'.

# Now, every time after:
$ git push # Just this! Git knows where.
$ git pull # This works too!

📖 What Does -u (or --set-upstream) Do?

It creates a link between your local branch and the remote branch:

local: feature ──tracks──▶ origin/feature

After this link is set, Git remembers: "When I'm on feature, push/pull to/from origin/feature."

Behind the scenes
# What -u adds to your .git/config:

[branch "feature"]
remote = origin
merge = refs/heads/feature
git push origin feature

Push once, no tracking

git push -u origin feature

Push + set tracking ✓

git push

Works after -u is set

Open Source Collaboration

Forking a Repository

A fork is your personal copy of someone else's repository. It's how you contribute to projects you don't own.

Fork vs Clone: What's the Difference?

git clone

Downloads a repo to your computer

☁️ GitHub
↓
💻 Your Computer

Same repo, just downloaded locally

Fork (on GitHub)

Creates YOUR copy on GitHub

☁️ Original
→
☁️ Your Fork
↓
💻 Your Computer

Your own GitHub copy you can push to

When Do You Need a Fork?

✅
You DON'T need a fork when:
  • You own the repository
  • You're a collaborator with push access
  • You're on a team with shared repo access
🍴
You NEED a fork when:
  • Contributing to open source projects
  • You don't have push access to the repo
  • You want to experiment without affecting original

🍴 The Fork Workflow (Contributing to Open Source)

1. Fork the repo on GitHub (click "Fork" button)

2. git clone https://github.com/YOU/repo.git — Clone YOUR fork

3. git remote add upstream https://github.com/ORIGINAL/repo.git

4. Create a branch, make changes, commit

5. git push origin your-branch — Push to YOUR fork

6. Create a Pull Request from your fork to original

🔄 How do I keep my fork up-to-date with the original repository?

The original repository will get new commits over time. Here's how to sync:

# Make sure you have upstream set
$ git remote add upstream https://github.com/ORIGINAL/repo.git

# Fetch latest from original repo
$ git fetch upstream

# Merge upstream changes into your main
$ git checkout main
$ git merge upstream/main

# Push updated main to your fork
$ git push origin main
Code Review & Collaboration

Pull Requests (PRs)

A Pull Request is a proposal to merge your changes into another branch. It's where code review happens.

🔀 What is a Pull Request?

"Hey, I made some changes on my branch. Can you pull them into the main branch?"

feature/login

Your branch

→ PR →
main

Target branch

A PR is NOT a Git feature — it's a GitHub/GitLab/Bitbucket feature built on top of Git.

The Pull Request Lifecycle

1

Create a Branch & Make Changes

Work on your feature branch, commit your changes locally.

2

Push to Remote

git push origin feature/your-branch
3

Open a Pull Request

On GitHub, click "New Pull Request". Write a description of your changes.

4

Code Review

Teammates review your code, leave comments, request changes.

5

Merge!

Once approved, click "Merge Pull Request". Your code is now in main! 🎉

✨ Writing a Good Pull Request

✅ Do

  • Write a clear title describing the change
  • Explain WHY, not just WHAT
  • Keep PRs small and focused (one feature/fix)
  • Add screenshots for UI changes
  • Link related issues

❌ Don't

  • Create huge PRs with 50+ files
  • Leave the description empty
  • Mix multiple unrelated changes
  • Ignore reviewer comments
  • Merge without approval

The Inevitable

Merge Conflicts

What happens when two people edit the same line? Git needs YOUR help to decide.

⚠️ When Do Conflicts Happen?

A conflict occurs when Git can't automatically merge because the same lines were changed differently in both branches.

Alice's Branch
price = 99
💥
Bob's Branch
price = 149

Both changed price on line 42. Git asks: "Which one should I keep? 99 or 149?"

What a Conflict Looks Like in Your File

config.py - CONFLICT
# Above this is code that merged fine

<<<<<<< HEAD
price = 99 # Your current branch
=======
price = 149 # Incoming branch
>>>>>>> feature/new-pricing

# Below this is more code that merged fine
<<<<<<< HEAD

Start of conflict (your version)

=======

Separator between versions

>>>>>>> branch

End of conflict (incoming version)

How to Resolve a Merge Conflict

1

Git tells you there's a conflict

$ git merge feature/new-pricing
Auto-merging config.py
CONFLICT (content): Merge conflict in config.py
Automatic merge failed; fix conflicts and then commit.
2

Open the file and decide what to keep

Remove the conflict markers and keep the code you want. You have options:

  • Keep YOUR version (HEAD)
  • Keep THEIR version (incoming)
  • Keep BOTH (if they should both exist)
  • Write something completely NEW
3

Edit the file to resolve

# After editing - keep price = 99 (or merge both ideas)
price = 99

Make sure to remove ALL conflict markers: <<<<<<<, =======, >>>>>>>

4

Stage and commit the resolution

$ git add config.py
$ git commit -m "fix: Resolve pricing conflict, keep original price"

💡 Common Conflict Scenarios

Same Line, Different Changes

Most common. Two people edited line 42. Pick one or merge ideas.

One Deleted, One Modified

You deleted a function, teammate edited it. Keep or delete?

File Renamed vs Modified

You renamed utils.py to helpers.py, teammate edited utils.py.

Both Added Similar Lines

Two people added imports at the same location. Usually keep both!

🆘 Help! I'm confused and want to start over. How do I abort a merge?

If you're in the middle of a conflicted merge and want to cancel it completely:

# Abort the merge and go back to before you started
$ git merge --abort

This returns your working directory to the state before the merge. No harm done!

More Git Tools to Know

These aren't essential for beginners, but knowing they exist will help you when you need them.

Essential 📦 git stash — Temporarily save uncommitted changes

Need to switch branches but have uncommitted work? git stash saves your changes temporarily.

# Save current changes to stash
$ git stash

# Do other work, switch branches, etc.
$ git checkout other-branch

# Come back and restore your changes
$ git checkout original-branch
$ git stash pop

# View all stashed changes
$ git stash list

Think of it as a clipboard for uncommitted work!

Important ↩️ git reset — Understanding the Command

Breaking Down: git reset --hard HEAD~1

git reset --hard HEAD~1

git reset

Move HEAD to a different commit

--hard

Reset mode (how much to undo)

HEAD~1

Where to reset to (1 commit back)

The Three Reset Modes

Remember Git's three areas? Reset modes control which ones get affected:

Working Directory

Your actual files

→

Staging Area

git add puts files here

→

Repository (HEAD)

Committed history

--soft
git reset --soft HEAD~1

Moves HEAD only. Changes stay staged.

✅ Repository: reset

❌ Staging: unchanged

❌ Working Dir: unchanged

Use: Undo commit but keep changes staged for recommit

--mixed (default)
git reset HEAD~1

Moves HEAD and unstages. Files intact.

✅ Repository: reset

✅ Staging: reset

❌ Working Dir: unchanged

Use: Undo commit and unstage, but keep file changes

--hard ⚠️
git reset --hard HEAD~1

Nuclear option. Destroys everything.

✅ Repository: reset

✅ Staging: reset

✅ Working Dir: reset

Warning: Uncommitted changes are LOST!

Where to Reset To (Different Targets)

Different ways to specify the target
# Using HEAD~ (relative to current)
$ git reset --hard HEAD~1 # 1 commit back
$ git reset --hard HEAD~3 # 3 commits back
$ git reset --hard HEAD~10 # 10 commits back

# Using commit SHA (exact commit)
$ git reset --hard a1b2c3d # Reset to this specific commit
$ git reset --hard a1b2c3d4e5f6 # Full SHA also works

# Using branch name
$ git reset --hard origin/main # Match remote exactly
$ git reset --hard main # Match local main branch

# Using HEAD^ (parent commits)
$ git reset --hard HEAD^ # Same as HEAD~1 (first parent)
$ git reset --hard HEAD^^ # Same as HEAD~2
💡 HEAD~1 vs HEAD@{1} — Common Confusion!

These look similar but are completely different:

HEAD~1

Commit Ancestry (Graph)

"The parent commit of where HEAD points"

Navigates the commit tree — goes back in commit history

A ← B ← C ← D (HEAD)
HEAD~1 = C
HEAD~2 = B
HEAD@{1}

Reflog History (Time)

"Where HEAD was pointing before the last operation"

Navigates the reflog timeline — goes back in YOUR actions

HEAD@{0} = now (D)
HEAD@{1} = before last action
HEAD@{2} = two actions ago

Example: Why they're different

You're on commit D, then run: git checkout feature

Now HEAD points to commit X (tip of feature branch)

HEAD~1 = parent of X (commit W on feature branch)

HEAD@{1} = commit D (where you WERE before checkout)

Syntax Meaning Navigates
HEAD~1 1 commit back (first parent) Commit graph
HEAD^1 First parent (same as ~1) Commit graph
HEAD^2 Second parent (merge commits) Commit graph
HEAD@{1} Where HEAD was 1 operation ago Reflog (time)
main@{1} Where main branch was 1 operation ago Reflog (time)

Common Use Cases

"Oops, I committed too early"

git reset --soft HEAD~1

Undo commit, keep changes staged, fix and recommit

"I want to reorganize my commits"

git reset HEAD~3

Undo 3 commits, changes stay in files, recommit differently

"Throw away all my local changes"

git reset --hard origin/main

Match remote exactly, discard everything local

"Unstage files I added by mistake"

git reset HEAD file.txt

Unstage specific file (without --hard, keeps file content)

⚠️ Golden Rule

Never reset commits that have been pushed to a shared repository. Use git revert instead!

Safe ↩️ git revert — Undo pushed commits safely

Unlike reset, revert creates a new commit that undoes changes. History is preserved!

# Revert a specific commit
$ git revert abc1234

# Revert without auto-commit (if you want to edit)
$ git revert --no-commit abc1234

# Revert multiple commits
$ git revert abc1234 def5678

reset

Removes commits (rewrites history)

❌ Don't use on pushed commits

revert

Creates new "undo" commit

✅ Safe for pushed commits

Useful ✏️ git commit --amend — Fix the last commit

Made a typo in your commit message? Forgot to add a file? --amend lets you modify the last commit.

git commit --amend variants
# Fix the commit message only
$ git commit --amend -m "New corrected message"

# Add forgotten files to last commit (keep same message)
$ git add forgotten-file.txt
$ git commit --amend --no-edit

# Add files AND change message
$ git add extra-file.txt
$ git commit --amend -m "feat: Add feature with all files"

# Opens your default editor (vim, nano, etc.)

Change message

--amend --no-edit

--amend

--amend rewrites history (creates a new commit with different SHA). Don't amend commits that have been pushed, unless you're the only one working on the branch!

Lifesaver 🔮 git reflog — Your Git Time Machine

reflog = reference log. It's a diary of everywhere your HEAD has pointed.

What is HEAD?

HEAD is a pointer to "where you are now" in Git. Every time you:

  • commit — HEAD moves to the new commit
  • checkout — HEAD moves to the branch/commit
  • reset — HEAD moves backward or forward
  • merge — HEAD moves to the merge commit
  • rebase — HEAD moves through replayed commits

Git records EVERY move HEAD makes in the reflog!

Understanding reflog Output

git reflog
$ git reflog

a1b2c3d HEAD@{0}: commit: Add payment feature
e4f5g6h HEAD@{1}: checkout: moving from main to feature/pay
i7j8k9l HEAD@{2}: reset: moving to HEAD~3 ← Oops!
m0n1o2p HEAD@{3}: commit: Fix critical bug
q3r4s5t HEAD@{4}: commit: Add user auth
u6v7w8x HEAD@{5}: commit: Initial commit
a1b2c3d

Commit SHA (first 7 chars)

HEAD@{0}

Position (0 = current, 1 = previous)

action: description

🆘 Common Recovery Scenarios

"I ran git reset --hard and lost commits!"
# Find the commit before reset
$ git reflog
# Find the SHA you want to recover

# Go back to it!
$ git reset --hard m0n1o2p
"I deleted a branch with unmerged work!"
# Find the last commit on that branch
$ git reflog | grep "feature"

# Recreate the branch at that commit
$ git checkout -b feature/recovered abc1234

Why Does This Work?

When you "delete" commits or branches, Git doesn't actually remove the data immediately. The commits become "unreachable" — they're still in the .git folder, just not connected to any branch.

Git's garbage collector (git gc) only removes unreachable commits after ~30 days. Until then, reflog is your recovery tool!

💡 "In Git, nothing is truly lost until garbage collection runs"

Advanced 🍒 git cherry-pick — Copy a specific commit

Need ONE specific commit from another branch without merging everything?

# Copy commit abc123 to current branch
$ git cherry-pick abc123

# Cherry-pick multiple commits
$ git cherry-pick abc123 def456

Useful for: hotfixes, backporting bug fixes to release branches.

Advanced 🔍 git bisect — Find which commit broke something

Something broke but you don't know when? bisect does binary search through commits to find the culprit.

# Start bisecting
$ git bisect start

# Mark current (broken) as bad
$ git bisect bad

# Mark a known good commit
$ git bisect good abc123

# Git checks out middle commit, you test, then:
$ git bisect good # or git bisect bad

# Repeat until Git finds the bad commit!
$ git bisect reset # when done

Can test 1000 commits in ~10 steps (log₂ 1000 ≈ 10)!

Good to Know 🏷️ git tag — Mark releases and versions

Tags are permanent labels for specific commits, commonly used for releases.

# Create a tag
$ git tag v1.0.0

# Create annotated tag with message
$ git tag -a v1.0.0 -m "First stable release"

# Push tags to remote
$ git push origin --tags

# List all tags
$ git tag

Convention: v1.0.0 (major.minor.patch) - see Semantic Versioning

Prep Time

Interview Questions

Common Git questions you might face. Click to reveal answers.

1. What is a remote in Git?

A remote is a copy of the repository hosted elsewhere (like GitHub, GitLab). It allows collaboration by syncing changes between local and remote repositories. The default remote after cloning is called origin.

2. What's the difference between fork and clone?

Clone downloads a repo to your computer. Fork creates your own copy on GitHub (server-side), which you then clone. Forking is used when you don't have push access to the original repo.

3. What is a Pull Request?

A PR is a proposal to merge changes from one branch to another. It enables code review before integration. PRs are a GitHub/GitLab feature, not a Git feature.

4. When would you use git fetch instead of git pull?

Use fetch when you want to see what changed on remote without immediately merging. Fetch is safer — you can review changes before integrating. pull = fetch + merge.

5. How do you resolve a merge conflict?

1) Open the conflicted file(s), 2) Look for conflict markers (<<<<<<<, =======, >>>>>>>), 3) Decide what code to keep, 4) Remove the conflict markers, 5) Stage the file with git add, 6) Commit the resolution.

6. What's the difference between git reset and git revert?

reset rewrites history by removing commits — only use for local, unpushed work. revert creates a new commit that undoes changes — safe for pushed commits because it doesn't rewrite history.
