💡 What do modern projects use?

Most modern production applications run on Python 3.10 or 3.11. Here's why:

  • 3.10+ introduced structural pattern matching (match/case) — like switch statements
  • 3.11 is up to 25% faster than 3.10 with better error messages
  • Major frameworks (Django, FastAPI) fully support these versions
  • All popular packages are compatible

Rule of thumb: For this course, use Python 3.10 or 3.11.

⚠ What does "Add Python to PATH" mean?

PATH is an environment variable that tells your computer where to find programs when you type commands.

When you type python in the terminal, your computer looks through all folders listed in PATH to find the Python executable.

  • Without PATH: You'd have to type the full path every time: C:\Users\You\AppData\Local\Programs\Python\Python311\python.exe
  • With PATH: Just type python — the system knows where to find it!

If you forgot to check the box: Reinstall Python and check it this time, or manually add it to PATH in System Environment Variables.

Part 2

Virtual Environments

Your project needs its own isolated workspace — separate from other projects!

Why Do We Need Virtual Environments?

🤔 The Real Problem: Conflicting Dependencies

Imagine you're working on two different projects on the same computer:

  • Project A (an old client project) needs requests==2.20.0 (from 2018)
  • Project B (your new project) needs requests==2.31.0 (latest)

Without virtual environments, you can only have one version installed at a time. Installing 2.31.0 might break Project A!

The Problem: Dependency Conflicts 🔥

Project A
requests 2.20
numpy 1.19
⚡
Project B
requests 2.31
numpy 1.24

Both installed globally → CONFLICT! Only one version can exist.

The Solution: Virtual Environments ✓

📦
Project A (venv)
requests 2.20 ✓
numpy 1.19 ✓
📦
Project B (venv)
requests 2.31 ✓
numpy 1.24 ✓

Each project has its own isolated packages — no conflicts!

🎯 Think of it like separate toolboxes

A virtual environment is like giving each project its own toolbox:

  • Each toolbox has exactly the tools (packages) that project needs
  • Adding a tool to one toolbox doesn't affect others
  • You can delete a toolbox without affecting your main system
  • Different projects can use different versions of the same tool

Creating a Virtual Environment

Terminal
$ cd ~/projects/my_restaurant
$ python3 -m venv venv
# Creates a folder called "venv" with isolated Python
$ source venv/bin/activate
(venv) $ ← Notice the (venv) prefix! You're now "inside" the venv.
(venv) $ deactivate
$ ← Back to normal (no prefix)
Command Prompt / PowerShell
> cd C:\projects\my_restaurant
> python -m venv venv
# Creates a folder called "venv" with isolated Python
> venv\Scripts\activate
(venv) > ← Notice the (venv) prefix! You're now "inside" the venv.
(venv) > deactivate
> ← Back to normal (no prefix)

💡 Can I name my virtual environment something other than "venv"?

Yes! The name "venv" is just a convention. You can use any name:

$ python3 -m venv myenv
$ python3 -m venv .venv
$ python3 -m venv restaurant_env

Best Practice: Use venv or .venv (with dot, makes it hidden on Unix systems). This is what most projects use, so other developers will immediately recognize it.

💡 Best Practice: Never Commit venv to Git!

Add these to your .gitignore:

venv/
.venv/
env/
__pycache__/

Why? Virtual environments are:

  • Large (can be 50-100+ MB)
  • System-specific (won't work on different OS)
  • Easily recreatable from requirements.txt

Instead, share dependencies via requirements.txt:

(venv) $ pip freeze > requirements.txt
# Creates a file listing all installed packages
(venv) $ pip install -r requirements.txt
# Another developer runs this to get the same packages

pip: Python Package Manager

pip installs Python packages from the Python Package Index (PyPI) — a repository of 400,000+ packages.

📦 What Is a Package Manager?

At its core, a package manager is just a tool that answers one question:

"I need software X — go find it, download it, and set it up for me."

Before package managers existed, you'd manually:

  1. Google the library
  2. Find the right version
  3. Download a zip file
  4. Figure out where to put it
  5. Manually handle anything it depends on

A package manager automates all of that. ✅

🏫 The Real-World Analogy

Think of it like an app store, but for code libraries:

🤔 Where does pip come from?

pip comes bundled with Python! When you install Python 3.4+, pip is automatically included.

  • pip = "Pip Installs Packages" (recursive acronym!)
  • It downloads packages from pypi.org (Python Package Index)

Part 4

What Is Django?

A high-level Python web framework that encourages rapid development and clean, pragmatic design.

🎸 Where does the name "Django" come from?

Django is named after Django Reinhardt (1910-1953), a legendary Belgian-born Romani-French jazz guitarist. The framework's creators, Adrian Holovaty and Simon Willison, were jazz enthusiasts.

"Django makes it easier to build better web apps more quickly and with less code."

🔋 What does "batteries-included" mean?

Like how a toy that comes with batteries is ready to play immediately, Django comes with everything you need to build web applications out of the box:

  • Admin Panel — Auto-generated interface to manage your data
  • ORM (Object-Relational Mapping) — Talk to databases using Python, not SQL
  • Authentication — User login, logout, permissions built-in
  • Form handling — Validation and rendering
  • Security — Protection against common attacks (XSS, CSRF, SQL injection)

Compare to Flask: Flask is a "micro-framework" — it gives you the minimum and you add batteries (extensions) yourself.

📝 __init__.py — The Package Marker

Why does this file exist? In Python, a folder is just a folder. But a folder with __init__.py becomes a Python package that can be imported.

⚙ settings.py — The Complete Configuration Hub

This file contains ALL configuration for your Django project. Let's understand every default setting!

1. BASE_DIR — The Project Root

What does this do?
from pathlib import Path

# Build paths inside the project like this: BASE_DIR / 'subdir'
BASE_DIR = Path(__file__).resolve().parent.parent

# Breaking it down:
# Path(__file__)      → Path to this settings.py file
# .resolve()          → Convert to absolute path (no .. or .)
# .parent             → Go up one directory (to restaurant_project/)
# .parent             → Go up again (to project root with manage.py)

Why? All other paths (database, templates, static files) are relative to this. Using Path makes paths work on Windows, macOS, and Linux.

The middleware stack

    # ↑ URL normalization (adds trailing slashes)

📚 Templates vs JSON APIs

Templates (old approach): Server generates complete HTML pages. Used when browsers couldn't run JavaScript well.

JSON APIs (modern approach): Server sends data as JSON. Frontend (React, Vue, mobile apps) renders the UI. This is what we'll build!

Django Apps

Apps are modular building blocks — each one handles a specific feature.

🤔 Why can't we just put everything in the project folder?

You could, but it would quickly become a mess! Here's why Django uses apps:

  • Separation of concerns: Menu logic stays in menu/, orders in orders/
  • Reusability: You can copy an app (e.g., users/) to another project
  • Team collaboration: Different developers can work on different apps without conflicts
  • Testing: Test each app independently
  • Maintainability: A 1000-line models.py is harder to manage than 10 files of 100 lines each

🤔 Django Apps vs Microservices: A Common Confusion

Many developers ask: "Are Django apps like microservices?"
Short answer: No, but they share the same motivation.

✅ The Similarities

Both Django apps and microservices aim for:

  • Modularity: Separate code for separate features
  • Separation of concerns: menu/ handles menus, orders/ handles orders
  • Team ownership: Teams can work on different parts
  • Focused scope: One component = one domain

📚 Historical Context

  • Django (2005) predates microservices trend (~2011-2014 by Netflix, Amazon)
  • Django apps are inspired by Unix philosophy: "Do one thing well"
  • More accurately called a "Modular Monolith"
  • Instagram, Pinterest, Disqus scaled Django monoliths to millions of users

💡 Which Should You Use?

Start with Django apps (monolith) → Graduate to microservices only if needed.

Why? Simpler to develop, test, deploy. No network latency. Shared transactions. Easier debugging.

"Don't start with microservices. Start with a modular monolith, and extract services only when you have a proven need." — Martin Fowler

Views & URLs

Views are your waiters — they receive requests and return responses. URLs route customers to the right waiter.

How URL Routing Works (Detailed)

Request: GET /menu/items/5/

1
Browser sends request to Django
User visits http://localhost:8000/menu/items/5/
2
Project urls.py checks the URL
Sees path('menu/', include('menu.urls'))
→ "URL starts with 'menu/', let me delegate to the menu app!"
3
menu/urls.py takes over
Receives remaining URL: items/5/
Matches path('items/<int:item_id>/', views.item_detail)
4
View function is called
views.item_detail(request, item_id=5)
The 5 from URL becomes item_id parameter!
5
View returns response
View processes the request and returns HttpResponse or JsonResponse

Step 1: Create a View

menu/views.py 
from django.http import HttpResponse, JsonResponse

def home(request):
    """The simplest possible view."""
    return HttpResponse("Welcome to our restaurant!")

def menu_list(request):
    """Return menu as JSON (modern API style)."""
    menu = [
        {'name': 'Pasta', 'price': 15},
        {'name': 'Pizza', 'price': 12},
    ]
    return JsonResponse({'items': menu})

def item_detail(request, item_id):
    """View that receives item_id from URL."""
    return HttpResponse(f"Showing item #{item_id}")

📚 Different Ways to Write Views in Django

Django offers multiple approaches. Understanding these helps you choose the right tool.

Official Docs: docs.djangoproject.com/en/5.0/topics/http/views/

1. Function-Based Views (FBVs)

The core idea — one function, branch on request.method:

from django.shortcuts import render
from django.http import JsonResponse

def my_view(request):

    if request.method == 'GET':
        return render(request, 'home.html')

    elif request.method == 'POST':
        data = request.POST.get('username')
        return render(request, 'home.html', {'user': data})

    elif request.method == 'DELETE':
        return JsonResponse({'message': 'deleted'})

Wire it in urls.py — same URL, different behavior based on HTTP method:

# urls.py
from django.urls import path
from . import views

urlpatterns = [
    path('home/', views.my_view),
]
MethodTypical Use
GETFetch/display data
POSTSubmit a form, create something
PUT/PATCHUpdate something
DELETEDelete something

For APIs, Django REST Framework's @api_view decorator is cleaner:

from rest_framework.decorators import api_view
from rest_framework.response import Response

@api_view(['GET', 'POST'])
def my_view(request):
    if request.method == 'GET':
        return Response({'message': 'hello'})
    elif request.method == 'POST':
        return Response({'received': request.data})

2. Class-Based Views (CBVs)

Object-oriented approach — inherit from Django's view classes:

from django.views import View
from django.http import JsonResponse

class MenuListView(View):
    """Class-based view for menu items."""

    def get(self, request):
        """Handle GET requests."""
        items = MenuItem.objects.all()
        return JsonResponse({'items': list(items.values())})

    def post(self, request):
        """Handle POST requests."""
        pass

# In urls.py: path('', MenuListView.as_view(), name='menu_list')

3. Generic Class-Based Views (GCBVs)

Pre-built views for common CRUD patterns:

from django.views.generic import ListView, DetailView, CreateView

class MenuListView(ListView):
    model = MenuItem
    template_name = 'menu/list.html'
    context_object_name = 'items'

class MenuDetailView(DetailView):
    model = MenuItem
    template_name = 'menu/detail.html'

class MenuCreateView(CreateView):
    model = MenuItem
    fields = ['name', 'price', 'description']
    success_url = '/menu/'

Industry Practice: Most production apps use a mix — GCBVs for standard CRUD, FBVs for custom endpoints, DRF ViewSets for APIs.

⚠ Should All Logic Live in Views? The Service Layer

Short answer: No! Views should be thin. They should handle HTTP concerns only.

❌ The Problem: Fat Views

DON'T do this!
# ❌ BAD: Too much logic in the view
def create_order(request):
    data = json.loads(request.body)

    # Validation, business logic, database queries ALL mixed together
    total = 0
    for item in data['items']:
        menu_item = MenuItem.objects.get(id=item['id'])
        total += menu_item.price * item['quantity']
        if menu_item.stock < item['quantity']:
            return JsonResponse({'error': 'Out of stock'}, status=400)

    # Apply discount
    if total > 100:
        total *= 0.9

    # ... more and more logic ...

Problems: Hard to test, hard to reuse, hard to read!

✅ The Solution: Service Layer

Views Layer      (HTTP handling only)
      │
      ▼
Services Layer   (Business logic) ← NEW!
      │
      ▼
Models Layer     (Database operations)

Where to Put Services?

📁 menu/
📝 models.py Data models
📝 views.py HTTP handlers (thin!)
📝 services.py Business logic ⭐ NEW!
📝 selectors.py Complex queries (optional)
services.py
# menu/services.py
class OrderService:
    """Handles all order-related business logic."""

    @staticmethod
    def calculate_total(items_data):
        # Business logic here...
        pass

    @classmethod
    def create_order(cls, user, items_data):
        """Can be called from views, CLI, Celery tasks..."""
        cls.check_inventory(items_data)
        total = cls.calculate_total(items_data)
        return Order.objects.create(user=user, total=total)
views.py (now thin!)
# menu/views.py
from .services import OrderService

def create_order(request):
    """Handle HTTP request only."""
    data = json.loads(request.body)

    # Delegate to service layer
    order = OrderService.create_order(
        user=request.user,
        items_data=data['items']
    )

    return JsonResponse({'order_id': order.id})
Benefits:
  • Testable: Test business logic without HTTP overhead
  • Reusable: Call from views, CLI commands, Celery tasks
  • Readable: Views are thin, services are focused

Step 2: Create App URLs

menu/urls.py (CREATE this file!) 
from django.urls import path
from . import views  # Relative import: "." = current folder

urlpatterns = [
    path('', views.home, name='home'),
    path('list/', views.menu_list, name='menu_list'),
    path('item/<int:item_id>/', views.item_detail, name='item_detail'),
]
💡 Why from . import views?

The dot (.) means "current package" — like ./ in terminal. This relative import keeps your app portable. Never write from myproject.menu import views — that hardcodes the project name and breaks if you rename or reuse the app.

🔍 Breaking Down Each path() Call — Parameter by Parameter

The path() function signature:

path(route, view, kwargs=None, name=None)

Pattern 1: path('', views.home, name='home')

ParameterValueExplanation
route '' Empty string = matches base URL. If included at /menu/, this matches /menu/ exactly
view views.home The function home from views.py that handles this request
name 'home' Unique identifier. Used in templates: {% url 'home' %}

Pattern 2: path('list/', views.menu_list, name='menu_list')

ParameterValueExplanation
route 'list/' Matches path list/. Combined with /menu/ prefix = /menu/list/
view views.menu_list The function menu_list handles this request
name 'menu_list' Unique name for reverse URL lookup

Trailing slash: Django convention is to always include trailing slashes. CommonMiddleware redirects /menu/list to /menu/list/.

Pattern 3: path('item/<int:item_id>/', views.item_detail, name='item_detail')

ParameterValueExplanation
route 'item/<int:item_id>/' Matches item/ followed by an integer. <int:item_id> is a path converter
view views.item_detail The function receives the captured value as an argument
name 'item_detail' Used as {% url 'item_detail' item_id=5 %}

Breaking down <int:item_id>:

<int:item_id>
  │     │
  │     └── Variable name passed to view function
  └── Converter type (int = only matches integers)
What happens at runtime
# URL: /menu/item/42/

def item_detail(request, item_id):  # item_id passed automatically!
    # item_id = 42 (as an integer, not string!)
    return HttpResponse(f"Showing item #{item_id}")

from django.contrib import admin

]

Breaking Down include()

ParameterValueExplanation
route 'menu/' URL prefix — all menu URLs start with /menu/
include(...) include('menu.urls') Delegate to menu/urls.py for further matching

⚠ How include() Works — Step by Step

  1. User visits /menu/item/42/
  2. Django checks restaurant_project/urls.py
  3. Does path start with admin/? No
  4. Does path start with menu/? Yes!
  5. Strip prefix: remaining path is item/42/
  6. Django now checks menu/urls.py with item/42/
  7. Match found: item/<int:item_id>/ → captures item_id=42
  8. Call views.item_detail(request, item_id=42)

URL Path Converters

Converter Matches Example URL
<str:name> Any non-empty string (no /) /category/desserts/
<int:id> Zero or positive integers /item/42/
<slug:title> Letters, numbers, hyphens, underscores /dish/pasta-carbonara/
<uuid:pk> UUID format /order/550e8400-e29b-.../

Career Prep

Interview Questions

What is a virtual environment and why use one? +

A virtual environment is an isolated Python environment for project-specific dependencies. It prevents conflicts between projects that need different package versions. For example, Project A can use Django 3.2 while Project B uses Django 4.2.

What's the difference between a Django project and an app? +

Project: The entire website configuration (settings, root URLs). One per site.
App: A module handling one feature (menu, orders, users). Multiple per project.

Apps are reusable — you can copy an app to another project!

How does URL routing work in Django? +

When a request comes in, Django matches the URL against patterns in urls.py. When matched, it calls the associated view function. The include() function delegates to app-specific URL configurations, enabling modular URL design.

What does DEBUG = True do and why is it dangerous in production? +

DEBUG mode shows detailed error pages with code snippets, stack traces, and variable values. This is helpful for development but dangerous in production as it exposes sensitive information like database queries, file paths, and secret keys.
