Pre-Class Quiz

Before diving into REST APIs, let's review Django Models & Admin.

💡 Question 1
Your MenuItem model has 10,000 rows in production. You add this new field and run makemigrations: 
category = models.ForeignKey(Category, on_delete=models.CASCADE)
A Migration creates successfully with no prompts
B Django asks for a default value because existing rows need a category_id
C Django automatically sets all existing rows to the first Category
D The migration fails because ForeignKey cannot be added to existing tables

Explanation

When you add a NOT NULL field (which ForeignKey is by default) to a table with existing rows, Django must know what value to put in that column for those rows.

Django prompts:

  • Option 1: Provide a one-time default now
  • Option 2: Quit and add null=True or default= to the field

Safe approach: Add null=True first, migrate, backfill data, then remove null=True in a second migration.

💡 Question 2
You added a new field is_featured = models.BooleanField() to MenuItem. What happens when you run makemigrations?
A The field is added to the database immediately
B Django asks for a default value (existing rows need a value)
C Migration succeeds silently with no prompts
D Django deletes all existing rows to add the field

Explanation

When adding a field to a table with existing rows, Django needs to know what value to put in that field for existing rows.

Best practice: Add default=False to the field definition to avoid the prompt:

is_featured = models.BooleanField(default=False)

The Database Design Principle: NOT NULL + DEFAULT

When you add a new column to a table that already has rows, the database needs a value for that column in existing rows. This is the NOT NULL constraint — a fundamental database integrity rule.

BEFORE MIGRATION:
┌────┬────────────┬───────┐
│ id │ name       │ price │
├────┼────────────┼───────┤
│ 1  │ Pasta      │ 15.00 │
│ 2  │ Salmon     │ 22.00 │
└────┴────────────┴───────┘

AFTER ADDING is_featured (BooleanField):
┌────┬────────────┬───────┬─────────────┐
│ id │ name       │ price │ is_featured │
├────┼────────────┼───────┼─────────────┤
│ 1  │ Pasta      │ 15.00 │ ??? ← What goes here? │
│ 2  │ Salmon     │ 22.00 │ ??? ← Django doesn't know! │
└────┴────────────┴───────┴─────────────┘

Rule: Every column must either:

  • Allow NULL (null=True) — empty is okay
  • Have a DEFAULT — use this value when not specified
  • Be required on INSERT — reject incomplete data

Best Practice Examples

# ✅ GOOD: Provide a sensible default
is_featured = models.BooleanField(default=False)
created_at = models.DateTimeField(auto_now_add=True)
status = models.CharField(max_length=20, default='pending')
quantity = models.IntegerField(default=1)

# ✅ ALSO OK: Allow NULL (when "unknown" is valid)
deleted_at = models.DateTimeField(null=True, blank=True)
middle_name = models.CharField(max_length=50, null=True, blank=True)

# ❌ BAD: No default, not nullable — will prompt!
is_featured = models.BooleanField()  # Migration will ask!

When to Use Each Approach

Scenario Approach Example
Boolean flags default=False is_active, is_featured
Timestamps auto_now_add=True created_at
Status fields default='initial' status='pending'
Optional data null=True, blank=True deleted_at, bio
Numeric counters default=0 view_count, stock
💡 Production Mindset:

Always think about existing data. Your migration will run on a database that already has rows. What should those rows contain for this new field? Plan it in your model definition, not during migration.

Core Concept

What is REST?

REST is a set of conventions for building APIs. It's not a technology — it's a design philosophy.

📚 REST = REpresentational State Transfer

Invented by Roy Fielding in his 2000 PhD thesis. It's a way to design web services that are simple, scalable, and predictable.

REST Principles

🔗

1. Resource-Based URLs

URLs represent things (nouns), not actions (verbs).

✓ /api/menu-items/

✗ /api/getMenuItems/

🎮

2. HTTP Methods = Actions

Use HTTP verbs to indicate what you want to do with the resource.

GET = Read
POST = Create

📦

3. Stateless

Each request contains all info needed. Server doesn't remember previous requests.

📄

4. Standard Responses

Use HTTP status codes (200, 201, 404, etc.) and consistent JSON format.

HTTP Methods (CRUD Operations)

HTTP Method CRUD URL Example Purpose
GET Read /api/menu-items/ List all items
GET Read /api/menu-items/5/ Get item #5
POST Create /api/menu-items/ Create new item
PUT Update (full) /api/menu-items/5/ Replace item #5
PATCH Update (partial) /api/menu-items/5/ Update some fields
DELETE Delete /api/menu-items/5/ Delete item #5

Edge Cases: Validation & Complex Queries

🤔 What method for validation without saving?

Use POST even though you're not creating anything!

POST /orders/validate/
Body: {"customer_name": "John", "customer_phone": "123"}
Response: {"valid": false, "errors": {"customer_phone": "Must be 10+ digits"}}

Why POST? We're sending a request body. GET requests shouldn't have bodies.

🔍 When does GET become POST?

When the query is too complex for URL parameters!

Simple Query — Use GET 
GET /search?q=pizza&price_max=500
Complex Query — Use POST 
POST /search/
{
  "query": {
    "bool": {
      "must": [
        {"match": {"name": "pizza"}},
        {"range": {"price": {"lte": 500}}},
        {"geo_distance": {...}}
      ]
    }
  }
}

Why? URL length limits (~2000 chars), complex nested JSON, better readability.

Real examples: Elasticsearch, GraphQL (always POST), analytics dashboards, report filters.

HTTP Status Codes

Code Meaning When to Use
200 OK Success GET, PUT, PATCH succeeded
201 Created Resource created POST succeeded
204 No Content Success, no body DELETE succeeded
400 Bad Request Invalid data Validation failed
401 Unauthorized Not logged in Missing/invalid auth token
403 Forbidden No permission Logged in but can't access

Stateless: What Does It Mean?

In REST, the server doesn't remember anything about previous requests. Each request must contain all the information needed to process it.

✗ Stateful (NOT REST)

Server remembers who you are between requests

Client                                    Server
  │                                         │
  │─────── Request 1: "Login as Prateek" ──►│
  │                                         │  Server stores: session["user"] = "Prateek"
  │◄────── "OK, I remember you" ────────────│
  │                                         │
  │─────── Request 2: "Show my orders" ────►│
  │                                         │  Server thinks: "Who is this? Let me check session..."
  │                                         │  Server finds: session["user"] = "Prateek"
  │◄────── "Here are Prateek's orders" ─────│
  │                                         │

  Problem: Server must remember state for EVERY user!
           What if you have 1 million users? 10 servers?

✓ Stateless (REST)

Each request contains everything needed

Client                                              Server
  │                                                   │
  │─── Request 1: "Login" ───────────────────────────►│
  │                                                   │  Server: "Here's a token, keep it safe"
  │◄── "Here's your token: abc123xyz" ────────────────│
  │                                                   │
  │─── Request 2: "Show orders" + Token: abc123xyz ──►│
  │                                                   │  Server: "Token says this is Prateek"
  │                                                   │  (No session lookup needed!)
  │◄── "Here are Prateek's orders" ───────────────────│
  │                                                   │

  Benefit: Server doesn't store anything!
           Any server can handle any request.
           Easy to scale to millions of users.

Why Stateless Matters

Aspect Stateful Stateless (REST)
Server Memory Must store session for each user Stores nothing between requests
Scaling Hard — sessions must be shared across servers Easy — any server can handle any request
Load Balancing Sticky sessions needed (same user → same server) Any server works, simple round-robin
Server Crash User loses session, must re-login No impact — next request goes to another server
Authentication Session cookies Token in header (JWT, API key)
🤔 "But wait, what about login?"

Great question! In stateless REST:

  • Client logs in once → receives a token (like a JWT)
  • Client sends this token with every request in the header
  • Server validates the token → knows who the user is
  • Server never stores "who is logged in" — the token contains that info!
Stateless Request with Token
# Every request includes the token
curl -X GET http://api.restaurant.com/orders/ \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIs..."

# Server decodes token → knows user is "Prateek"
# No session lookup, no server-side storage!
💡 Key Insight

In REST, the URL identifies the resource and the HTTP method determines the action. You don't need verbs in URLs!

What are Serializers?

from rest_framework import serializers
from .models import Order


class OrderSerializer(serializers.ModelSerializer):
    class Meta:
        model = Order
        fields = [
            'id',
            'customer_name',
            'customer_phone',
            'status',
            'total_amount',
            'notes',
            'created_at',

Step 3: Adding Validation

Serializers can validate data automatically based on model fields, but you can also add custom validation:

orders/serializers.py - With Custom Validation
from rest_framework import serializers
from .models import Order


class OrderSerializer(serializers.ModelSerializer):
    class Meta:
        model = Order
        fields = ['id', 'customer_name', 'customer_phone', 'status',
                  'total_amount', 'notes', 'created_at', 'updated_at']
        read_only_fields = ['id', 'created_at', 'updated_at']

    # Field-level validation
    def validate_customer_name(self, value):
        if len(value) < 2:
            raise serializers.ValidationError("Name must be at least 2 characters")
        return value.strip()

    def validate_customer_phone(self, value):
        if len(value) < 10:
            raise serializers.ValidationError("Phone must be at least 10 digits")
        if not value.isdigit():
            raise serializers.ValidationError("Phone must contain only digits")
        return value

    # Object-level validation (across multiple fields)
    def validate(self, data):
        if data.get('status') == 'delivered' and not data.get('total_amount'):
            raise serializers.ValidationError("Delivered orders must have a total amount")
        return data
💡 Validation Methods
  • validate_<field>(self, value) — Validates a single field
  • validate(self, data) — Validates across multiple fields
  • raise serializers.ValidationError("message") — Reject invalid data

Serializer Field Types

Django Model Field Serializer Handles JSON Output

DecimalField Converts to string automatically "199.99"
DateTimeField ISO format conversion "2024-01-15T10:30:00Z"

DRF Views

Step 3: DRF Views

Now let's replace our messy plain Django views with clean DRF views.

Before vs After: The Power of DRF

✗ Plain Django (~50 lines per view)

@csrf_exempt
def create_order(request):
    if request.method != 'POST':
        return JsonResponse({"error": ...})
    try:
        data = json.loads(request.body)
    except:
        return JsonResponse({"error": ...})
    if not data.get('customer_name'):
        return JsonResponse({"error": ...})
    # ... 30+ more lines ...
    return JsonResponse({
        "id": order.id,
        # ... manual conversion ...
    })

✓ With DRF (~15 lines total)

@api_view(['GET', 'POST'])
def order_list(request):
    if request.method == 'GET':
        orders = Order.objects.all()
        serializer = OrderSerializer(orders, many=True)
        return Response(serializer.data)

    elif request.method == 'POST':
        serializer = OrderSerializer(data=request.data)
        if serializer.is_valid():
            serializer.save()
            return Response(serializer.data, status=201)
        return Response(serializer.errors, status=400)
💡 What Changed?
  • No @csrf_exempt — DRF handles CSRF properly
  • No json.loads() — request.data is already parsed
  • No manual validation — serializer.is_valid() does it all
  • No manual conversion — serializer.data handles types

Three Ways to Write DRF Views

Approach What It Is Best For Example Endpoints
@api_view Function decorator Simple, one-off endpoints that don't fit CRUD /health/, /login/, /send-otp/
APIView Class with separate methods Custom logic, override auth/permissions /dashboard-stats/, /reports/
ViewSet All CRUD in one class Model-backed resources (90% of APIs) /orders/, /menu/, /users/
💡 The Practical Rule
  • Start with ModelViewSet for any model-backed resource
  • Use APIView when you need custom behavior that doesn't fit CRUD
  • Use @api_view for standalone utility endpoints
  • In a real project, you'll use all three!

Option 1: @api_view Decorator

📌 When to use @api_view:

Simple one-off endpoints that don't fit the standard CRUD pattern. Things like /health-check/, /login/, /send-otp/. It's a function — quick to write, easy to read. But if you start adding list, create, retrieve, update, delete — it gets messy fast with if/elif chains.

✅ Good use: Simple utility endpoints
from rest_framework.decorators import api_view
from rest_framework.response import Response

@api_view(['GET'])
def health_check(request):
    return Response({'status': 'healthy'})

@api_view(['POST'])
def send_otp(request):
    phone = request.data.get('phone')
    # send OTP logic...
    return Response({'message': 'OTP sent'})
⚠️ Gets messy: CRUD operations (use ViewSet instead!)
@api_view(['GET', 'POST'])
def order_list(request):
    if request.method == 'GET':
        orders = Order.objects.all()
        serializer = OrderSerializer(orders, many=True)
        return Response(serializer.data)
    elif request.method == 'POST':
        serializer = OrderSerializer(data=request.data)
        if serializer.is_valid():
            serializer.save()
            return Response(serializer.data, status=201)
        return Response(serializer.errors, status=400)

# Need another function for detail view (GET, PUT, DELETE)...
# This gets repetitive fast!

Option 2: APIView Class

📌 When to use APIView:

The class-based version of @api_view. Gives you more structure — separate get(), post(), put() methods instead of one function with if/else. Good when you need custom logic that doesn't map cleanly to CRUD, or when you want to override things like authentication or permissions at the view level.

✅ Good use: Dashboard stats (not standard CRUD)
from rest_framework.views import APIView
from rest_framework.response import Response

class DashboardStatsView(APIView):
    """Custom endpoint for dashboard statistics"""

    def get(self, request):
        return Response({
            'total_orders': Order.objects.count(),
            'pending': Order.objects.filter(status='pending').count(),
            'delivered_today': Order.objects.filter(
                status='delivered',
                updated_at__date=date.today()
            ).count(),
        })

# urls.py: path('dashboard-stats/', DashboardStatsView.as_view())
⚠️ Also works for CRUD but gets verbose
class OrderList(APIView):
    def get(self, request):
        orders = Order.objects.all()
        serializer = OrderSerializer(orders, many=True)
        return Response(serializer.data)

    def post(self, request):
        serializer = OrderSerializer(data=request.data)
        if serializer.is_valid():
            serializer.save()
            return Response(serializer.data, status=201)
        return Response(serializer.errors, status=400)

# Need another class for detail view... Use ViewSet instead!

Option 3: ViewSets (Recommended)

📌 When to use ViewSet:

For standard CRUD resources — anything that maps to a database model with list, create, retrieve, update, destroy operations. Combined with a Router, it eliminates boilerplate entirely. ModelViewSet goes even further — connect it to a serializer and a queryset and you get all 5 endpoints with almost zero code.

orders/views.py - ViewSet (minimal code!)
from rest_framework import viewsets
from .models import Order
from .serializers import OrderSerializer


class OrderViewSet(viewsets.ModelViewSet):
    """
    A ViewSet for viewing and editing orders.
    Provides: list, create, retrieve, update, destroy
    """
    queryset = Order.objects.all()
    serializer_class = OrderSerializer
🚀 That's it! 5 lines of code gives you full CRUD!

ViewSets automatically provide: list, create, retrieve, update, partial_update, destroy.
