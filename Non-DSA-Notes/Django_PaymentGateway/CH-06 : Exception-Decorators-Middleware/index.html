Part 1

Exception Handling

Gracefully handle errors instead of crashing your application.

🤔 Let's Start with a Question
What happens when you call MenuItem.objects.get(id=99999) and no item with that ID exists? 
# In your Django view:

A Returns None silently
B Returns an empty QuerySet []

💥 The Program Crashes!

When .get() doesn't find a matching record, it raises a MenuItem.DoesNotExist error (called an exception). If you don't handle it, Django shows a 500 Internal Server Error to the user.

This is the problem we'll solve in this class!

📚 So What is an Exception?

An exception is Python's way of saying "something went wrong!" When code fails (database lookup finds nothing, file doesn't exist, network times out), Python raises an exception. If you don't handle it, your program crashes.

💡 The Goal of Exception Handling:

Catch these errors and respond gracefully instead of crashing. Show users a friendly message, log the error for debugging, and keep your app running.

The Solution: try-except Block

Catching the Error 
try:
    # Code that might fail
    item = MenuItem.objects.get(id=99999)
except MenuItem.DoesNotExist:
    # What to do if it fails - instead of crashing!
    item = None
    print("Item not found")
WITHOUT try-except: .get(99999) → 💥 CRASH! → User sees 500 error page WITH try-except: .get(99999) → except catches it → Your code handles it gracefully

The Full Exception Flow: try-except-else-finally

🤔 Why do we need else? Can't we just write code after the except block?

Code after the except block runs regardless of whether an exception occurred. The else block runs only when there was NO exception.

# WITHOUT else - this runs even after exception!
try:
    file = open("data.txt")
except FileNotFoundError:
    print("File not found")
data = file.read()  # CRASH! file was never opened!

# WITH else - this only runs if file opened successfully
try:
    file = open("data.txt")
except FileNotFoundError:
    print("File not found")
else:
    data = file.read()  # Safe! Only runs if open() succeeded
🔒 When to use finally?

Use finally for cleanup code that MUST run - even if an exception occurs, even if there's a return statement!

🤔 What is a "Resource Leak"?

Your computer has limited resources (memory, file handles, network connections). When you open a file or database connection, the OS reserves resources for it. If you don't close them:

  • File handles: OS limits how many files can be open simultaneously (often ~1000). If you keep opening without closing, eventually you can't open any more files!
  • Database connections: Databases have a "connection pool" (e.g., 20 connections). Each open connection uses one. If your code crashes without closing, that connection stays reserved until the database times it out (could be minutes!).

Multiple Exception Handling Patterns

# Pattern 1: Different handlers for different exceptions
try:
    item = MenuItem.objects.get(id=item_id)
except MenuItem.DoesNotExist:
    return Response({"error": "Item not found"}, status=404)
except MenuItem.MultipleObjectsReturned:
    return Response({"error": "Multiple items found"}, status=400)

# Pattern 2: Same handler for multiple exceptions
try:
    process_data()
except (ValueError, TypeError, KeyError) as e:
    logger.error(f"Data error: {e}")
    return None

# Pattern 3: Catch-all (use sparingly!)
try:
    external_api_call()
except Exception as e:
    logger.error(f"Unexpected error: {e}")
    raise  # Re-raise after logging
⚠ Avoid bare except:

Never use except: without specifying an exception type. It catches everything including KeyboardInterrupt and SystemExit!

Re-raising Exceptions

🔄 When to re-raise an exception?

Sometimes you want to catch an exception, do something (like logging), but still let the exception propagate up. Use raise without arguments.

How would we handle this in production?

✔ Production-Ready Code
from django.db import transaction

def create_order(request):
    # 1. Validate input first (before any DB operations)
    item_id = request.data.get('item_id')  # .get() returns None, no KeyError
    quantity = request.data.get('quantity')

    if not item_id or not quantity:
        return Response({"error": "item_id and quantity required"}, status=400)

    try:
        quantity = int(quantity)
    except (ValueError, TypeError):
        return Response({"error": "quantity must be a number"}, status=400)

    # 2. Database operations with proper error handling
    try:
        item = MenuItem.objects.get(id=item_id)
    except MenuItem.DoesNotExist:
        return Response({"error": "Item not found"}, status=404)

    # 3. Business logic validation
    if item.stock < quantity:
        return Response({"error": f"Only {item.stock} available"}, status=400)

    # 4. Create order in a transaction
    with transaction.atomic():
        order = Order.objects.create(user=request.user, item=item, qty=quantity)
        item.stock -= quantity
        item.save()

    return Response(OrderSerializer(order).data, status=201)
Key patterns:
  • Use .get() instead of [] to avoid KeyError
  • Validate input types before using them
  • Handle specific exceptions with appropriate error messages
  • Use transactions for operations that must succeed together

Alternative: Let Serializers Handle Validation

🔨 With DRF Serializers (Cleaner!)
# serializers.py
class OrderCreateSerializer(serializers.Serializer):
    item_id = serializers.IntegerField()
    quantity = serializers.IntegerField(min_value=1)

    def validate_item_id(self, value):
        try:
            self.item = MenuItem.objects.get(id=value)
        except MenuItem.DoesNotExist:
            raise serializers.ValidationError("Item not found")
        return value

    def validate(self, data):
        if self.item.stock < data['quantity']:
            raise serializers.ValidationError(
                f"Only {self.item.stock} available"
            )
        return data

# views.py - Much cleaner!
@api_view(['POST'])
def create_order(request):
    serializer = OrderCreateSerializer(data=request.data)
    serializer.is_valid(raise_exception=True)  # Auto 400 on error

    with transaction.atomic():
        order = Order.objects.create(
            user=request.user,
            item=serializer.item,
            qty=serializer.validated_data['quantity']
        )
    return Response(OrderSerializer(order).data, status=201)
💡 Why serializers? They handle type conversion, validation, and error responses automatically. Your view stays clean and focused on business logic.

User Enumeration Vulnerability

An attacker can determine which emails are registered by observing different responses:

test@victim.com → "User not found" (404) = NOT registered john@victim.com → "Reset email sent" (200) = IS registered!

The Fix:

def reset_password(request):
    email = request.data['email']
    try:
        user = User.objects.get(email=email)
        send_reset_email(user)
    except User.DoesNotExist:
        pass  # Silently ignore

    # ALWAYS return same message
    return Response({
        "message": "If this email is registered, you'll receive a reset link."
    })

Creating Custom Exceptions

🤔 When to create custom exceptions?
  • Business logic errors specific to your domain (InsufficientStock, RestaurantClosed)
  • When you need custom error codes for API clients
  • When built-in exceptions don't express the error clearly
myapp/exceptions.py 
# Where to put this file: myapp/exceptions.py
# Keep exceptions close to the app that uses them

class InsufficientStockError(Exception):
    """Raised when order quantity exceeds available stock."""
    def __init__(self, available, requested):
        self.available = available
        self.requested = requested
        self.message = f"Only {available} available, {requested} requested"
        super().__init__(self.message)

class RestaurantClosedError(Exception):
    """Raised when attempting to order outside business hours."""
    pass

# Usage in views:
from myapp.exceptions import InsufficientStockError

def create_order(request):
    if item.stock < quantity:
        raise InsufficientStockError(item.stock, quantity)
📌 Best Practices for Custom Exceptions:
  • Location: Create exceptions.py in your app folder
  • Naming: Always end with Error (e.g., InsufficientStockError)
  • Inherit from Exception (or more specific built-in exceptions)
  • Add context: Store relevant data as attributes
  • Document: Add a docstring explaining when it's raised

Functions that wrap other functions to add behavior without modifying the original.

Prerequisite: Understanding *args and **kwargs

Before learning decorators, you need to understand *args and **kwargs - they allow functions to accept any number of arguments.

*args collects ALL extra positional arguments into a tuple.

💡 Why are *args and **kwargs important for decorators?

A decorator's wrapper function needs to accept ANY arguments that the original function might receive. Using *args, **kwargs lets the wrapper forward all arguments to the original function.

def wrapper(*args, **kwargs):
    # Now this wrapper can accept ANY function's arguments!
    result = original_func(*args, **kwargs)
    return result

What is a Decorator?

📚 Decorators in One Sentence

A decorator is a function that takes another function and extends its behavior without modifying the original function's code. Think of it like wrapping a gift - the gift stays the same, but you add something extra around it.

Understanding the @ Syntax

The @ syntax is just shorthand 
# This decorator syntax:
@my_decorator
def my_function():
    pass

# Is EXACTLY the same as:
def my_function():
    pass
my_function = my_decorator(my_function)

# The decorator receives your function
# and returns a NEW function that wraps it

How to Write a Decorator

The Decorator Template (memorize this!) 
import functools

def my_decorator(func):
    @functools.wraps(func)  # Preserves function name/docstring
    def wrapper(*args, **kwargs):
        # Code to run BEFORE the function
        print("Before function call")

        # Call the original function
        result = func(*args, **kwargs)

        # Code to run AFTER the function
        print("After function call")

        return result
    return wrapper

# Usage:
@my_decorator
def greet(name):
    print(f"Hello, {name}!")

greet("Alice")
# Output:
# Before function call
# Hello, Alice!
# After function call

🔎 What is @functools.wraps(func) and why do we need it?

Every Python function has hidden attributes that describe it:

Function Metadata
def greet(name):
    """Say hello to someone."""   # ← This is a DOCSTRING (documentation)
    return f"Hello, {name}!"

# Every function has these hidden attributes:
print(greet.__name__)    # "greet"         - the function's name
print(greet.__doc__)     # "Say hello..."  - the docstring (help text)
📝 What's a docstring?

The triple-quoted string right after def is called a docstring. It's documentation that explains what the function does. You see it when you type help(greet) or hover over a function in your IDE.

The Problem: Decorators Hide the Original Function

WITHOUT @functools.wraps
def my_decorator(func):
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper

@my_decorator
def greet(name):
    """Say hello to someone."""
    return f"Hello, {name}!"

print(greet.__name__)    # "wrapper"  ← WRONG! Should be "greet"
print(greet.__doc__)     # None       ← WRONG! Lost the docstring

After decoration, greet is actually wrapper, so it has wrapper's name and no docstring!

WITH @functools.wraps - Fixed!
import functools

def my_decorator(func):
    @functools.wraps(func)   # ← Copies metadata from func to wrapper
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper

@my_decorator
def greet(name):
    """Say hello to someone."""
    return f"Hello, {name}!"

print(greet.__name__)    # "greet"         ← Correct!
print(greet.__doc__)     # "Say hello..."  ← Preserved!
💡 Why does this matter?
  • Debugging: Error messages show "wrapper" instead of actual function name
  • IDE help: Hover tooltips won't show the docstring
  • Logging: func.__name__ in logs shows wrong name
  • Testing: Some test frameworks rely on function names

Rule: Always use @functools.wraps(func) in your decorators!

Practical Example: Timing Decorator

decorators.py 
import time
import functools
import logging

logger = logging.getLogger(__name__)

def timing_decorator(func):
    """Log how long a function takes to execute."""
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        start_time = time.time()
        result = func(*args, **kwargs)
        end_time = time.time()

        duration = end_time - start_time
        logger.info(f"{func.__name__} took {duration:.4f} seconds")

        return result
    return wrapper

# Usage:
@timing_decorator
def slow_query():
    items = MenuItem.objects.select_related('category').all()
    return list(items)

# Log output: slow_query took 0.0234 seconds

Decorators with Arguments

What if you want @rate_limit(max_calls=5) instead of just @rate_limit? You need 3 levels of nesting.

# WITHOUT arguments (2 levels): @timing_decorator → timing_decorator(my_view) → wrapper def my_view(): ↑ ↑ takes function returns wrapper # WITH arguments (3 levels): @rate_limit(max_calls=5) → rate_limit(5) → decorator(my_view) → wrapper def my_view(): ↑ ↑ ↑ takes config takes function returns wrapper
The 3-Level Pattern
import functools

def repeat(times=1):                     # Level 1: Takes the CONFIG
    def decorator(func):                  # Level 2: Takes the FUNCTION
        @functools.wraps(func)
        def wrapper(*args, **kwargs):     # Level 3: Takes the ARGUMENTS
            for _ in range(times):
                result = func(*args, **kwargs)
            return result
        return wrapper                    # Level 2 returns Level 3
    return decorator                      # Level 1 returns Level 2

# Usage:
@repeat(times=3)
def say_hello():
    print("Hello!")

say_hello()
# Output: Hello! Hello! Hello!
💡 How to remember:
  • No arguments: @decorator → 2 levels (decorator → wrapper)
  • With arguments: @decorator(arg) → 3 levels (config → decorator → wrapper)

The outer function "configures" the decorator, then returns the actual decorator.

Practical Example: Rate Limiting Decorator

decorators.py - Rate Limiting 
import time
import functools
from django.core.cache import cache
from rest_framework.exceptions import Throttled

def rate_limit(max_calls=10, period=60):
    """
    Limit function calls per user.
    max_calls: Maximum calls allowed
    period: Time period in seconds
    """
    def decorator(func):
        @functools.wraps(func)
        def wrapper(request, *args, **kwargs):
            # Create unique key per user
            user_id = request.user.id or request.META.get('REMOTE_ADDR')
            cache_key = f"rate_limit:{func.__name__}:{user_id}"

            # Get current call count
            calls = cache.get(cache_key, 0)

            if calls >= max_calls:
                raise Throttled(detail=f"Rate limit exceeded. Try again later.")

            # Increment counter
            cache.set(cache_key, calls + 1, timeout=period)

            return func(request, *args, **kwargs)
        return wrapper
    return decorator

# Usage:
@api_view(['POST'])
@rate_limit(max_calls=5, period=60)  # 5 calls per minute
def create_order(request):
    # ... create order logic
    pass
Part 3

Middleware

Hooks into Django's request/response processing - runs on EVERY request!

📚 What is Middleware?

Middleware is a framework of hooks that sits between the web server and your views. Every request passes through ALL middleware (top to bottom), and every response passes back through (bottom to top). Think of it like airport security - everyone goes through it, no exceptions.

The Request/Response Pipeline

Client Request
→
Middleware 1
→
Middleware 2
→
Middleware 3
→
Your View
Your View
→
Middleware 3
→
Middleware 2
→
Middleware 1
→
Client Response
💡 Key Point: Request goes top-to-bottom, response goes bottom-to-top (like an onion - LIFO). The last middleware to touch the request is the first to touch the response.

How to Write a Middleware

The Middleware Template (memorize this!) 
class MyMiddleware:
    def __init__(self, get_response):
        """
        Called ONCE when server starts.
        get_response is the next middleware or view.
        """
        self.get_response = get_response

    def __call__(self, request):
        """Called on EVERY request."""

        # === BEFORE VIEW ===
        # Code here runs before your view
        print(f"Incoming: {request.method} {request.path}")

        # Call the next middleware/view
        response = self.get_response(request)

        # === AFTER VIEW ===
        # Code here runs after your view
        print(f"Response: {response.status_code}")

        return response
🤔 What is get_response?

get_response is a callable that Django passes to your middleware. When you call self.get_response(request), it either:

  • Calls the next middleware in the chain, OR
  • Calls the actual view (if this is the last middleware)

Think of it as "continue with the request" - everything before the call is "pre-processing", everything after is "post-processing".

📌 Where to keep custom middleware?
  • Create myapp/middleware.py in your app folder
  • Or create a shared core/middleware.py for project-wide middleware
  • Register in settings.py using the dotted path
# File: myapp/middleware.py # In settings.py, add your middleware: MIDDLEWARE = [ 'django.middleware.security.SecurityMiddleware', 'django.contrib.sessions.middleware.SessionMiddleware', ... 'myapp.middleware.MyMiddleware', ← dotted path to your class ]

Unlike decorators which only run on specific views, middleware runs on every single request that hits your Django application. This makes it perfect for cross-cutting concerns like logging, authentication, or security headers.

Middleware Class Structure

class SimpleMiddleware:
    def __init__(self, get_response):
        """
        One-time configuration when server starts.
        get_response is the next middleware or view.
        """
        self.get_response = get_response

    def __call__(self, request):
        """Called on every request."""

        # Code BEFORE view (and later middleware)
        print("Before view")

        # Call the next middleware/view
        response = self.get_response(request)

        # Code AFTER view (and earlier middleware)
        print("After view")

        return response

Optional Hook Methods

class AdvancedMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        response = self.get_response(request)
        return response

    def process_view(self, request, view_func, view_args, view_kwargs):
        """Called just before the view function."""
        print(f"About to call: {view_func.__name__}")
        return None  # Continue normally

    def process_exception(self, request, exception):
        """Called if view raises an exception."""
        print(f"Exception: {exception}")
        return None  # Let Django handle it

Practical Example: Request Logging Middleware

middleware.py 
import time
import logging

logger = logging.getLogger(__name__)

class RequestLoggingMiddleware:
    """Log every request with timing information."""

    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        # Start timer
        start_time = time.time()

        # Get response from view
        response = self.get_response(request)

        # Calculate duration
        duration = time.time() - start_time

        # Log the request
        logger.info(
            f"{request.method} {request.path} "
            f"{response.status_code} "
            f"{duration:.3f}s "
            f"user={request.user}"
        )

        return response

#     ...
#     'myapp.middleware.RequestLoggingMiddleware',
# ]

# INFO: GET /api/menu/ 200 0.045s user=john
# INFO: POST /api/orders/ 201 0.123s user=jane

Practical Example: Maintenance Mode Middleware

middleware.py - Maintenance Mode 
from django.http import JsonResponse
from django.conf import settings

class MaintenanceModeMiddleware:
    """Return 503 for all requests when in maintenance mode."""

    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        # Check if maintenance mode is enabled
        if getattr(settings, 'MAINTENANCE_MODE', False):
            # Allow admin access
            if request.path.startswith('/admin/'):
                return self.get_response(request)

            # Allow staff users
            if hasattr(request, 'user') and request.user.is_staff:
                return self.get_response(request)

            # Return 503 for everyone else
            return JsonResponse(
                {"error": "Site is under maintenance. Please try again later."},
                status=503
            )

        return self.get_response(request)

# In settings.py:
# MAINTENANCE_MODE = True  # Enable maintenance
# MAINTENANCE_MODE = False # Disable maintenance

💡 Middleware vs Decorator - When to Use Each?

Click each scenario to reveal the best choice:

Scenario Best Choice Why?
Log ALL incoming requests ✔ Middleware Applies to every request automatically
Auth check on 5 specific views ✔ Decorator @login_required on just those views
Add security headers to all responses ✔ Middleware Global, don't want to miss any
Rate limit one API endpoint ✔ Decorator @rate_limit on just that view
Site-wide maintenance mode ✔ Middleware Block all requests at once
Cache a specific page for 15 minutes ✔ Decorator @cache_page(60 * 15)

Rule of Thumb:

  • Middleware: Global behavior (every request)
  • Decorator: Selective behavior (specific views)
