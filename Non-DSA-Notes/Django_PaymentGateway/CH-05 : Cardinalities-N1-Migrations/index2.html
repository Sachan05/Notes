💡 Question 2: Chained Filters
What does this query return? 
Order.objects.filter(status='pending').filter(total_price__gte=500)
A Orders that are pending AND total >= 500
B Orders that are pending OR total >= 500
C All pending orders, then filters to >= 500 (two separate database queries)
D Error: can't chain filter() calls

Chained Filters = AND (Single Query!)

Chaining .filter() creates AND conditions. It's equivalent to:

Order.objects.filter(status='pending', total_price__gte=500)

Key insight: QuerySets are lazy - this creates ONE SQL query with both conditions. The database is only hit when you iterate!

💡 Question 3: Q Objects - OR Condition
How do you get orders that are either pending OR preparing?
A Order.objects.filter(status='pending').filter(status='preparing')
B Order.objects.filter(status='pending' or status='preparing')
C Order.objects.filter(Q(status='pending') | Q(status='preparing'))
D Order.objects.filter(status='pending' || 'preparing')

Q Objects for OR/NOT

Use Q objects when you need OR conditions or NOT:

from django.db.models import Q

# OR
Order.objects.filter(Q(status='pending') | Q(status='preparing'))

# NOT
Order.objects.filter(~Q(status='cancelled'))

# Complex
Order.objects.filter(
    (Q(status='pending') | Q(status='preparing')) &
    Q(total_price__gte=500)
)

Also valid: Order.objects.filter(status__in=['pending', 'preparing'])

💡 Question 3b: Q Objects - NOT Condition
Get all orders that are NOT cancelled. Which is correct?
A Order.objects.filter(status != 'cancelled')
B Order.objects.filter(not status='cancelled')
C Order.objects.filter(~Q(status='cancelled')) or exclude(status='cancelled')
D Order.objects.filter(Q(status!='cancelled'))

Two Ways to Negate

# Method 1: exclude()
Order.objects.exclude(status='cancelled')

# Method 2: ~Q() (NOT operator)
from django.db.models import Q
Order.objects.filter(~Q(status='cancelled'))

# ~Q is more powerful for complex conditions:
Order.objects.filter(
    Q(total_price__gte=500) &
    ~Q(status='cancelled') &
    ~Q(customer_name__icontains='test')
)
💡 Question 3c: Complex Q Objects
Get orders that are: (pending OR preparing) AND total >= 500 AND NOT from test customers.
Which query is correct?
A Order.objects.filter(status='pending' | status='preparing', total_price__gte=500)
B Order.objects.filter((Q(status='pending') | Q(status='preparing')) & Q(total_price__gte=500) & ~Q(customer_name__icontains='test'))

Important: Use parentheses () to group OR conditions when combining with AND!

# Correct: Parentheses around OR
(Q(status='pending') | Q(status='preparing')) & Q(total_price__gte=500)

# Wrong: Without parentheses, operator precedence issues
Q(status='pending') | Q(status='preparing') & Q(total_price__gte=500)
# This means: pending OR (preparing AND high_value) - not what we want!

💡 Question 4: When Does the Query Run?
At which line does Django actually hit the database? 
qs = Order.objects.filter(status='pending')  # Line 1
qs = qs.filter(total_price__gte=500)           # Line 2
qs = qs.order_by('-created_at')                 # Line 3
orders = list(qs)                               # Line 4
print(orders[0].customer_name)                  # Line 5
A Line 1 - as soon as filter() is called
B Line 3 - when ordering is applied
C Line 4 - when list() forces evaluation
D Line 5 - when accessing an attribute

QuerySets are Lazy!

Lines 1-3 just build the query - no database hit. The query executes when you:

  • Iterate: for order in qs:
  • Convert to list: list(qs)
  • Slice with step: qs[::2]
  • Call len(), bool(), list()
  • Print or repr

This is crucial for understanding N+1! The problem happens when queries are triggered inside loops.

Understanding N+1

The N+1 Problem - Conceptual

Before we see it live, let's understand what causes this performance killer.

💡 Question 5: What is N+1?
Look at this code. How many database queries will it execute if there are 100 menu items? 
items = MenuItem.objects.all()  # Get all items

for item in items:
    print(f"{item.name} - {item.category.name}")  # Access category
A 1 query (Django is smart enough to JOIN automatically)
B 2 queries (one for items, one for all categories)
C 101 queries (1 for items + 100 for each item's category)
D 100 queries (one per item)

This is the N+1 Problem!

1 query to fetch all items + N queries (one per item to fetch its category) = N+1 queries!

Django doesn't automatically fetch related objects. Each item.category access triggers a new query:

SELECT * FROM category WHERE id = 1;
SELECT * FROM category WHERE id = 2;
SELECT * FROM category WHERE id = 1;  -- duplicate!
SELECT * FROM category WHERE id = 3;
... (97 more queries)
💡 Question 6: Why is N+1 Bad?
With 1000 orders, each with 3 items, and each item from a category - how many queries for this code? 
for order in Order.objects.all():               # Query 1
    for item in order.items.all():               # ??? per order
        print(item.menu_item.category.name)      # ??? per item
A 3 queries (orders, items, categories)
B 1001 queries
C 4001 queries
D 7001 queries (1 + 1000 + 3000 + 3000)

Nested N+1 is Exponential!

Let's trace each line:

# LINE 1: for order in Order.objects.all()
Query #1: SELECT * FROM orders_order;
→ Returns 1000 order rows

# LINE 2: for item in order.items.all()
# This runs ONCE PER ORDER (1000 times!)
Query #2:    SELECT * FROM orders_orderitem WHERE order_id = 1;
Query #3:    SELECT * FROM orders_orderitem WHERE order_id = 2;
Query #4:    SELECT * FROM orders_orderitem WHERE order_id = 3;
...
Query #1001: SELECT * FROM orders_orderitem WHERE order_id = 1000;
→ 1000 queries, each returns ~3 items

# LINE 3: item.menu_item (accessing FK)
# This runs ONCE PER ITEM (3000 times!)
Query #1002: SELECT * FROM menu_menuitem WHERE id = 5;
Query #1003: SELECT * FROM menu_menuitem WHERE id = 12;
...
Query #4001: SELECT * FROM menu_menuitem WHERE id = 89;
→ 3000 queries (3 items × 1000 orders)

# LINE 3 continued: .category.name (another FK!)
# This runs ONCE PER MENU_ITEM (3000 times!)
Query #4002: SELECT * FROM menu_category WHERE id = 2;
Query #4003: SELECT * FROM menu_category WHERE id = 1;
...
Query #7001: SELECT * FROM menu_category WHERE id = 3;
→ 3000 more queries!

At 5ms per query: 7001 × 5ms = 35 seconds!

With proper optimization: 4 queries × 5ms = 20ms

Order.objects.prefetch_related(

🔗 Understanding the Chain: items__menu_item__category

The double underscore __ follows the relationship path from Order to Category:

Order  ──(items)──▶  OrderItem  ──(menu_item)──▶  MenuItem  ──(category)──▶  Category
  │                      │                        │                       │
  │                      │                        │                       │
orders_order        orders_orderitem         menu_menuitem          menu_category
table                table                     table                   table
Chain PartFrom ModelField NamePoints ToRelationship
items Order related_name='items' OrderItem Reverse FK (one order → many items)
menu_item OrderItem menu_item = ForeignKey(MenuItem) MenuItem Forward FK
category MenuItem category = ForeignKey(Category) Category Forward FK

The 4 Queries Generated:

Query 1: SELECT * FROM orders_order;
         → Get all orders

Query 2: SELECT * FROM orders_orderitem WHERE order_id IN (1,2,3,...,1000);
         → Get all order items for those orders

Query 3: SELECT * FROM menu_menuitem WHERE id IN (5,12,8,23,...);
         → Get all menu items referenced by order items

Query 4: SELECT * FROM menu_category WHERE id IN (1,2,3,4,5);
         → Get all categories referenced by menu items

💡 Key insight: Each __ adds ONE query. The chain a__b__c = 4 queries total (1 for parent + 1 for each level).

💡 Question 7: select_related
What does select_related('category') do?
A Runs a separate query to fetch all categories, then matches in Python
B Uses SQL JOIN to fetch items and categories in a single query
C Caches the category for later use
D Pre-loads categories when the server starts

select_related = SQL JOIN

MenuItem.objects.select_related('category')

-- Generates:
SELECT menu_menuitem.*, menu_category.*
FROM menu_menuitem
INNER JOIN menu_category ON menu_menuitem.category_id = menu_category.id

When to use: ForeignKey and OneToOneField (forward direction)

The "S" in select_related = Single object per row

💡 Question 8: prefetch_related
What does prefetch_related('items') do for fetching a Category's items?
A Uses SQL JOIN to fetch categories and items together
B Runs 2 queries: one for categories, one for all related items, then matches in Python
C Lazy loads items only when accessed
D Same as select_related but for reverse relationships

prefetch_related = Separate Queries + Python Matching

Category.objects.prefetch_related('items')

# Generates TWO queries:
SELECT * FROM menu_category;
SELECT * FROM menu_menuitem WHERE category_id IN (1, 2, 3, 4, 5);
⚙ How Python Matches Internally: 
# Step 1: Query 1 returns categories
categories = [
    {id: 1, name: "Appetizers"},
    {id: 2, name: "Main Course"},
    {id: 3, name: "Desserts"}
]

# Step 2: Query 2 returns ALL items for these categories
all_items = [
    {id: 1, name: "Samosa",     category_id: 1},
    {id: 2, name: "Pakora",     category_id: 1},
    {id: 3, name: "Butter Chicken", category_id: 2},
    {id: 4, name: "Dal Makhani", category_id: 2},
    {id: 5, name: "Gulab Jamun", category_id: 3}
]

# Step 3: Python builds a mapping dictionary
cache = {}
for item in all_items:
    cat_id = item.category_id
    if cat_id not in cache:
        cache[cat_id] = []
    cache[cat_id].append(item)

# cache now looks like:
{
    1: [Samosa, Pakora],           # Appetizers
    2: [Butter Chicken, Dal Makhani], # Main Course
    3: [Gulab Jamun]               # Desserts
}

# Step 4: Attach to each category object
for category in categories:
    category._prefetched_objects_cache['items'] = cache[category.id]

# Now when you access category.items.all(), Django returns from cache!
for cat in categories:
    for item in cat.items.all():  # ← No DB hit! Uses cache
        print(item.name)

When to use: ManyToMany, reverse ForeignKey (one-to-many reverse)

The "P" in prefetch_related = Plural objects (many per parent)

Reverse FK = prefetch_related

Why not select_related? Let's see what happens with INNER JOIN:

# If select_related('items') was allowed, the JOIN would be:
SELECT category.*, menuitem.*
FROM category
INNER JOIN menuitem ON menuitem.category_id = category.id;

-- Result: DUPLICATE CATEGORIES!
+-------------+----------------+-------------+------------+
| category_id | category_name  | menuitem_id | item_name  |
+-------------+----------------+-------------+------------+
|     1       | Appetizers     |     1       | Samosa     |
|     1       | Appetizers     |     2       | Pakora     |  ← Category 1 repeated!
|     1       | Appetizers     |     3       | Spring Roll|  ← Category 1 repeated!
|     2       | Main Course    |     4       | Butter Chicken |
|     2       | Main Course    |     5       | Dal Makhani|  ← Category 2 repeated!
+-------------+----------------+-------------+------------+

Problem: One category row per item = duplicated data!
Django can't "unflatten" this back to: Category → [items list]

Solution: prefetch_related (2 separate queries)

# Query 1: Get all categories
SELECT * FROM category;
+----+-------------+
| id | name        |
+----+-------------+
| 1  | Appetizers  |
| 2  | Main Course |
+----+-------------+

# Query 2: Get all items for these categories
SELECT * FROM menuitem WHERE category_id IN (1, 2);
+----+---------------+-------------+
| id | name          | category_id |
+----+---------------+-------------+
| 1  | Samosa        | 1           |
| 2  | Pakora        | 1           |
| 3  | Spring Roll   | 1           |
| 4  | Butter Chicken| 2           |
| 5  | Dal Makhani   | 2           |
+----+---------------+-------------+

# Python builds the mapping:
{
    1: [Samosa, Pakora, Spring Roll],  # Attached to Category 1
    2: [Butter Chicken, Dal Makhani]   # Attached to Category 2
}
💡 But wait - why does select_related work for OneToOne (reverse)?

Because OneToOne guarantees exactly ONE related object!

# Example: Order has ONE Invoice (FK in Invoice)
class OrderInvoice(models.Model):
    order = models.OneToOneField(Order, related_name='invoice')

# This works! JOIN returns exactly 1 row per order
Order.objects.select_related('invoice')  ✓

# Generated SQL:
SELECT order.*, invoice.*
FROM order
LEFT JOIN invoice ON invoice.order_id = order.id;

+----------+--------+------------+---------+
| order_id | status | invoice_id | amount  |
+----------+--------+------------+---------+
|    1     | done   |     1      | 500.00  |  ← One row per order!
|    2     | done   |     2      | 750.00  |
+----------+--------+------------+---------+

🧠 Deep Dive: Why Are Duplicates Actually Bad?

The SQL JOIN works fine at the database level. The problem is on the Python/ORM side.

Django's select_related design: One row = One object

# When Django processes the JOIN result:
Row 1: Category 1, Samosa     → creates Category object #1
Row 2: Category 1, Pakora     → create another Category? Or same one?
Row 3: Category 1, Spring Roll → again?

# Django would need to:
# 1. Realize rows 1,2,3 belong to the SAME Category
# 2. Group items into a list: Category → [Samosa, Pakora, Spring Roll]
# This "unflattening" is complex - Django chose to keep select_related simple.

# Also: With 1000 items per category, you transfer the category
# name and description 1000 times over the wire. Wasted bandwidth!

✅ Why Forward FK Doesn't Have This Problem

Because duplication goes the "safe" direction:

# Forward FK: MenuItem.objects.select_related('category')
SELECT menuitem.*, category.*
FROM menuitem
INNER JOIN category ON menuitem.category_id = category.id;

+-------------+-----------+-------------+---------------+
| menuitem_id | item_name | category_id | category_name |
+-------------+-----------+-------------+---------------+
|     1       | Samosa    |     1       | Appetizers    |
|     2       | Pakora    |     1       | Appetizers    |  ← "Appetizers" repeated, but OK!
|     3       | Dal       |     2       | Main Course   |
+-------------+-----------+-------------+---------------+

# Each row is ONE UNIQUE MenuItem.
# Yes, "Appetizers" appears twice, but Django doesn't care —
# it reads row by row to build MenuItem objects.
# Each row = one MenuItem with its category attached. Done!
🎯 The Golden Rule

select_related gives you one row per object you're querying for.

MenuItem → Category  (many to one)
    ↑           ↑
  many        one

→ Going TOWARD the "one" side (MenuItem → Category): select_related ✓
  Every MenuItem has exactly ONE Category. JOIN adds columns, not rows.

→ Going TOWARD the "many" side (Category → MenuItems): select_related ✗
  Every Category has MULTIPLE MenuItems. JOIN multiplies rows.
  Use prefetch_related instead!

The "one" side can tag along via JOIN. The "many" side cannot.

RelationshipRows per ParentUse
FK forward (item.category)1select_related
OneToOne (any direction)1select_related
Reverse FK (category.items)Manyprefetch_related
ManyToManyManyprefetch_related
💡 Question 8c: prefetch_related - ManyToMany
Scenario: MenuItem has many Tags (ManyToManyField).
You want to get all menu items with their tags. What do you use? 
# Model setup:
class MenuItem(models.Model):
    tags = models.ManyToManyField(Tag, related_name='menu_items')

# Query:
items = MenuItem.objects.???
for item in items:
    for tag in item.tags.all():  # M2M access
        print(tag.name)
A select_related('tags')
B prefetch_related('tags')
C select_related('tags__name')
D prefetch_related('tag_set')

ManyToMany = Always prefetch_related

Why? M2M relationships go through a junction table. Can't JOIN MenuItem directly to Tag!

# ManyToMany: MenuItem ↔ Tag (via junction table)
MenuItem.objects.prefetch_related('tags')

# Query 1: Get all menu items
SELECT * FROM menu_menuitem;

# Query 2: Get tags for these items
# WHY JOIN HERE? Because we need the menuitem_id from junction table!
SELECT
    menu_tag.*,                           -- tag data
    menu_menuitem_tags.menuitem_id        -- which item owns this tag
FROM menu_tag
INNER JOIN menu_menuitem_tags             -- junction table
    ON menu_tag.id = menu_menuitem_tags.tag_id
WHERE menu_menuitem_tags.menuitem_id IN (1, 2, 3, 4, 5);

-- Result includes menuitem_id so Python knows which tag belongs to which item:
| tag_id | tag_name  | menuitem_id |
|--------|-----------|-------------|
|   1    | "Spicy"   |      1      |  ← Tag 1 belongs to MenuItem 1
|   2    | "Vegan"   |      1      |  ← Tag 2 also belongs to MenuItem 1
|   1    | "Spicy"   |      3      |  ← Tag 1 also belongs to MenuItem 3
|   3    | "Popular" |      2      |

# Python builds: {1: [Spicy, Vegan], 2: [Popular], 3: [Spicy], ...}
💡 Why can't we just SELECT * FROM both tables separately?

Because we need to know which tags belong to which items. That mapping is stored in the junction table (menu_menuitem_tags). Without JOINing it, we'd just have a list of tags with no idea which item they belong to!

Rule: ManyToMany in either direction = prefetch_related

BAD: N+1 Problem 
from menu.models import MenuItem

reset_queries()

# This looks innocent...
items = MenuItem.objects.all()
for item in items:
    _ = item.category.name  # Access category (triggers query!)
    # print(f"{item.name} - {item.category.name}")

print(f"Total items: {len(items)}")
show_queries()  # Watch the query count explode! 1 + N queries

✓ Demo 2: The Fix with select_related

GOOD: Using select_related 
reset_queries()

# Add select_related - single JOIN query!
items = MenuItem.objects.select_related('category')
for item in items:
    _ = item.category.name  # No extra query - already fetched!
    # print(f"{item.name} - {item.category.name}")

print(f"Total items: {len(items)}")
show_queries()  # Just 1 query!

# See the actual JOIN query:
print(connection.queries[0]['sql'])
⚠ Gotcha: Ordering by FK Causes Implicit JOIN!

If your model has ordering = ['category', 'name'] in Meta, Django will JOIN even on .all()!

# BAD: Ordering by FK field
class MenuItem(models.Model):
    category = models.ForeignKey(Category, ...)

    class Meta:
        ordering = ['category', 'name']  ← Causes JOIN on every query!

# MenuItem.objects.all() generates:
SELECT ... FROM menu_menuitem
INNER JOIN menu_category ON ...  ← Unexpected JOIN!
ORDER BY menu_category.name, menu_menuitem.name

# GOOD: Order by local fields only
class Meta:
    ordering = ['name']  ← No implicit JOIN

💥 Demo 3: Nested N+1 with Orders

BAD: Nested N+1 
from orders.models import Order

reset_queries()

# Nested loops = exponential queries!
orders = Order.objects.all()[:5]  # Just 5 orders
total_items = 0
for order in orders:
    for item in order.items.all():
        _ = item.menu_item.name  # Triggers query for each item!
        total_items += 1
        # print(f"  - {item.quantity}x {item.menu_item.name}")

print(f"Orders: 5, Total items: {total_items}")
show_queries()  # Way too many!

✓ Demo 4: The Fix with prefetch_related (Chained)

GOOD: prefetch_related (chained) 
reset_queries()

# Prefetch items AND their menu_items (chained)
orders = Order.objects.prefetch_related('items__menu_item')[:5]
total_items = 0
for order in orders:
    for item in order.items.all():
        _ = item.menu_item.name  # No extra query - already prefetched!
        total_items += 1
        # print(f"  - {item.quantity}x {item.menu_item.name}")

print(f"Orders: 5, Total items: {total_items}")
show_queries()  # Just 3 queries!

# See the prefetch queries:
for q in connection.queries:
    print(q['sql'][:80])

Without --empty: Django scans models for changes and auto-generates migration.

# Create empty migration
python manage.py makemigrations orders --empty -n populate_totals

# Edit the migration file:
def calculate_totals(apps, schema_editor):
    Order = apps.get_model('orders', 'Order')
    for order in Order.objects.filter(total_price__isnull=True):
        order.total_price = sum(
            item.quantity * item.unit_price
            for item in order.items.all()
        )
        order.save()

class Migration(migrations.Migration):
    operations = [
        migrations.RunPython(calculate_totals, reverse_totals),
    ]
💡 Why Data Migration? What's Wrong with Alternatives?
Approach Problem
Option A: Management Command
  • Must remember to run it on every server (dev, staging, prod)
  • No record that it was run - might run twice or forget
  • Not versioned in git with migrations
  • New developer sets up DB → data is missing!
Option C: Edit Schema Migration
  • Mixes schema + data concerns (bad practice)
  • If schema migration fails, data code also fails
  • Can't rollback schema without rolling back data
Option D: Django Shell
  • Manual process - human error prone
  • No audit trail - "Did I run this on prod?"
  • Not repeatable - new DB setup is manual
  • Not in version control
Option B: Data Migration ✓
  • ✓ Versioned - lives in git with code
  • ✓ Automatic - runs on migrate
  • ✓ Tracked - Django knows it ran
  • ✓ Reversible - can undo with reverse_code
  • ✓ Repeatable - new DB gets same data

🚨 Why Do We Need Exception Handling?

In a restaurant API, many things can go wrong:

  • User requests a menu item that doesn't exist
  • User tries to order when restaurant is closed
  • Payment gateway times out
  • Database connection fails
  • Invalid data in request body

Without exception handling: App crashes, user sees ugly error, no useful info

With exception handling: Graceful error response, proper HTTP status, helpful message

DoesNotExist Exception

.get() raises Model.DoesNotExist when no match is found. Without handling, this becomes HTTP 500 (Internal Server Error).

# The exception:
menu.models.MenuItem.DoesNotExist: MenuItem matching query does not exist.

# User sees: HTTP 500 Internal Server Error (bad!)
💡 Question 15: Basic Try-Except
Which is the correct way to handle the DoesNotExist exception?
A try: ... except: ... (catch everything)
B try: ... except MenuItem.DoesNotExist: ...
C if MenuItem.exists(id): ...

Specific Exception Handling

def get_menu_item(request, item_id):
    try:
        item = MenuItem.objects.get(id=item_id)
        return JsonResponse({'name': item.name})
    except MenuItem.DoesNotExist:
        return JsonResponse(
            {'error': 'Menu item not found'},
            status=404
        )

Rule: Always catch specific exceptions, not bare except:

Handle All Expected Exceptions

def get_order_item(request, order_id, item_index):
    try:
        order = Order.objects.get(id=order_id)
        item = list(order.items.all())[item_index]
        return JsonResponse({'item': item.menu_item.name})
    except Order.DoesNotExist:
        return JsonResponse({'error': 'Order not found'}, status=404)
    except IndexError:
        return JsonResponse({'error': 'Item index out of range'}, status=400)

The finally Block - Guaranteed Cleanup

finally ALWAYS runs - use it for cleanup code that must execute regardless of success or failure.

try:
    item = MenuItem.objects.get(id=99999)
    print(f"Found: {item.name}")
except MenuItem.DoesNotExist:
    print("Item not found!")
finally:
    print("This ALWAYS runs - cleanup here")

# Common use cases for finally:
# - Closing file handles
# - Releasing database connections
# - Releasing locks
# - Closing external API connections
# - Logging completion status

# Real-world example:
f = None
try:
    f = open('data.txt', 'r')
    data = f.read()
except FileNotFoundError:
    print("File not found")
finally:
    if f:
        f.close()  # Always close, even if error!

💡 Note: In Django, connection pooling handles DB cleanup automatically. But finally is crucial for files, locks, and external resources.
