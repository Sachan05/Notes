📚 Model Inheritance

💡 Question 1: Shared Fields Problem
You have 5 models (User, Order, Product, Category, Review) and each needs created_at and updated_at fields. What's the best approach?
A Copy-paste the fields into each model manually
B Create an Abstract Base Model with class Meta: abstract = True
C Use Multi-Table Inheritance from a TimestampedModel
D Create a Proxy Model for timestamps

Abstract Base Model is Perfect Here!

Why Abstract? It shares fields WITHOUT creating a database table. The fields are copied into each child model's table.

Why not Multi-Table? That would create an actual TimestampedModel table and require JOINs for every query - unnecessary overhead!

Why not Proxy? Proxy models don't add fields - they only change Python behavior on existing models.

💡 Question 2: Payment Processing System
A payment system has different payment methods. Credit cards need card_number, cvv, expiry_date. UPI payments need upi_id, bank_name. Both share amount, timestamp, status. You need to query "all payments" regardless of type. Which inheritance type?
A Abstract Base Model
B Multi-Table Inheritance (CreditCardPayment and UPIPayment inherit from Payment)
C Proxy Model
D Single table with nullable columns for all fields

Multi-Table Inheritance!

Why? You need to query "all payments" (parent) AND have type-specific fields (children). Multi-table creates:

  • payments_payment table (shared: amount, timestamp, status)
  • payments_creditcardpayment table (card_number, cvv, expiry + FK to payment)
  • payments_upipayment table (upi_id, bank_name + FK to payment)

Trade-off: Queries on specific types require JOINs, but you can query all payments via Payment.objects.all()!

Why not Abstract? Abstract would create separate tables with no common parent - you couldn't query "all payments"!

💡 Question 3: Vehicle Fleet Management
A fleet management system has a Vehicle model. You want to display "Available Vehicles" and "Under Maintenance" as separate lists in Django Admin. Both use the same database table with a status field. Which approach?
A Create two separate tables (AvailableVehicle, MaintenanceVehicle)
B Use Multi-Table Inheritance from Vehicle
C Proxy Models with custom Managers that filter by status
D Abstract Base Model

Proxy Models are Perfect!

Key insight: Same table, different Python behavior. Proxy models:

  • Don't create new database tables
  • Can have different default managers (filters by status)
  • Can have different Meta options (ordering)
  • Show as separate sections in Django Admin

Example:

class AvailableVehicle(Vehicle):
    class Meta:
        proxy = True
    objects = AvailableVehicleManager()  # filters status='available'

🔑 IDs & Security

💡 Question 4: Public API IDs
Your API exposes order URLs like /api/orders/1/, /api/orders/2/. A competitor checks /api/orders/50000/ and gets data. What information did you leak?
A Nothing - they just got one order
B Your business has approximately 50,000 orders (business intelligence)
C Your database password
D Nothing if authentication is required

Sequential IDs Leak Business Intelligence!

This is the German Tank Problem from WWII. Sequential IDs reveal:

  • Total number of records (business size)
  • Growth rate (check IDs over time)
  • Valid IDs to enumerate and scrape

Solution: Use UUIDs for public-facing IDs!

💡 Question 5: Best of Both Worlds
UUIDs are secure but slower for database JOINs. Integer IDs are fast but insecure for APIs. What's the recommended approach?
A Always use UUID as primary key
B Always use integer auto-increment
C Integer PK for internal JOINs + UUID public_id for API exposure
D Use hashed IDs that can be reversed

Dual ID Approach!

Keep the auto-increment id as primary key for fast JOINs. Add a public_id UUID field for API exposure:

public_id = models.UUIDField(default=uuid.uuid4, unique=True, db_index=True)

Use lookup_field = 'public_id' in your ViewSet.

👤 OneToOneField — One-to-One

When to use: Each record on side A has exactly ONE record on side B, and vice versa.
Examples: User ↔ Profile, Order ↔ Invoice, Employee ↔ EmployeeDetails
orders/models.py - OneToOne Example 
class Order(models.Model):
    customer_name = models.CharField(max_length=100)
    total_price = models.DecimalField(max_digits=10, decimal_places=2)


class OrderInvoice(models.Model):
    # Each Order has exactly ONE Invoice
    order = models.OneToOneField(
        Order,
        on_delete=models.CASCADE,
        related_name='invoice'  # order.invoice (not order.orderinvoice!)
    )
    invoice_number = models.CharField(max_length=20, unique=True)
    generated_at = models.DateTimeField(auto_now_add=True)


# Usage:
order.invoice              # Direct access to single object
invoice.order              # Reverse also works

💡 Why is OneToOneField in OrderInvoice, not in Order?

The "Dependent Model" Principle: The OneToOneField goes in the model that depends on or extends the other.

  • Order can exist without Invoice (order placed, invoice not yet generated)
  • Invoice CANNOT exist without Order (invoice is always for an order)
  • Invoice is the "dependent" or "child" model → OneToOneField goes here

Same pattern: Profile depends on User → OneToOneField in Profile

⚠ What if we don't specify related_name?

Django creates a default reverse accessor using lowercase model name:

# Without related_name:
order.orderinvoice         # Default: lowercase model name

# With related_name='invoice':
order.invoice              # Much cleaner!

For ForeignKey (One-to-Many): Default is modelname_set (e.g., category.menuitem_set)

For OneToOneField: Default is just modelname (e.g., order.orderinvoice) - no _set because it's singular!

🔗 ForeignKey — One-to-Many

⚠ Golden Rule: ForeignKey goes on the "MANY" side!
Think: "Each [Child] belongs to ONE [Parent]" → FK in Child model
📚 Scenario: Category & MenuItems

Each Category has MANY MenuItems. Each MenuItem belongs to ONE Category.

Question: Where does ForeignKey go?

Answer: In MenuItem! Because "Each MenuItem belongs to ONE Category"

menu/models.py - ForeignKey Example 
class Category(models.Model):
    name = models.CharField(max_length=50)
    # NO ForeignKey here! Category is the "ONE" side.


class MenuItem(models.Model):
    name = models.CharField(max_length=100)
    price = models.DecimalField(max_digits=6, decimal_places=2)

    # ForeignKey on "MANY" side - each item belongs to ONE category
    category = models.ForeignKey(
        Category,
        on_delete=models.CASCADE,     # Delete items when category deleted
        related_name='items'          # Enables: category.items.all()
    )


# Usage:
item.category              # Forward: Get the ONE category
item.category.name         # Access category's fields

category.items.all()       # Reverse: Get ALL items in category
category.items.count()     # Count items
on_delete What happens when parent is deleted?
CASCADE Delete all children (most common)
PROTECT Prevent deletion if children exist
SET_NULL Set FK to NULL (requires null=True)
SET_DEFAULT Set FK to a default value

👥 ManyToManyField — Many-to-Many

When to use: Records on both sides can have multiple related records.
Examples: Students ↔ Courses, Orders ↔ Products, Articles ↔ Tags

Option 1: Simple ManyToMany (Django creates the table)

🍽 Restaurant Scenario: MenuItem Tags

MenuItems can have multiple Tags (Spicy, Vegan, Gluten-Free, Chef's Special). Each Tag can be on multiple MenuItems.

No extra data needed on the relationship → Simple ManyToMany!

menu/models.py - Simple ManyToMany for Tags 
class Tag(models.Model):
    name = models.CharField(max_length=50, unique=True)
    # Examples: "Spicy", "Vegan", "Gluten-Free", "Chef's Special"


class MenuItem(models.Model):
    name = models.CharField(max_length=100)
    price = models.DecimalField(max_digits=6, decimal_places=2)

    # ManyToMany - Django creates menu_menuitem_tags table automatically!
    tags = models.ManyToManyField(
        Tag,
        related_name='menu_items',
        blank=True  # Items can have no tags
    )


# Django automatically creates a junction table:
# menu_menuitem_tags (id, menuitem_id, tag_id)

# Usage:
item.tags.all()                  # All tags for this item
item.tags.add(spicy_tag)         # Add "Spicy" tag
item.tags.remove(spicy_tag)      # Remove tag

spicy_tag.menu_items.all()       # All spicy items
MenuItem.objects.filter(tags__name='Vegan')  # Filter by tag
Django auto-creates this junction table: ┌───────────────────────────────────────────┐ │ menu_menuitem_tags │ ├───────────────────────────────────────────┤ │ id │ menuitem_id │ tag_id │ ├────┼─────────────┼─────────────────────────┤ │ 1 │ 1 │ 1 │ (Butter Chicken + Spicy) │ 2 │ 1 │ 4 │ (Butter Chicken + Chef's Special) │ 3 │ 2 │ 2 │ (Dal Makhani + Vegan) │ 4 │ 3 │ 1 │ (Paneer Tikka + Spicy) │ 5 │ 3 │ 2 │ (Paneer Tikka + Vegan) └────┴─────────────┴─────────┴───────────────┘ No extra fields needed - just linking items to tags!

Option 2: Through Model (When you need extra fields)

Use Through Model when: You need to store additional data about the relationship.
Examples: Enrollment date, grade, quantity, price at time of order
orders/models.py - Through Model for Order Items 
class Order(models.Model):
    customer_name = models.CharField(max_length=100)

    # ManyToMany VIA the through model
    menu_items = models.ManyToManyField(
        'menu.MenuItem',
        through='OrderItem',  # Use our custom junction table
        related_name='orders'
    )


class OrderItem(models.Model):
    """Through model - stores extra relationship data."""

    order = models.ForeignKey(
        Order, on_delete=models.CASCADE, related_name='items'
    )
    menu_item = models.ForeignKey(
        'menu.MenuItem', on_delete=models.PROTECT
    )

    # Extra fields on the relationship!
    quantity = models.PositiveIntegerField(default=1)
    unit_price = models.DecimalField(max_digits=6, decimal_places=2)

    class Meta:
        unique_together = ['order', 'menu_item']  # No duplicates


# Usage:
order.items.all()              # Get OrderItem objects (with quantity!)
order.menu_items.all()         # Get MenuItem objects directly

for item in order.items.all():
    print(f"{item.quantity}x {item.menu_item.name}")
Through Model gives you control over the junction table: ┌─────────────────────────────────────────────────────────────┐ │ orders_orderitem │ ├─────────────────────────────────────────────────────────────┤ │ id │ order_id │ menu_item_id │ quantity │ unit_price │ ├────┼──────────┼──────────────┼──────────┼───────────────────┤ │ 1 │ 1 │ 5 │ 2 │ 329.00 │ (2x Butter Chicken) │ 2 │ 1 │ 8 │ 3 │ 49.00 │ (3x Naan) │ 3 │ 2 │ 5 │ 1 │ 329.00 │ (1x Butter Chicken) └────┴──────────┴──────────────┴──────────┴───────────────────┘ You can store quantity, price at time of order, notes, etc!

Performance

The N+1 Problem

The most common Django performance issue. Let's understand it step by step.

⚠ What is N+1? When you access related objects in a loop, Django runs a separate query for EACH iteration. 1 query becomes 1 + N queries!

Simple Example: Categories & Items

The Problem - Looks innocent! 
# Get all menu items and print their category name
items = MenuItem.objects.all()  # Query 1: Get all items

for item in items:
    print(item.category.name)   # Query 2, 3, 4... per item!

What Actually Happens

1️⃣
Query 1
Get all items
→
🔄
Query 2
Get category for item 1
→
🔄
Query 3
Get category for item 2
→
💥
... N more
One per item!
With 100 menu items: 1 query (get items) + 100 queries (get category for each) = 101 queries! At 5ms per query: 101 × 5ms = 505ms Should be: 2 × 5ms = 10ms

Nested N+1: Orders with Items

Even Worse - Nested loops! 
orders = Order.objects.all()  # Query 1: Get all orders

for order in orders:
    for item in order.items.all():    # Query per ORDER (N queries)
        print(item.menu_item.name)    # Query per ITEM (M queries)

# Total: 1 + N + M queries!
# With 100 orders, 3 items each: 1 + 100 + 300 = 401 queries!

💡 How to Detect N+1

  • Django Debug Toolbar - Shows all queries in SQL panel
  • Query logging - Enable in settings
  • Pattern - Many similar queries with different IDs

Duplicate query! (0.001) SELECT * FROM menu_category WHERE id = 3; (0.001) SELECT * FROM menu_category WHERE id = 2; ... (continues for each item) Red flag: Multiple similar queries with different/same IDs = N+1 problem!

Optimization

Solving N+1: select_related vs prefetch_related

Two tools to eliminate N+1. Understanding the difference is crucial!

🔗

select_related()

How: SQL JOIN
For: ForeignKey, OneToOne
Direction: Forward only
Result: Single query

📦

prefetch_related()

How: Separate queries + Python
For: ManyToMany, Reverse FK
Direction: Any direction
Result: 2+ queries

select_related — Uses SQL JOIN

Fixing N+1 with select_related 
# ❌ BEFORE: N+1 problem (101 queries)
items = MenuItem.objects.all()
for item in items:
    print(item.category.name)


# ✅ AFTER: Single query with JOIN
items = MenuItem.objects.select_related('category')
for item in items:
    print(item.category.name)  # No extra query!
select_related generates ONE query with JOIN: SELECT menu_menuitem.*, menu_category.* FROM menu_menuitem INNER JOIN menu_category ON (menu_menuitem.category_id = menu_category.id) All data fetched in single round-trip to database!

📈 What the JOIN Actually Returns

The database returns ONE table with ALL columns from both tables: ┌─────────────────────────────────────────────────────────────────────────────────┐ │ RESULT OF JOIN QUERY │ ├────────────────────────────────────────┬────────────────────────────────────────┤ │ menu_menuitem columns │ menu_category columns │ ├────┬──────────────────┬────────┬───────┼────┬──────────────┬────────────────────┤ │ id │ name │ price │cat_id │ id │ name │ description │ ├────┼──────────────────┼────────┼───────┼────┼──────────────┼────────────────────┤ │ 1 │ Butter Chicken │ 329.00 │ 1 │ 1 │ Main Course │ Hearty dishes │ │ 2 │ Dal Makhani │ 199.00 │ 1 │ 1 │ Main Course │ Hearty dishes │ │ 3 │ Paneer Tikka │ 249.00 │ 2 │ 2 │ Starters │ Light appetizers │ │ 4 │ Gulab Jamun │ 99.00 │ 3 │ 3 │ Desserts │ Sweet endings │ └────┴──────────────────┴────────┴───────┴────┴──────────────┴────────────────────┘ Django then automatically creates Python objects from this: - Creates MenuItem objects with item.category already populated - No additional queries needed when you access item.category!

prefetch_related — Separate Queries + Python Matching

Fixing N+1 with prefetch_related 
# ❌ BEFORE: N+1 problem (101 queries)
categories = Category.objects.all()
for cat in categories:
    print(cat.items.all())  # Query per category!


# ✅ AFTER: Just 2 queries
categories = Category.objects.prefetch_related('items')
for cat in categories:
    print(cat.items.all())  # Uses prefetched data!
prefetch_related generates 2 queries: Query 1: SELECT * FROM menu_category Query 2: SELECT * FROM menu_menuitem WHERE category_id IN (1, 2, 3, 4, 5) Python then matches items to categories in memory. Still much better than 101 queries!

Key Difference: Why Two Methods?

💡 Why can't we always use SQL JOIN?

Problem with JOINing reverse relationships:

If Category has 100 items, a JOIN would return 100 rows per category, duplicating category data. With 10 categories × 100 items = 1000 rows with lots of duplicate data!

Solution: prefetch_related does 2 small queries and matches in Python - much more efficient for "many" sides.

💡 Quick Review: Chained Filters
What does this query return? 
Order.objects.filter(status='pending').filter(total_price__gte=500)
A Orders that are pending AND total >= 500
B Orders that are pending OR total >= 500
C All pending orders, then filters to >= 500 (two database queries)

Chained Filters = AND

Chaining .filter() calls creates AND conditions. It's the same as:

Order.objects.filter(status='pending', total_price__gte=500)

Important: QuerySets are lazy - this creates ONE SQL query with both conditions!

💡 Q Objects: OR, AND, NOT

When you need complex conditions beyond simple AND:

Q Object Examples 
from django.db.models import Q

# OR: Orders that are pending OR confirmed
Order.objects.filter(
    Q(status='pending') | Q(status='confirmed')
)
# SQL: WHERE status = 'pending' OR status = 'confirmed'


# NOT: Orders that are NOT cancelled
Order.objects.filter(
    ~Q(status='cancelled')
)
# SQL: WHERE NOT status = 'cancelled'


# Complex: (pending OR confirmed) AND high value AND NOT "test"
Order.objects.filter(
    (Q(status='pending') | Q(status='confirmed')) &
    Q(total_price__gte=1000) &
    ~Q(customer_name__icontains='test')
)
Q Object Operators:
| = OR
& = AND
~ = NOT

The Repetition Problem

Same queries scattered everywhere...
# In views.py
Order.objects.filter(status='pending').prefetch_related('items')

# In admin.py
Order.objects.filter(status='pending')  # Forgot prefetch!

# In tasks.py
Order.objects.filter(status='pending').prefetch_related('items')

# Problems:
# 1. DRY violation - same filter repeated
# 2. Easy to forget optimization
# 3. If "pending" logic changes, update everywhere

The Solution: Custom QuerySet

orders/models.py 
class OrderQuerySet(models.QuerySet):

    def with_items(self):
        """Prefetch items and menu_items - solves N+1!"""
        return self.prefetch_related('items__menu_item')

    def pending(self):
        return self.filter(status='pending')

    def active(self):
        return self.exclude(status__in=['cancelled', 'completed'])

    def today(self):
        return self.filter(created_at__date=timezone.now().date())


class Order(models.Model):
    # ... fields ...
    objects = OrderQuerySet.as_manager()


# Now use clean, chainable methods:
Order.objects.pending()                   # Pending orders
Order.objects.pending().with_items()      # With optimization!
Order.objects.active().today()            # Chain multiple
✓ Benefits:
• Query logic defined once
• Optimizations built into methods like with_items()
• Methods are chainable
• Self-documenting business rules

🔥 Fake Migrations

When to use --fake: When the database already has the changes, but Django doesn't know about it. This marks migrations as applied WITHOUT actually running them.
📚 Common Scenario: Production DB already modified

You manually added a column to production DB. Now Django thinks migration isn't applied, but running it would fail because column already exists!

Fake migration commands
# Mark a specific migration as applied (without running it)
python manage.py migrate orders 0003 --fake

# Mark ALL migrations as applied (fresh DB that matches models)
python manage.py migrate --fake

# Mark initial migration as applied (starting fresh)
python manage.py migrate orders --fake-initial

# Reset: Mark all as unapplied, then fake to current state
python manage.py migrate orders zero --fake
python manage.py migrate orders --fake
⚠ Warning: Only use --fake when you're CERTAIN the database schema matches what the migration would do. Using it incorrectly can cause schema mismatches!

Data Migrations

Create empty migration for data 
# Create empty migration file
python manage.py makemigrations orders --empty -n populate_totals
orders/migrations/0003_populate_totals.py
from django.db import migrations


def calculate_totals(apps, schema_editor):
    # Use apps.get_model - NOT direct import!
    Order = apps.get_model('orders', 'Order')

    for order in Order.objects.filter(total_price__isnull=True):
        order.total_price = sum(
            item.quantity * item.unit_price
            for item in order.items.all()
        )
        order.save()


def reverse_totals(apps, schema_editor):
    Order = apps.get_model('orders', 'Order')
    Order.objects.update(total_price=None)


class Migration(migrations.Migration):
    dependencies = [('orders', '0002_orderitem')]

    operations = [
        migrations.RunPython(calculate_totals, reverse_totals),
    ]
💡 Always provide reverse function! This allows rolling back the migration if needed.

Summary

What We Learned

👤

OneToOne (1:1)

Each record has exactly one related record.

🔗

ForeignKey (1:N)

FK goes on the MANY side!

👥

ManyToMany (M:N)

Use through model for extra fields.

💥

N+1 Problem

Loop + related access = disaster!

⚡

select_related

SQL JOIN for FK/O2O forward.

📦

prefetch_related

Separate queries for M2M/reverse.
