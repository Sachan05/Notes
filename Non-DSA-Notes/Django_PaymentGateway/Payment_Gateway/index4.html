Before We Begin

Two quick questions from last class to warm up.

Q1: Your API supports searching menu items. Which HTTP method should you use for a simple text search like ?search=chicken?

A) POST — search modifies server state
B) GET — search is just reading/filtering data
C) PUT — search updates search history
D) Either works, doesn't matter
Reveal Answer
B) GET. Search is a read operation — it doesn't modify data. GET requests are cacheable, bookmarkable, and shareable. POST is only needed when the search query is too complex for a URL (like Elasticsearch's nested boolean queries).

Q2: Elasticsearch uses POST for its _search endpoint instead of GET. Why?

A) POST is more secure
B) POST is faster than GET
C) The query body is complex JSON that doesn't fit in a URL (~2000 char limit)
D) Elasticsearch doesn't support GET
Reveal Answer
C. ES queries can have deeply nested boolean logic, aggregations, geo-filters, and scoring rules — hundreds of lines of JSON. This doesn't fit in a URL. The HTTP spec allows GET with a body, but most tools don't support it well, so POST is the practical choice.

Python dict / Django LocMemCache.
Simplest: just a dictionary in your app's RAM.

Why Redis Won

Redis won because it does caching AND more. Sessions, rate limiting, leaderboards, pub/sub, job queues — all in one tool. That's why 90%+ of companies choose Redis over Memcached.

The Tool

Redis — Remote Dictionary Server

An in-memory key-value store. Think of it as a giant Python dictionary that lives outside your app.

A dictionary in RAM, shared across servers, that auto-deletes old entries. 100,000+ operations/second.

Redis Data Types — Syntax + What's Stored

Redis isn't just key-value strings. It has 5 core data types, each stored differently in memory. Here's how each one works:

1. String — The simplest. One key = one value.

COMMANDS

SET menu:items '[{"id":1,"name":"Butter Chicken"}]'
GET menu:items
SET visitor_count 0
INCR visitor_count    # → 1
INCR visitor_count    # → 2

WHAT'S STORED IN REDIS

key               → value
"menu:items"      → "[{\"id\":1,...}]"
"visitor_count"   → "2"

Think of it as a Python variable: menu_items = "[...]"

Use for: Caching API responses, counters, simple flags, OTP storage

2. Hash — A dictionary inside a key. Like a Python dict.

COMMANDS

HSET user:42 name "Prateek" email "p@x.com" age 28
HGET user:42 name         # → "Prateek"
HGETALL user:42           # → all fields
HINCRBY user:42 age 1    # → 29

WHAT'S STORED IN REDIS

"user:42" → {
    "name":  "Prateek"
    "email": "p@x.com"
    "age":   "29"
}

Like user_42 = {"name": "Prateek", "email": "p@x.com"}

Use for: Cache user profiles, product details, session data — read/update individual fields without fetching the whole object

3. List — An ordered sequence. Like a Python list. Push/pop from both ends.

COMMANDS

LPUSH recent:orders "ord_003"  # push left
LPUSH recent:orders "ord_004"
LPUSH recent:orders "ord_005"
LRANGE recent:orders 0 -1     # all items
LLEN recent:orders              # → 3
RPOP recent:orders              # remove oldest

WHAT'S STORED IN REDIS

"recent:orders" → [
    "ord_005"    ← newest (left)
    "ord_004"
    "ord_003"    ← oldest (right)
]

Like recent_orders = ["ord_005", "ord_004", "ord_003"]

Use for: Recent activity feeds, notification queues, "last 10 orders" widgets, message queues (Celery uses this!)

4. Set — Unordered, unique values. Like a Python set. No duplicates.

COMMANDS

SADD online:users "user_1" "user_2" "user_3"
SADD online:users "user_1"            # ignored! already exists
SMEMBERS online:users               # → all members
SCARD online:users                  # → 3 (count)
SISMEMBER online:users "user_1"    # → 1 (true)
SREM online:users "user_2"         # remove

WHAT'S STORED IN REDIS

"online:users" → {
    "user_1"
    "user_2"    ← no order
    "user_3"    ← no duplicates
}

Like online_users = {"user_1", "user_2", "user_3"}

Use for: Track unique visitors, "who's online" features, tags, unique IP tracking, "has this user already voted?"

5. Sorted Set — Like a Set, but each member has a score. Auto-sorted by score.

COMMANDS

ZADD leaderboard 100 "Prateek"
ZADD leaderboard 250 "Ananya"
ZADD leaderboard 80  "Rahul"
ZRANGE leaderboard 0 -1 WITHSCORES  # asc
ZREVRANGE leaderboard 0 2            # top 3
ZINCRBY leaderboard 50 "Rahul"       # → 130
ZRANK leaderboard "Prateek"          # → 1 (rank)

WHAT'S STORED IN REDIS (auto-sorted by score)

"leaderboard" → {
    80:  "Rahul"      ← lowest score
    100: "Prateek"
    250: "Ananya"     ← highest score
}
After ZINCRBY Rahul 50:
{
    100: "Prateek"
    130: "Rahul"      ← moved up!
    250: "Ananya"
}

No Python equivalent — closest is a dict sorted by values, but Redis does it automatically on every insert.

Use for: Leaderboards, "top trending dishes", priority queues, rate limiting (sorted by timestamp)

Quick Reference

String = single value (like a Python variable)  |  Hash = dict inside a key  |  List = ordered, allows duplicates  |  Set = unordered, no duplicates  |  Sorted Set = scored + auto-sorted

For API caching, you'll use Strings 90% of the time. The others shine for real-time features (leaderboards, online users, feeds).

TTL — Time To Live

Every cached value has an expiry time. After TTL expires, Redis deletes it automatically.

Important Redis Interview Questions

👉 What happens when Redis runs out of memory?

▼

Redis uses eviction policies. Default: noeviction (returns error on new writes). Common production setting: allkeys-lru (removes least recently used keys to make room). Configure via maxmemory and maxmemory-policy in redis.conf.

👉 Is Redis single-threaded? How is it so fast?

▼

Yes, Redis is single-threaded for command execution. It's fast because: (1) all data in RAM, (2) no context switching overhead, (3) non-blocking I/O (epoll/kqueue), (4) simple data structures optimized for speed. Single thread avoids lock contention.

👉 Redis vs Memcached — when to use which?

▼

Redis for: multiple data types, persistence, pub/sub, sorted sets (leaderboards).
Memcached for: pure simple caching with slightly better multi-threaded performance for simple GET/SET at massive scale.
In practice: Redis for almost everything.

👉 How does Redis persist data?

▼

Two options: RDB (periodic snapshots to disk) and AOF (append-only file logging every write). RDB is faster recovery, AOF is more durable. You can use both. For caching, persistence is optional — data can be rebuilt from the database.

Redis stores data in RAM. If the server crashes, is all cached data lost?

Reveal Answer

Yes — and that's fine. Redis is a cache, not a database. The source of truth is your PostgreSQL/SQLite DB. When Redis restarts, the cache is empty. The next request hits the DB, rebuilds the cache, and life goes on. This is by design.

Patterns

Caching Strategies

Three patterns for how and when to populate the cache.

1 Cache-Aside (Lazy Loading)

The most common pattern. Check cache first — if miss, query DB and store result.

Cache-Aside Flow
👤 Request
➔
⚡ Check Cache
➔
HIT? Return!~2ms
↓ MISS
🗃️ Query DB~45ms
➔
⚡ Store in CacheSET + TTL
➔
Return Data
Cache-Aside in Python
▼
from django.core.cache import cache

def get_menu_items():
    # 1. Check cache
    cached = cache.get('menu_items')
    if cached:
        return cached              # Cache HIT — 2ms

    # 2. Cache MISS — query DB
    items = MenuItem.objects.all()
    data = serialize(items)

    # 3. Store in cache for 5 minutes
    cache.set('menu_items', data, timeout=300)

    return data

✅ Pros

  • Simple to implement
  • Only caches what's actually requested

❌ Cons

  • First request always slow (miss)
  • Possible stale data until TTL expires

2 Write-Through

When data changes, immediately update the cache too.

Write-Through Pattern
▼
def update_menu_item(item_id, new_data):
    # 1. Write to DB
    item = MenuItem.objects.get(id=item_id)
    item.name = new_data['name']
    item.save()

    # 2. Immediately invalidate cache
    cache.delete('menu_items')
    # Next read will rebuild the cache (cache-aside kicks in)

3 Write-Around

Write goes only to DB, skipping the cache entirely. Cache gets populated only when someone reads the data (via cache-aside).

Write-Around Flow
✏️ Write Request"Update price"
➔
🗃️ DB OnlyCache untouched
↓
👤 Next ReadCache MISS
➔
🗃️ DB
➔
⚡ Cache UpdatedFresh data now cached

✅ Pros

  • Cache doesn't get polluted with rarely-read data
  • Writes are fast (only DB, no cache overhead)

❌ Cons

  • First read after a write is always slow (cache miss)
  • Stale data if someone reads before cache rebuilds

4 Write-Back (Write-Behind)

Write goes to cache first, then asynchronously flushed to DB later. Fastest writes, but risky.

Write-Back Flow
✏️ Write Request"Update price"
➔
⚡ Cache UpdatedInstant! User gets 200 OK
↓ later (async)
⏰ Background JobCelery / cron
➔
🗃️ DB UpdatedEventually consistent

✅ Pros

  • Fastest writes (user doesn't wait for DB)
  • Reduces DB write load (batch writes)

❌ Cons

  • Data loss risk — if Redis crashes before flushing to DB, writes are lost
  • Complex — need async workers, retry logic, conflict resolution

Your restaurant app tracks "number of times each dish was viewed" for analytics. You get 50,000 view events per hour. Which caching pattern should you use?

Reveal Answer

Write-Back. Use INCR dish:views:42 in Redis (instant, handles 50K/hour easily). Then a background job flushes counts to PostgreSQL every 5 minutes. If Redis crashes, you lose at most 5 minutes of view counts — acceptable for analytics. Writing 50K rows/hour directly to DB would be wasteful.

Our Restaurant API Uses: Cache-Aside

Menu data is read 1000x more than it's written. Cache-aside with TTL is perfect — simple, effective, and the slight staleness (up to 5 minutes) is acceptable. For the "dish views" counter, we'd use write-back. Most apps use cache-aside for 90% of their caching needs.

5 Cache Invalidation

"There are only two hard things in Computer Science:
cache invalidation and naming things."

— Phil Karlton

When underlying data changes, you must either:

Delete the key
Next read rebuilds from DB. Simple & safe.
Update the cache
Write-through. Cache always fresh, writes slower.
Let TTL expire
Lazy. Stale for up to TTL seconds. Simplest.

Our Strategy

For the restaurant API: delete on write + TTL as safety net. Admin updates a dish? Delete cache key. Even if we forget, TTL ensures stale data disappears in 5 minutes.

These solutions look simple. So why is cache invalidation one of the hardest problems?

Real Incident: Facebook Outage (2021)

A configuration change triggered a cascade of cache invalidations across their global CDN. Billions of requests that were normally served from cache suddenly hit the origin servers. The servers couldn't handle the load and crashed — taking down Facebook, Instagram, and WhatsApp for 6+ hours.

The root cause wasn't that caching is hard to implement — it's that caching creates invisible dependencies. When cache works, you forget it exists. When it breaks, everything behind it was never built to handle the real load.

Why It's Actually Hard:

💥 Multiple copies: The same data lives in browser cache, CDN, Redis, and DB query cache — which one do you invalidate?
💥 Distributed systems: 10 servers, each with their own local cache. You update on server 1 — servers 2-10 still have stale data.
💥 Race conditions: Two requests arrive simultaneously. One reads stale cache while the other is updating it.
💥 The delete vs update dilemma: If you delete the cache key, 1000 users hit the DB simultaneously (stampede). If you update it, the update might fail, leaving stale data.

Admin updates the price of Butter Chicken. Customers still see the old price for 5 minutes. Is this acceptable?

Reveal Answer

For a restaurant menu? Yes. 5 minutes of stale prices is fine — the order system uses the DB price anyway, not the cached display price. For a stock trading app? Absolutely not. TTL depends on how stale your data is allowed to be.

Level 1: Per-View Cache (Easiest)

cache_page decorator
▼
from django.utils.decorators import method_decorator
from django.views.decorators.cache import cache_page

class MenuItemListView(generics.ListAPIView):
    # ... existing queryset, serializer, pagination ...

    @method_decorator(cache_page(60 * 5))  # Cache for 5 minutes
    def list(self, request, *args, **kwargs):
        return super().list(request, *args, **kwargs)

# Each unique URL gets its own cache entry
# /menu/items/?page=1  → cached separately from
# /menu/items/?page=2  → cached separately from
# /menu/items/?search=chicken  → each is independent

Level 2: Manual Cache (Most Control)

Manual cache.get / cache.set
▼
from django.core.cache import cache
from rest_framework.response import Response

class MenuItemListView(generics.ListAPIView):
    # ... existing config ...

    def list(self, request, *args, **kwargs):
        # Build unique key from query params
        cache_key = f"menu_items:{request.GET.urlencode()}"

        # Check cache
        cached_data = cache.get(cache_key)
        if cached_data is not None:
            return Response(cached_data)  # HIT — 2ms

        # MISS — get from DB
        response = super().list(request, *args, **kwargs)

        # Store for 5 minutes
        cache.set(cache_key, response.data, timeout=300)
        return response

Beyond Views — Everything Django Can Cache

We cached a view. But Django's cache framework can do much more:

Low-Level API cache.get() / cache.set() Cache anything — query results, computed values, API responses from external services

Cache Middleware UpdateCacheMiddleware + FetchFromCacheMiddleware Caches every page on your entire site automatically. UpdateCacheMiddleware (first in MIDDLEWARE) saves responses to cache on the way out. FetchFromCacheMiddleware (last in MIDDLEWARE) checks cache on the way in — if cached, the request never reaches your views at all. Rarely used for APIs because it ignores auth headers, query params can get tricky, and you lose fine-grained control. Good for static content sites.
Cache-Control Header @cache_control(max_age=300) Tell the browser/CDN to cache the response — no server hit at all

# 3. Per-Object Cache
def get_menu_item(item_id):
    cache_key = f"menu_item:{item_id}"
    item = cache.get(cache_key)
    if item is None:
        item = MenuItem.objects.get(id=item_id)
        cache.set(cache_key, item, timeout=300)
    return item

Which One Should You Use?

For APIs: Start with per-view cache (cache_page) or manual cache (cache.get/set). Add cache_control header if you want CDN/browser caching too.
For sessions: Switch to Redis session backend — it's faster than DB sessions and you already have Redis running.
For everything else: Low-level API (cache.get/set) gives you full control.

Watch Out

Common Caching Pitfalls

Caching is simple to add, tricky to get right.

💥
Cache Stampede
Cache expires, 1000 users hit DB simultaneously. Fix: cache lock, staggered TTL, or pre-warm cache before expiry.
💤
Stale Data
Price updated but cache shows old price for 5 minutes. Fix: invalidate on write, or accept staleness if business allows it.
💫
Redis Goes Down
App crashes because it can't reach cache. Fix: try/except around cache calls, fallback to DB query.
📈
No TTL
Cached data lives forever, never refreshes. Redis fills up. Fix: always set TTL, set maxmemory policy.

Your cache TTL is 5 minutes. At exactly 5:00 PM, the cache expires. 500 users are on the site. What happens?

Reveal Answer

Cache stampede. All 500 users hit the DB simultaneously because the cache is empty. The DB gets 500 identical queries at once. Fix: use a cache lock (only one request rebuilds cache, others wait) or staggered TTL (add random jitter: TTL = 300 + random(0, 60)).

🔒 Further Reading: Web Cache Poisoning

What is Cache Poisoning?

An attacker tricks your cache into storing a malicious response — and then every subsequent user gets served that poisoned version.

Simple example:

# Normal request — server returns menu page
GET /menu/items/
Host: restaurant.com
→ Cache stores: {"items": [...]}

# Attacker sends a request with a sneaky header
GET /menu/items/
Host: restaurant.com
X-Forwarded-Host: evil.com
# If the server blindly uses this header in the response
# (e.g., generating links), the cache now stores a response
# with links pointing to evil.com
→ Cache stores: {"links": "https://evil.com/..."}

# Now EVERY user who hits /menu/items/ gets the
# poisoned response from cache — redirected to evil.com

Why it matters: The attacker only needs to send one malicious request. The cache does the rest — serving the poisoned response to thousands of users until TTL expires. It's like poisoning a water tank instead of individual glasses.

Prevention: Only cache based on URL + known query params. Never let unvalidated headers (like X-Forwarded-Host) influence cached responses. Use Vary headers carefully.

Mentioning Your Project on a Resume

You built real systems. Now learn to describe them so interviewers actually care.

Who Needs What

For beginners (0-2 years): Projects are ESSENTIAL. They're often the only thing that proves you can actually build something. A strong project section can get you the interview even without brand-name experience.

For experienced engineers (3+ years): Your company experience matters more. Projects become supplementary. But even then, a side project that shows you explored something NEW (a new language, a system design concept, a performance optimization) stands out.

What Grabs Attention Over Toy Projects:

• Projects that solve a REAL problem (not just a tutorial clone)
• Projects with measurable impact (response time improvements, query optimizations)
• Projects that demonstrate understanding of trade-offs (why Redis over Memcached, why webhooks over polling)
