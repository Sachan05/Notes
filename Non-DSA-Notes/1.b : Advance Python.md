**1. Magic (Dunder) Methods :**  
Magic methods are special methods surrounded by double underscores. They are called magic methods because Python automatically calls them when certain operations happen. **You don't call them directly. Python runtime calls them.**
  
  ```__init__
      __str__
      __repr__
      __len__
      __eq__

      Ex : len(obj)   ----> Internally : obj.__len__()
      Ex : print(obj) ----> Internally : obj.__str__()
```

* Common Methods

| Method | Purpose |
|---------|---------|
| `__init__` | Constructor | Called when object is created.
| `__str__` | Human-readable string (`print`) |
| `__repr__` | Developer/debug representation (`repr`) |
| `__len__` | Supports `len(obj)` |
| `__eq__` | Equality (`==`) |
| `__lt__` | Less than (`<`) |
| `__add__` | Addition (`+`) |
| `__iter__` | Returns iterator |
| `__next__` | Returns next item |
| `__getitem__` | Indexing (`obj[i]`) |
| `__contains__` | Membership (`in`) |




Q1 What are magic methods?  
Q2 Why are they called automatically?  
  - Because Python translates common operations into special method calls. You don't call them directly. Python runtime calls them. For example:  
      ```
      len(obj) → obj.__len__()
      print(obj) → obj.__str__()
      a + b → a.__add__(b)
      ```

| `__str__` | `__repr__` |
|------------|------------|
| Human-readable | Developer/debug representation |

| Optional | Recommended |

> If `__str__` is missing, Python falls back to `__repr__`.

**2. Mutable vs Immutable :**

Mutable : A mutable object can be modified after it is created. Ex : list, dict, set, bytearray
Immutable : An immutable object cannot be modified after creation. Ex : int, float, bool, str, tuple, frozenset, bytes

    ```
      Ex : x = "hello"
           x += " world"
           print(x)    ----> output ; hello world
    
    **Looks like the string changed. It didn't. Python created a new string object.**
    ```


> ## Object Identity : Use `id(obj)` to check whether an object changed or a new one was created.




Q : Why are Strings Immutable?
  - Thread-safe to share across code.
  - Can be hashed (used as dictionary keys).
  - Performance optimizations (interning/caching).
  - Prevent accidental modification.

Q : Why are Lists Mutable?
  - Because they're designed for dynamic collections where items are added, removed, or updated efficiently.


Q : Why are dictionary keys immutable?
  - Because keys are stored using their hash value. If a key changed after insertion, its hash could change, making it impossible to find the entry reliably.

Q : Is Tuple mutable ? 
  - Tuples: Immutable Container, Mutable Contents
  - The tuple structure is immutable, but it can contain mutable objects whose contents can change.
          ```
          t = ([1, 2], 3)
          t[0].append(4)
          print(t)   ---->  Output : ([1,2,4],3)
          The tuple still points to the same list; only the list's contents changed.
          ```

Q : Does += always create a new object?
  - For immutable types (int, str, tuple): Yes
  - For mutable types like list: No, it usually modifies the existing object in place.

Q : What parameter should you use instead of [] as a default value?
  - Use None, then create the mutable object inside the function.

**Q : Does Python use pass-by-value or pass-by-reference?  
Python uses pass-by-object-reference (also called call-by-sharing). Function parameters receive a reference to the same object. If the object is mutable and modified in place, the caller sees the change. If the parameter is reassigned to a new object, only the local reference changes, and the caller is unaffected.**
          ```
          Function call: object reference is copied    
          Mutable object: Modify object → Caller sees changes   
          Immutable object: Modification creates new object → Caller doesn't see changes
    **Rule to remember: Modifying an object and reassigning a variable are two completely different operations in Python.**
          ```

* Pass by object refernece :
  ```
  Ex 1 : Mutable object.
  Modify the object. The original object changes. Everyone pointing to it sees the change. ex :  lst.append(), lst.pop(), dict["x"] = 10, set.add()


  def add(lst):
    lst.append(100)
  
  nums = [1, 2]
  add(nums) 
  print(nums)
  * Initially : nums ─────────► [1,2]. When the function add(nums) is called: Python creates a new local variable lst. But it points to the same object. nums and lst ----> [1,2]
  * Inside the function, lst.append(100), append() modifies the existing object.
  * Now, nums and lst ----> [1,2,100]. After the function returns, nums ─────────► [1,2,100]. So the caller also sees the change.


  Ex 2 : Immutable Object. Reassign the local variable. Only the local variable changed. The caller is unaffected.

  def add(x):
    x += 10

  a = 5
  add(a)
  print(a)
  * Initially, a ─────────► 5, Function call, a and x ----> 5. Now, after function is called, x += 10.
  * x doesnt become 15. Python creates a new integer object. Memory becomes, a ─────────► 5, x ─────────► 15. The original object 5 is untouched. So after returning, a = 5\
   ```
