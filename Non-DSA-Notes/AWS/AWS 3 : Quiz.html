Click each component to understand what you need to launch an EC2:

🖼️ AMI (Amazon Machine Image) ▼
What: A template containing OS + pre-installed software.

Common choices:
• Ubuntu 22.04 LTS (most Django tutorials use this)
• Amazon Linux 2023 (AWS optimized)

Think of it as: A "snapshot" you can clone to create new servers.
⚙️ Instance Type ▼
Format: t2.micro

• t = Family (t=general, c=compute, r=memory)
• 2 = Generation (higher = newer)
• micro = Size (CPU + RAM)

FamilyOptimized ForUse Case
T (t2, t3)General, burstableWeb servers, dev
M (m5, m6)BalancedProduction apps
C (c5, c6)CPU intensiveML, video encoding
R (r5, r6)Memory intensiveDatabases, caching
Free Tier: t2.micro or t3.micro (750 hrs/month)
🔑 Key Pair ▼
What: SSH key to connect to your instance.

Important: Download the .pem file immediately - you can't get it again!

# Set permissions (required!) chmod 400 my-key.pem # Connect to EC2 ssh -i my-key.pem ubuntu@<public-ip>
chmod 400: Owner read-only. SSH refuses keys with loose permissions.
🌍 Regions & Availability Zones ▼
Region: Geographic location (ap-south-1 = Mumbai)
AZ: Independent data center within a region

Choose region based on:
• Latency: Close to your users
• Cost: US regions often cheapest
• Compliance: Data residency laws

Region: ap-south-1 (Mumbai) ├── AZ: ap-south-1a ← Independent data center ├── AZ: ap-south-1b ← If one fails, others survive └── AZ: ap-south-1c
Compute 750 hrs/month free 🔗 AWS Console

🔄 Stateful Behavior ▼
Key concept: If you allow traffic IN, the response OUT is automatic!

Request: Client ──allowed──► EC2 Response: EC2 ──auto-allowed──► Client ← No rule needed!
This is what "stateful" means. NACLs are stateless (you'd need both rules).

🔢 CIDR Notation ▼
The /number tells how many bits are fixed (network part):

CIDRAvailable IPsMeaning
/321Exact single IP
/24256x.x.x.0 to x.x.x.255
/1665,536x.x.0.0 to x.x.255.255
/0All0.0.0.0/0 = anywhere
In Security Groups:
0.0.0.0/0 = any IPv4
::/0 = any IPv6
🔗 SG Reference Pattern (Important!) ▼
Instead of hardcoding IPs, reference other Security Groups:

Web-SG: Allow 80,443 from 0.0.0.0/0 App-SG: Allow 8000 from Web-SG-ID ← Not an IP! DB-SG: Allow 5432 from App-SG-ID ← Not an IP!
Why? EC2 IPs change. SG IDs don't. Auto-scaling friendly!

🔷 Nginx (Reverse Proxy) ▼
What it does:
• Handles HTTPS/SSL termination
• Serves static files directly (CSS, JS, images)
• Load balances across multiple Gunicorns
• Buffers slow clients (protects backend)
• Gzip compression

Why? Nginx handles 10,000+ concurrent connections. Gunicorn can't.
🐍 Gunicorn (WSGI Server) ▼
What: Python HTTP server that runs Django

gunicorn mysite.wsgi:application \ --bind 0.0.0.0:8000 \ --workers 3
Workers formula: (2 × CPU cores) + 1

Each worker = independent Python process handling requests.
🔄 Forward vs Reverse Proxy (Interview Question!) ▼
FORWARD PROXY (client-side, protects clients): ┌────────┐ ┌─────────┐ ┌────────┐ │ Client │ ──► │ Forward │ ──► │ Server │ │ (You) │ │ Proxy │ │ │ └────────┘ └─────────┘ └────────┘ Client knows, Server doesn't server doesn't know client Use cases: VPNs, corporate firewalls, anonymity REVERSE PROXY (server-side, protects servers): ┌────────┐ ┌─────────┐ ┌────────┐ │ Client │ ──► │ Reverse │ ──► │ Server │ │ │ │ Proxy │ │ (Nginx)│ └────────┘ └─────────┘ └────────┘ Client doesn't Server knows know it exists Use cases: Load balancing, SSL, caching, security
TypeSits in front ofExamples
ForwardClientsVPNs, Squid
ReverseServersNginx, HAProxy, AWS ALB

🔄 NAT Gateway ▼
Problem: Private EC2 needs to run pip install django. But it has no public IP!

Solution: NAT Gateway sits in public subnet and translates private → public.

Private EC2 ──► NAT Gateway ──► Internet Gateway ──► Internet │ (Lives in public subnet) (One-way: outbound only)
Cost: ~$32/month! For dev, use NAT Instance (t3.micro) instead.

MetricWhat It MeasuresAlert When
CPUUtilizationCPU usage %> 80% sustained
NetworkIn/OutBytes transferredSudden spike
StatusCheckFailedInstance healthAny failure

⚠️ Memory & Disk NOT Included! ▼
Why? CloudWatch runs outside the EC2. It can see CPU (hypervisor schedules it), but can't see inside the guest OS.
Solution: Install CloudWatch Agent inside EC2:

# Install agent wget https://s3.amazonaws.com/.../amazon-cloudwatch-agent.deb sudo dpkg -i amazon-cloudwatch-agent.deb # Configure sudo /opt/aws/.../amazon-cloudwatch-agent-config-wizard
Now you get: Memory %, Disk %, process counts, etc.
🔔 Setting Up Alarms ▼
Step 1: Create SNS Topic (email notifications) Step 2: Create CloudWatch Alarm └─ Metric: CPUUtilization > 80% └─ Period: 5 minutes Step 3: Set Action → notify SNS topic When CPU > 80% for 5 min → Email sent!
Pro tip: Set up billing alarms at $5, $10, $20 to avoid surprise bills!
Free: 10 metrics, 3 dashboards $0.10/alarm/month 🔗 AWS Console

📁 Django Project Structure for EB ▼
myproject/ ├── requirements.txt ← Python dependencies ├── .ebextensions/ │ └── django.config ← EB configuration ├── myapp/ │ └── ... └── manage.py
⚖️ Auto-Scaling with EB ▼
Default: Scale up when CPU > 70% Customizable: ├── Trigger metric: CPU, Memory, Network ├── Thresholds: 50%, 80%, etc. ├── Min instances: 1 ├── Max instances: 4 └── Cool-down period: 5 minutes
❌ When NOT to Use EB ▼
• Need fine-grained control over infrastructure
• Complex networking requirements
• Using containers (use ECS/EKS instead)
• Serverless (use Lambda)

EB is great for: Standard web apps, Django/Flask, quick deployments, MVPs
FREE (pay for resources) Auto-scaling included 🔗 AWS Console

🔄 Request Flow (Step by Step) ▼
1. User types myapp.com
2. Route 53 returns ALB IP (Alias record)
3. Request hits Internet Gateway
4. ALB terminates SSL, health checks
5. Security Group allows port 443 ✓
6. ALB forwards to EC2 in private subnet
7. Django processes request
8. Queries RDS (SG allows from App-SG)
9. Response returns same path
