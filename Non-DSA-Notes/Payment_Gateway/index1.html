Fundamentals

Why Payments Are Different

Payment integration isn't like regular CRUD. Three fundamental challenges make it complex.

🌐
External Systems
You don't control payment gateways or banks. They can be slow, have outages, or return unexpected errors.
⏳
Asynchronous
Payment status can change minutes or hours later. Bank verifications, fraud checks happen in background.
💰
Real Money
Bugs mean financial loss. Double-charging, missing payments, or security breaches have serious consequences.

🤔 Why can't we build our own payment system?

▼

Three major barriers prevent most companies from handling payments directly:

1. PCI DSS Compliance

Payment Card Industry Data Security Standard — a set of 300+ security requirements you must follow to handle card data. Created after massive breaches at Target, Home Depot, etc.

  • Level 1 (handling card data): Annual audit ($50K-$200K), quarterly pen tests, dedicated security team
  • SAQ A (using Stripe/Razorpay): Just fill a questionnaire, no audit needed

2. Banking Relationships

You need direct contracts with Visa, Mastercard, RuPay, and acquiring banks. This takes months of negotiation, large deposits, and ongoing compliance.

3. Fraud Prevention

Stripe/Razorpay have ML models trained on billions of transactions. Building this from scratch would take years and massive datasets.

Bottom line: Using Stripe/Razorpay costs 2-3% per transaction. Building your own would cost $500K+ per year in compliance alone, plus engineering. Only Amazon/Netflix scale justifies it.

⏳ How Do We Handle Asynchronous Operations?

The core problem: Payments are asynchronous — the result isn't instant. Bank verifications, fraud checks, and OTP verification take time.

💡 "Why not just make an API call and wait for the response?"

A common first thought. Here's why it fails:

⏱️ Timeouts
Bank verification: 30+ sec. HTTP timeout: 30 sec.
📱 OTP/3D Secure
User takes 2+ min to enter OTP. Can't hold connection.
🔌 Network Drop
Payment succeeds, response lost. Money charged, DB not updated.
💥 Server Crash
Your server dies mid-request. No retry mechanism.

So how does My Site know when the Payment Gateway (PG) is done processing? Here are 4 patterns — each with trade-offs:

1. Polling (My Site asks PG repeatedly)

My Site keeps asking the Payment Gateway "Is the payment done yet?" every few seconds.

       │◀────────── { status: "pending" } ─────────────│
       │                                                │
       │  ⏳ wait 2 seconds...                         │
       │                                                │
       │─────── GET /status/pay_123 ──────────────────▶│
       │◀────────── { status: "pending" } ─────────────│
       │                                                │
       │  ⏳ wait 2 seconds...                         │
       │                                                │
       │─────── GET /status/pay_123 ──────────────────▶│
       │◀────────── { status: "pending" } ─────────────│
       │                                                │
       │  ⏳ wait 2 seconds...                         │

my_site/payment_checker.py 
# My Site polls the Payment Gateway every 2 seconds
import time, requests

def check_payment_status(payment_id):
    while True:
        response = requests.get(
            f"https://api.razorpay.com/v1/payments/{payment_id}",
            auth=(RAZORPAY_KEY, RAZORPAY_SECRET)
        )
        if response.json()['status'] == 'captured':
            return "Payment successful!"
        time.sleep(2)  # Ask again in 2 seconds

❌ Problem: Wasteful. 100,000 payments = 3 million requests/minute to Payment Gateway API. Rate limits will block you!

2. Long Polling (My Site holds connection to PG)

My Site makes a request to PG, and PG holds the connection open until payment status changes (or timeout). 

    My Site                                      Payment Gateway
       │                                                │
       │─────── GET /status/pay_123?wait=true ────────▶│
       │                                                │  PG holds connection open...
       │                   ⏳ 5 sec...                  │  waiting for bank response
       │                   ⏳ 10 sec...                 │
       │                   ⏳ 15 sec...                 │  Bank confirms!
       │◀────────── { status: "completed" } ───────────│
       │                                                │
my_site/payment_checker.py 
# My Site makes a blocking request to PG
def wait_for_payment(payment_id):
    while True:
        # PG holds this request for up to 30 seconds
        response = requests.get(
            f"https://api.gateway.com/payments/{payment_id}/wait",
            timeout=35  # Slightly longer than PG's hold time
        )
        if response.json()['status'] == 'completed':
            return "Payment successful!"
        # If timeout, loop again

⚠️ Problem: Ties up a server thread/connection per payment. If My Site has 10,000 pending payments, that's 10,000 open connections!

3. WebSockets (Persistent connection between My Site & PG)

My Site maintains a persistent bidirectional connection with PG. PG pushes updates as they happen. 

    My Site                                      Payment Gateway
       │                                                │
       │═══════ WebSocket Connection Established ══════│
       │◀═══════════════════════════════════════════════│
       │                                                │
       │─────── { "subscribe": "pay_123" } ───────────▶│
       │                                                │
       │               ⏳ time passes...               │
       │                                                │
       │◀────── { status: "processing" } ──────────────│  PG pushes update
       │                                                │
       │               ⏳ more time...                 │
       │                                                │
       │◀────── { status: "completed" } ───────────────│  PG pushes final status
       │                                                │
       │═══════════════ Connection Open ════════════════│
my_site/payment_listener.py 
# My Site maintains persistent WebSocket to PG
import websockets

async def listen_for_payments():
    async with websockets.connect("wss://gateway.com/ws") as ws:
        await ws.send('{"subscribe": "pay_123"}')
        async for message in ws:
            data = json.loads(message)
            if data['status'] == 'completed':
                update_order_in_db(data['payment_id'])

⚠️ Problem: Complex to manage. What if My Site restarts? Connection drops? Need reconnection logic, heartbeats, etc.

4. Webhooks (PG calls My Site when ready)

My Site gives PG a URL. When payment status changes, PG initiates a POST request to My Site. 

    My Site                                      Payment Gateway
       │                                                │
       │  (during initial setup)                        │
       │─────── "My webhook URL is /webhooks/pg" ─────▶│
       │                                                │
       │                                                │
       │  ⏳ Payment processing happens...             │
       │     My Site does other work                   │
       │     No polling, no waiting!                   │
       │                                                │
       │                                                │  Bank confirms payment!
       │                                                │
       │◀══════ POST /webhooks/pg ═══════════════════│  PG initiates call
       │        { event: "payment.success",           │
       │          payment_id: "pay_123" }             │
       │                                                │
       │─────────── 200 OK ───────────────────────────▶│
       │                                                │
my_site/views.py 
# PG calls this endpoint when payment status changes
@csrf_exempt
def razorpay_webhook(request):
    payload = json.loads(request.body)
    signature = request.headers.get('X-Razorpay-Signature')

    # Verify it's really from Razorpay (not a hacker)
    if verify_signature(payload, signature, WEBHOOK_SECRET):
        payment_id = payload['payload']['payment']['entity']['id']
        Order.objects.filter(payment_id=payment_id).update(status='paid')
        return HttpResponse(status=200)
    return HttpResponse(status=400)

✅ Best for payments: No open connections, no polling. PG retries automatically if My Site is temporarily down!

Why webhooks win for payments:
  1. Decoupled systems — My Site doesn't need to maintain connections to PG

  4. Resilient — Works even if My Site has temporary downtime
Architecture

Who Calls the Payment Gateway?

A common confusion: Does the frontend or backend talk to Razorpay/Stripe? Answer: Both! But for different things.

🖥️ Frontend (JavaScript)

  • Loads Razorpay/Stripe SDK
  • Creates secure iframe for card input
  • Sends card data directly to gateway
  • Receives payment confirmation
// Frontend: Opens Razorpay popup
const rzp = new Razorpay(options);
rzp.open();
&

⚙️ Backend (Python/Django)

  • Creates orders (with amount, currency)
  • Uses secret API key (never exposed)
  • Receives webhooks
  • Verifies signatures
# Backend: Creates order
razorpay_client.order.create({
    'amount': 10000,
    'currency': 'INR'
})
Frontend                    Backend                     Razorpay
    │                            │                             │
    │──── "Create Order" ───────▶│                             │
    │                            │──── POST /orders ──────────▶│
    │                            │◀─── { order_id: "..." } ────│
    │◀─── { order_id } ──────────│                             │
    │                            │                             │
    │════════ Card Data (iframe) ══════════════════════════════▶│
    │                            │                             │
    │◀═══════ { payment_id, signature } ═══════════════════════│
    │                            │                             │
    │──── "Verify Payment" ─────▶│                             │
    │                            │◀──── Webhook POST ──────────│
    │                            │───── 200 OK ───────────────▶│
Why this split?
  • Security: Backend holds the secret key — never expose it to JavaScript
  • PCI Compliance: Card data goes directly to gateway — never touches your servers
  • Trust: Backend verifies webhook signatures — frontend can be manipulated

📦 Installing the Razorpay Python SDK

🤔 What is an SDK?

SDK = Software Development Kit — a package of pre-written code that makes it easy to use someone's API.

Think of it like this:

❌ Without SDK:

You manually write HTTP requests, handle authentication, parse responses, verify signatures...

✅ With SDK:

Just call client.order.create() — the SDK handles all the messy details!

Analogy: An SDK is like a TV remote — you press "Volume Up" instead of manually adjusting circuits inside the TV.

Your Django backend needs the Razorpay SDK to create orders and verify webhooks:

Terminal
pip install razorpay
views.py
import razorpay

# Initialize with your keys
razorpay_client = razorpay.Client(
    auth=(RAZORPAY_KEY_ID, RAZORPAY_KEY_SECRET)
)

# Create an order — SDK handles the HTTP request!
order = razorpay_client.order.create({
    'amount': 10000,  # ₹100 in paise
    'currency': 'INR',
    'payment_capture': 1
})

# Verify webhook signature — SDK handles the HMAC math!
razorpay_client.utility.verify_webhook_signature(
    body, signature, webhook_secret
)

Sequence: Shopify Checkout with Discount Code

How does data flow when you apply a discount code AND enter card details on the same page?

User Merchant Frontend Merchant Backend Stripe/Razorpay
1 User enters discount code "SAVE10" → Merchant's JavaScript validates → Shows updated price
2 User enters card in Stripe iframe → Keystrokes go directly to Stripe (merchant can't see)
3 User clicks "Pay ₹450" → Merchant backend creates PaymentIntent with amount=450
4 Stripe confirms card + charges ₹450 → Sends webhook to merchant backend
5 Merchant backend updates order as "paid" → User sees success page
Key insight: The discount code logic runs on merchant's server (they control pricing). Card data goes to Stripe's server (merchant never sees it). They coordinate via PaymentIntent ID.

Core Concept

Callback AND Webhook — Both Happen!

They're not alternatives. They happen simultaneously but serve different purposes.

📱 Callback (Browser Redirect)

  • User's browser redirects to /success
  • Immediate (user sees it)
  • Purpose: Show "Thank You" page
  • ⚠️ Can be faked!
+

🖥️ Webhook (Server POST)

  • Gateway POSTs to your /webhook
  • May be delayed 2-30 seconds
  • Purpose: Update your database
  • ✅ Cryptographically signed

Callback tells the user what happened.
Webhook tells your server what happened.

Trust the webhook.

Critical Concept

Idempotency — Handle Duplicate Webhooks

Payment gateways retry webhooks if they don't get a 200 OK. Your code must handle the same webhook arriving multiple times.

🔁 When do duplicates happen?

  • Network timeout: Your server processed it but response was lost — gateway retries
  • Your server crashed: Mid-processing — gateway retries after recovery
  • Gateway hiccup: Their system sent it twice by mistake
Razorpay                                      Your Server
    │                                                │
    │──── POST /webhook (payment.captured) ─────────▶│ ✓ Processes, marks order as paid
    │                                                │
    │        ⏳ network timeout, no 200 OK received  │
    │                                                │
    │──── POST /webhook (payment.captured) ─────────▶│ ❌ If not idempotent: charges twice!
    │                                                │ ✓ If idempotent: "already processed"

How to Make Your Webhook Handler Idempotent

Option 1: Check payment_id before processing

views.py
def webhook_handler(request):
    payload = json.loads(request.body)
    razorpay_payment_id = payload['payload']['payment']['entity']['id']

    # Check if we already processed this payment
    payment = Payment.objects.filter(
        razorpay_payment_id=razorpay_payment_id,
        status='completed'
    ).first()

    if payment:
        # Already processed — just return 200 OK
        return HttpResponse('Already processed', status=200)

    # First time seeing this — process it
    process_payment(razorpay_payment_id)
    return HttpResponse('OK', status=200)

Option 2: Use a webhook log table

views.py
def webhook_handler(request):
    payload = json.loads(request.body)
    event_id = payload.get('event_id')  # Unique ID from gateway

    # Try to create a log entry (unique constraint on event_id)
    try:
        WebhookLog.objects.create(event_id=event_id, payload=payload)
    except IntegrityError:
        # Duplicate! Already processed.
        return HttpResponse('Duplicate', status=200)

    # First time — process it
    process_payment(payload)
    return HttpResponse('OK', status=200)

Option 3: Database transaction with SELECT FOR UPDATE

For high-concurrency systems, use a database lock to prevent race conditions:

views.py
from django.db import transaction

@transaction.atomic
def process_webhook(payment_id):
    # Lock the row to prevent concurrent processing
    payment = Payment.objects.select_for_update().get(
        razorpay_payment_id=payment_id
    )

    if payment.status == 'completed':
        return  # Already done

    payment.status = 'completed'
    payment.save()

🧠 Idempotency Quiz

Your webhook handler has a bug: it charges the customer's credit card again each time it's called. Razorpay sends the same webhook 3 times due to network issues. What happens?

A) Nothing — Razorpay handles duplicates
B) Customer is charged 3 times (your bug!)
C) Only the first charge goes through
D) Razorpay blocks the duplicates
Rule of thumb: Webhooks are "at least once" delivery. Your handler must be safe to run multiple times with the same data. Always check if you've already processed this event before taking action.
